SOLID, DRY, KISS, YAGNI

Принципы проектирования, которые на собеседованиях уровня senior ожидают увидеть в применении — а не в виде заученного пересказа. Каждый из них ниже сочетает чёткое определение с проблемой, которую он решает, и ❌/✅ кодом на Angular в контексте FinTech-трейдинга (Order, Quote, Instrument, Position).

Обзор

SOLID — это пять принципов объектно-ориентированного проектирования, сформулированных Робертом Мартином. Поверх них три прагматичные эвристики — DRY, KISS и YAGNI — удерживают код в лаконичной форме. WET — это антипаттерн, с которым борется DRY.

Как подать на собеседовании: назови принцип, объясни, что он означает, назови проблему, которую он решает, а затем покажи рефакторинг. Интервьюеры ценят применённое суждение выше дословного пересказа из учебника.

S — Single Responsibility Principle (Принцип единственной ответственности)

Single Responsibility Principle (SRP) — у класса, компонента или модуля должна быть одна причина для изменения, то есть одна ответственность, принадлежащая одному актору.

Что это значит: разделяй получение данных, отображение/форматирование и представление/состояние, чтобы изменение в одной области не перетекало в другие.

Какую проблему решает: «компоненты-боги», смешивающие HTTP, форматирование и логику шаблона, становятся нетестируемыми, хрупкими и непригодными для переиспользования. SRP делает каждую единицу маленькой, сфокусированной и независимо тестируемой.

❌ Плохо — компонент, делающий HTTP + форматирование + состояние в одном месте:

@Component({
  selector: 'app-positions',
  template: `<div *ngFor="let p of rows">{{ p.label }}</div>`,
})
export class PositionsComponent implements OnInit {
  rows: { label: string }[] = [];

  constructor(private http: HttpClient) {}

  ngOnInit(): void {
    // Responsibility 1: HTTP transport
    this.http.get<Position[]>('/api/positions').subscribe((positions) => {
      // Responsibility 2: formatting/currency logic
      this.rows = positions.map((p) => ({
        label: `${p.symbol}: ${(p.qty * p.price).toLocaleString('en-US', {
          style: 'currency',
          currency: 'USD',
        })}`,
      }));
      // Responsibility 3: view state — all tangled together
    });
  }
}

✅ Хорошо — разбито на сервис (транспорт), пайп (форматирование) и тонкий компонент (представление):

// 1. Service owns transport only
@Injectable({ providedIn: 'root' })
export class PositionService {
  private http = inject(HttpClient);
  getPositions(): Observable<Position[]> {
    return this.http.get<Position[]>('/api/positions');
  }
}

// 2. Pipe owns formatting only
@Pipe({ name: 'notional', standalone: true })
export class NotionalPipe implements PipeTransform {
  transform(p: Position): string {
    return (p.qty * p.price).toLocaleString('en-US', {
      style: 'currency',
      currency: 'USD',
    });
  }
}

// 3. Component owns view/state only
@Component({
  selector: 'app-positions',
  standalone: true,
  imports: [NgFor, AsyncPipe, NotionalPipe],
  template: `
    <div *ngFor="let p of positions$ | async">
      {{ p.symbol }}: {{ p | notional }}
    </div>`,
})
export class PositionsComponent {
  private service = inject(PositionService);
  positions$ = this.service.getPositions();
}

Контекст FinTech: форматирование валют/номинала регулируется и меняется по причинам, не связанным с тем, как получаются позиции. Наличие NotionalPipe означает, что смена локали или правила округления затрагивает один файл, а не каждый компонент.

O — Open/Closed Principle (Принцип открытости/закрытости)

Open/Closed Principle (OCP) — программные сущности должны быть открыты для расширения, но закрыты для изменения.

Что это значит: новое поведение нужно добавлять, дописывая новый код (новый класс/стратегию), а не редактируя существующий, уже протестированный код.

Какую проблему решает: разрастающийся switch по полю типа заставляет заново открывать и перетестировать одну и ту же функцию при каждом появлении нового варианта — магнит для регрессий. Полиморфизм локализует каждый вариант.

❌ Плохо — switch по типу инструмента, который приходится редактировать ради каждого нового типа:

type InstrumentKind = 'stock' | 'bond' | 'option';

@Injectable({ providedIn: 'root' })
export class FeeCalculator {
  calculate(instrument: Instrument, notional: number): number {
    switch (instrument.kind) {
      case 'stock':
        return notional * 0.001;
      case 'bond':
        return notional * 0.0005;
      case 'option':
        return notional * 0.002 + 0.65;
      // Adding 'future' means editing & re-testing THIS method again.
      default:
        throw new Error('Unknown instrument');
    }
  }
}

✅ Хорошо — интерфейс-стратегия; новые инструменты добавляют новый класс, существующий код не правится:

export interface FeeStrategy {
  calculate(notional: number): number;
}

export class StockFee implements FeeStrategy {
  calculate(notional: number): number { return notional * 0.001; }
}
export class BondFee implements FeeStrategy {
  calculate(notional: number): number { return notional * 0.0005; }
}
export class OptionFee implements FeeStrategy {
  calculate(notional: number): number { return notional * 0.002 + 0.65; }
}

// New 'future' fee? Add a class — this registry/consumer is untouched.
@Injectable({ providedIn: 'root' })
export class FeeCalculator {
  private readonly strategies: Record<string, FeeStrategy> = {
    stock: new StockFee(),
    bond: new BondFee(),
    option: new OptionFee(),
  };
  calculate(instrument: Instrument, notional: number): number {
    return this.strategies[instrument.kind].calculate(notional);
  }
}

Контекст FinTech: биржи и классы активов добавляются постоянно. Подход «стратегия/реестр» означает, что подключение нового продукта — это аддитивное изменение, рассматриваемое изолированно, что критично, когда логика комиссий проходит аудит.

L — Liskov Substitution Principle (Принцип подстановки Лисков)

Liskov Substitution Principle (LSP) — объекты подтипа должны быть взаимозаменяемы с объектами их базового типа без нарушения корректности программы.

Что это значит: подкласс должен соблюдать контракт базового класса — те же сигнатуры методов, без усиления предусловий, без ослабления постусловий, без неожиданных исключений.

Какую проблему решает: подкласс, который бросает исключение или ничего не делает там, где база обещала поведение, превращает полиморфизм в мину — вызывающий код, написанный под базовый тип, падает на подтипе.

❌ Плохо — подкласс, нарушающий контракт базы, бросая исключение вместо выполнения:

abstract class TradeExecutor {
  // Contract: returns a confirmed Order for the given quote.
  abstract execute(quote: Quote, qty: number): Order;
}

class MarketExecutor extends TradeExecutor {
  execute(quote: Quote, qty: number): Order {
    return { id: crypto.randomUUID(), symbol: quote.symbol, qty, price: quote.ask, status: 'filled' };
  }
}

class ReadOnlyExecutor extends TradeExecutor {
  // Violates LSP: callers expect an Order, this blows up instead.
  execute(_quote: Quote, _qty: number): Order {
    throw new Error('Read-only feed cannot execute trades');
  }
}

// Any code written against TradeExecutor now randomly crashes.
function place(exec: TradeExecutor, q: Quote): Order {
  return exec.execute(q, 100); // 💥 with ReadOnlyExecutor
}

✅ Хорошо — моделируем возможность явно, чтобы контракт исполнения удовлетворяли только настоящие исполнители:

// Separate the capabilities — don't force a non-executor to pretend.
interface QuoteReader {
  read(symbol: string): Quote;
}
interface TradeExecutor {
  execute(quote: Quote, qty: number): Order; // every implementer truly executes
}

class MarketExecutor implements TradeExecutor {
  execute(quote: Quote, qty: number): Order {
    return { id: crypto.randomUUID(), symbol: quote.symbol, qty, price: quote.ask, status: 'filled' };
  }
}

class LimitExecutor implements TradeExecutor {
  execute(quote: Quote, qty: number): Order {
    // Honors the contract: always returns a valid Order (here, working/pending).
    return { id: crypto.randomUUID(), symbol: quote.symbol, qty, price: quote.bid, status: 'pending' };
  }
}

// A read-only feed implements QuoteReader, never TradeExecutor — no false promise.
class MarketDataFeed implements QuoteReader {
  read(symbol: string): Quote {
    return { symbol, bid: 1.07, ask: 1.0702, ts: Date.now() };
  }
}

Характерный запах нарушения LSP: подкласс переопределяет метод, чтобы throw, вернуть null или проверять instanceof на стороне вызова. Если вызывающим нужно знать конкретный тип, взаимозаменяемость уже нарушена.

Контекст FinTech: маршрутизатор ордеров обращается с каждым исполнителем единообразно. Если один «исполнитель» молча отклоняет заявки, ты получаешь потерянные ордера в продакшене — нарушения LSP здесь это реальный финансовый риск, а не академическая абстракция.

I — Interface Segregation Principle (Принцип разделения интерфейсов)

Interface Segregation Principle (ISP) — ни один клиент не должен быть вынужден зависеть от методов, которые он не использует; предпочитай много маленьких, ролевых интерфейсов одному толстому интерфейсу.

Что это значит: разбей раздутый IService на сфокусированные контракты, чтобы реализации и потребители зависели только от той части, которая им действительно нужна.

Какую проблему решает: толстый интерфейс заставляет каждую реализацию заглушать методы, которые она не поддерживает (часто бросая исключение — что также нарушает LSP), и связывает потребителей с возможностями, которые они никогда не вызывают.

❌ Плохо — один гигантский ITradingService, который должна полностью реализовать каждая реализация:

interface ITradingService {
  getQuote(symbol: string): Quote;
  placeOrder(order: Order): Order;
  cancelOrder(id: string): void;
  streamPositions(): Observable<Position[]>;
  exportRegulatoryReport(): Blob;   // only the back-office cares
  reconcileLedger(): void;          // only settlement cares
}

// A read-only price widget must implement ALL of these, throwing for most.
class PriceWidgetService implements ITradingService {
  getQuote(symbol: string): Quote { /* ... */ return { symbol, bid: 1.07, ask: 1.0702, ts: Date.now() }; }
  placeOrder(): Order { throw new Error('not supported'); }
  cancelOrder(): void { throw new Error('not supported'); }
  streamPositions(): Observable<Position[]> { throw new Error('not supported'); }
  exportRegulatoryReport(): Blob { throw new Error('not supported'); }
  reconcileLedger(): void { throw new Error('not supported'); }
}

✅ Хорошо — сфокусированные интерфейсы; каждый клиент зависит только от того, что использует:

interface QuoteProvider {
  getQuote(symbol: string): Quote;
}
interface OrderManager {
  placeOrder(order: Order): Order;
  cancelOrder(id: string): void;
}
interface PositionStream {
  streamPositions(): Observable<Position[]>;
}
interface BackOfficeReports {
  exportRegulatoryReport(): Blob;
  reconcileLedger(): void;
}

// The widget implements ONLY what it needs — nothing to stub.
class PriceWidgetService implements QuoteProvider {
  getQuote(symbol: string): Quote {
    return { symbol, bid: 1.07, ask: 1.0702, ts: Date.now() };
  }
}

// A full trading desk service composes the contracts it actually provides.
class DeskService implements QuoteProvider, OrderManager, PositionStream {
  getQuote(symbol: string): Quote { /* ... */ return { symbol, bid: 1.07, ask: 1.0702, ts: Date.now() }; }
  placeOrder(order: Order): Order { return { ...order, status: 'filled' }; }
  cancelOrder(_id: string): void { /* ... */ }
  streamPositions(): Observable<Position[]> { return of([]); }
}

Контекст FinTech: регуляторная отчётность и сверка реестра принадлежат совсем другим командам, чем живой ценовой виджет. Разделённые интерфейсы позволяют каждому модулю компилироваться, разворачиваться и проходить аудит, не таща за собой остальные.

D — Dependency Inversion Principle (Принцип инверсии зависимостей)

Dependency Inversion Principle (DIP) — модули высокого уровня не должны зависеть от модулей низкого уровня; оба должны зависеть от абстракций. Абстракции не должны зависеть от деталей.

Что это значит: компонент должен зависеть от интерфейса/токена, а конкретную реализацию поставляет DI Angular — а не создаётся вручную.

Какую проблему решает: компонент, который через new создаёт конкретный сервис, намертво приварен к нему — нельзя подменить реализацию (REST vs WebSocket vs mock), а юнит-тесты не могут внедрить фейк.

❌ Плохо — компонент, создающий конкретный сервис через new:

@Component({ selector: 'app-ticker', template: `{{ quote?.bid }}` })
export class TickerComponent implements OnInit {
  quote?: Quote;
  // Hard-wired to one concrete implementation — untestable, unswappable.
  private feed = new RestQuoteService();

  ngOnInit(): void {
    this.quote = this.feed.getQuote('EURUSD');
  }
}

✅ Хорошо — зависим от абстракции, предоставленной через InjectionToken:

export interface QuoteFeed {
  getQuote(symbol: string): Quote;
}
export const QUOTE_FEED = new InjectionToken<QuoteFeed>('QUOTE_FEED');

// Two interchangeable implementations behind the same abstraction.
@Injectable()
export class RestQuoteService implements QuoteFeed {
  getQuote(symbol: string): Quote { return { symbol, bid: 1.07, ask: 1.0702, ts: Date.now() }; }
}
@Injectable()
export class WsQuoteService implements QuoteFeed {
  getQuote(symbol: string): Quote { return { symbol, bid: 1.071, ask: 1.0712, ts: Date.now() }; }
}

// Wire it at the application/route level:
// providers: [{ provide: QUOTE_FEED, useClass: WsQuoteService }]

@Component({ selector: 'app-ticker', standalone: true, template: `{{ quote?.bid }}` })
export class TickerComponent implements OnInit {
  quote?: Quote;
  // Depends on the abstraction; DI supplies the concrete impl (or a mock in tests).
  private feed = inject(QUOTE_FEED);

  ngOnInit(): void {
    this.quote = this.feed.getQuote('EURUSD');
  }
}

Контекст FinTech: dev/staging могут использовать опрашивающий REST-фид, тогда как продакшен — низколатентный WebSocket-фид, а тесты используют детерминированный mock. С DIP это подмена провайдера в одну строку — компонент при этом не меняется.

DRY — Don't Repeat Yourself (Не повторяйся)

Don't Repeat Yourself (DRY) — каждая часть знания должна иметь единое, однозначное, авторитетное представление в системе.

Что это значит: дублированная логика (валидация, форматирование, магический порог) должна жить ровно в одном месте — общей утилите, пайпе или сервисе.

Какую проблему решает: скопированная логика расходится. Исправь правило округления в трёх компонентах — и пропустишь четвёртый, отгрузив несогласованные, а в трейдинге — некорректные числа.

❌ Плохо — одни и те же валидация и форматирование пипсов дублируются по компонентам:

// order-entry.component.ts
isValid(order: Order): boolean {
  return order.qty > 0 && order.qty <= 1_000_000 && order.price > 0;
}
format(price: number): string { return price.toFixed(5); }

// modify-order.component.ts  — copy-pasted, will drift
isValid(order: Order): boolean {
  return order.qty > 0 && order.qty <= 1_000_000 && order.price > 0;
}
format(price: number): string { return price.toFixed(5); } // someone changes 5→4 here only

✅ Хорошо — одна утилита-валидатор и один общий пайп, переиспользуемые везде:

// order-rules.ts — single source of truth
export const MAX_QTY = 1_000_000;
export function isValidOrder(order: Order): boolean {
  return order.qty > 0 && order.qty <= MAX_QTY && order.price > 0;
}

// pip.pipe.ts — formatting in one place
@Pipe({ name: 'pip', standalone: true })
export class PipPipe implements PipeTransform {
  transform(price: number, digits = 5): string { return price.toFixed(digits); }
}

// Both components now reuse the same logic:
//   if (isValidOrder(order)) { ... }
//   {{ quote.bid | pip }}

Контекст FinTech: точность цены и лимиты количества критичны для комплаенса. Единый order-rules.ts означает, что аудитор просматривает один файл, а изменение лимита не может молча пропустить какой-то экран.

KISS — Keep It Simple, Stupid (Будь проще)

Keep It Simple, Stupid (KISS) — выбирай простейшее работающее решение; избегай ненужной изощрённости.

Что это значит: читаемый, очевидный код лучше плотного однострочника, на расшифровку которого уходит пять минут. Оптимизируй под следующего сопровождающего.

Какую проблему решает: изощрённый код прячет баги и замедляет ревью. Простота снижает когнитивную нагрузку и площадь поверхности для ошибок — особенно ценно в логике, двигающей деньги.

❌ Плохо — чересчур умный однострочник, который никто не может безопасно изменить:

// "Total exposure" — what does this even do?
const exposure = positions.reduce((a, p) => a + (p.side === 'BUY' ? 1 : -1) * p.qty * p.price, 0)
  * (positions.some((p) => p.symbol.includes('USD')) ? 1 : fxRate);

✅ Хорошо — разбито на именованные, очевидные шаги:

function signedNotional(p: Position): number {
  const direction = p.side === 'BUY' ? 1 : -1;
  return direction * p.qty * p.price;
}

function totalExposure(positions: Position[], fxRate: number): number {
  const gross = positions.reduce((sum, p) => sum + signedNotional(p), 0);
  const isUsdBook = positions.some((p) => p.symbol.includes('USD'));
  return isUsdBook ? gross : gross * fxRate;
}

Баланс: KISS — это не «упрости до тупизны». Хорошо названная цепочка map/filter может быть проще, чем императивный цикл. Цель — наименее удивляющий код, а не наименьшее число символов.

Контекст FinTech: расчёты экспозиции и P&L читают риск-менеджмент и комплаенс, а не только инженеры. Именованный, пошаговый код можно проверять не-авторам — в этом и смысл.

YAGNI — You Aren't Gonna Need It (Тебе это не понадобится)

You Aren't Gonna Need It (YAGNI) — не создавай функциональность, пока она реально не потребовалась.

Что это значит: сопротивляйся спекулятивной обобщённости — настраиваемым движкам, системам плагинов и слоям абстракции, добавленным «на всякий случай» под будущее требование.

Какую проблему решает: спекулятивный код — это неиспользуемая сложность, которую всё равно приходится читать, тестировать и сопровождать. Большинство предсказанных требований так и не появляются или появляются в форме, которую ты не предвидел.

❌ Плохо — спекулятивный, чрезмерно обобщённый конфигурируемый движок ради одного текущего сценария:

// We only ever place market orders today, but "what if"...
interface OrderPipelineStep<T> { run(ctx: T): T; }

class ConfigurableOrderEngine<T extends Order> {
  private steps: OrderPipelineStep<T>[] = [];
  registerStep(step: OrderPipelineStep<T>): this { this.steps.push(step); return this; }
  // generic plugin pipeline, dynamic dispatch, options object...
  execute(order: T, options?: Record<string, unknown>): T {
    return this.steps.reduce((o, s) => s.run(o), order);
  }
}
// 80 lines of framework — to send one kind of order.

✅ Хорошо — строй ровно то, что нужно сейчас; обобщай, когда реально появится второй случай:

@Injectable({ providedIn: 'root' })
export class OrderService {
  private http = inject(HttpClient);

  placeMarketOrder(order: Order): Observable<Order> {
    return this.http.post<Order>('/api/orders', order);
  }
}
// When limit/stop orders are real requirements, refactor THEN — with real constraints.

Контекст FinTech: требования меняются с регуляцией и запусками продуктов. Построить путь рыночного ордера сейчас и расширить его, когда лимитные ордера реально будут проработаны, лучше, чем спекулятивный движок, который угадывает неверно.

WET — Write Everything Twice (Пиши всё дважды, антипаттерн)

Write Everything Twice (WET) — ироничная противоположность DRY: одно и то же знание копипастится и сопровождается в нескольких местах. (Также расшифровывают как «We Enjoy Typing» / «Waste Everyone's Time».)

Что это значит: WET-код дублирует логику так, что одно концептуальное изменение требует правок в нескольких местах — и одно из них неизбежно пропускают.

Нюанс — когда немного дублирования допустимо: преждевременная абстракция может быть хуже дублирования, потому что неправильная абстракция связывает несвязанный код. Прагматичный ориентир — Правило трёх:

❌ WET — один и тот же расчёт спреда, набранный в трёх компонентах, гарантированно разойдётся:

// ticker.component.ts
const spread = quote.ask - quote.bid;
// order-book.component.ts
const spread = quote.ask - quote.bid; // duplicated
// risk-panel.component.ts
const spread = quote.ask - quote.bid; // duplicated again — now extract it

✅ DRY — как только это появляется в третий раз, выноси единый источник истины:

// quote-utils.ts
export function spread(quote: Quote): number {
  return quote.ask - quote.bid;
}
// All three components now call spread(quote).

Правило трёх: в первый раз, когда пишешь что-то, — просто напиши. Во второй раз, увидев дублирование, поморщись, но стерпи. На третий раз — рефактори. Два вхождения могут быть совпадением; три сигнализируют о реальном общем понятии, достойном имени.

Контекст FinTech: «похожее на вид» дублирование между, скажем, валидацией Order и валидацией Position может быть совпадением — насильственное сведение их в одну абстракцию может переплести два контракта, которые должны развиваться раздельно. Дождись третьего, настоящего повтора.

Вопросы и ответы — Вопросы с собеседования

1. Что на практике означает принцип единственной ответственности для Angular-компонента?

У компонента должна быть одна причина для изменения — обычно это отображение состояния. HTTP/транспорт принадлежит инъектируемому сервису, форматирование — пайпу, а бизнес-правила — чистым функциям или доменным сервисам. Так что компонент, который получает позиции по HTTP и форматирует валюту и управляет состоянием представления, имеет три причины для изменения; ты разбиваешь его на PositionService, NotionalPipe и тонкий компонент. Выгода — независимая тестируемость и переиспользование.

2. Как применить Open/Closed, когда switch по типу инструмента продолжает расти?

Замени условный оператор полиморфизмом. Определи интерфейс FeeStrategy и по одной реализации на тип инструмента, затем находи стратегию в реестре/мапе (или через Angular DI с мульти-провайдером). Добавление нового инструмента означает добавление класса и регистрации — существующий, протестированный FeeCalculator никогда не открывается заново. Это и есть «закрыт для изменения, открыт для расширения».

3. Приведи конкретное нарушение подстановки Лисков и способ его исправить.

База TradeExecutor обещает, что execute() возвращает подтверждённый Order, но подкласс ReadOnlyExecutor вместо этого бросает исключение. Любой код, написанный под базовый тип, падает на этом подтипе — взаимозаменяемость нарушена. Исправление — честно моделировать возможности: read-only-фид реализует интерфейс QuoteReader и просто никогда не реализует TradeExecutor, так что его нельзя передать туда, где требуется исполнение. Характерный запах — подкласс, который бросает исключение, возвращает null или вынуждает вызывающих проверять instanceof.

4. Как связаны разделение интерфейсов и инверсия зависимостей и чем помогает Angular DI?

ISP держит интерфейсы маленькими и ролевыми, чтобы клиенты зависели только от того, что используют; DIP говорит зависеть от этих абстракций, а не от конкретных классов. Вместе: определи узкие интерфейсы (например, QuoteProvider), ссылайся на них через InjectionToken и дай Angular DI поставить конкретную реализацию. Компонент инъектирует QUOTE_FEED и никогда не знает, REST это, WebSocket или тестовый mock — именно это делает его подменяемым и юнит-тестируемым.

5. DRY vs WET — когда немного дублирования на самом деле правильный выбор?

DRY говорит, что каждая часть знания живёт в одном месте; WET — антипаттерн его копипасты. Но преждевременная или неправильная абстракция часто хуже дублирования, потому что связывает код, который должен развиваться независимо. Прагматичное правило — Правило трёх: стерпи первое дублирование, поморщись на втором и вынеси на третьем — к этому моменту повторение доказанно является реальным общим понятием, а не совпадением (например, две валидации, которые лишь выглядят похоже, но должны меняться по разным причинам).

Тест

1. Компонент получает ордера по HTTP, форматирует валюту и управляет состоянием представления. Какой принцип он нарушает наиболее явно?
Три ответственности (транспорт, форматирование, представление) означают три причины для изменения — хрестоматийное нарушение SRP. Разбей на сервис, пайп и тонкий компонент.
2. Ты постоянно правишь растущий switch (instrument.kind), чтобы добавить логику комиссий. Какой рефактор лучше всего удовлетворяет Open/Closed?
Полиморфные стратегии позволяют добавить новый инструмент, добавив класс — существующий калькулятор закрыт для изменения, но открыт для расширения.
3. Подкласс переопределяет execute(), чтобы делать throw new Error('not supported'). Это классический признак нарушения…
Подтипы должны соблюдать контракт базы. Подкласс, который бросает исключение там, где база обещала результат, не взаимозаменяем — это нарушение LSP (и часто указывает на отсутствующее разделение интерфейсов / ISP).
4. Ты написал один и тот же расчёт спреда quote.ask - quote.bid в трёх компонентах. Правило трёх говорит…
Стерпи первые два вхождения (они могут быть совпадением); на третьем общее понятие доказано — выноси его. Прыжок к обобщённому движку нарушил бы YAGNI/KISS.
