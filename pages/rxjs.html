RxJS — операторы и паттерны

Реактивное программирование с Observable: холодные и горячие потоки, четыре вида Subject, pipe(), полный каталог операторов (создание, преобразование, фильтрация, комбинирование, обработка ошибок, утилиты), firstValueFrom и четыре способа избежать утечек памяти. Самая часто разбираемая тема на senior-собеседовании по Angular.

Что такое Observable

Observable — ленивый push-поток, который во времени испускает ноль или более значений, а затем либо завершается (complete), либо завершается с ошибкой (error). Ничего не происходит, пока вы не подпишетесь; после подписки Observable проталкивает значения своему наблюдателю по мере их поступления.

Observer (наблюдатель) — потребитель: объект с тремя колбэками. next(value) получает каждое испущенное значение, error(err) вызывается один раз при сбое (завершая поток), а complete() вызывается один раз при завершении потока (без значения).

Паттерн Observer vs Observable — классический паттерн Observer работает только на проталкивание и не имеет понятия о завершении, отмене или композиции. Observable добавляет терминальный контракт (complete/error), ленивое выполнение, отмену через unsubscribe() и богатую алгебру операторов для композиции потоков.

subscribe() — активирует Observable (запускает его функцию-продюсер), подключая колбэки наблюдателя. Возвращает дескриптор Subscription, представляющий текущее выполнение.

unsubscribe() — вызывается на возвращённой Subscription; останавливает выполнение и освобождает ресурсы (таймеры, слушатели, сокеты). Критически важен для предотвращения утечек памяти.

Контракт next / error / complete

Грамматика строгая: ноль или более уведомлений next, за которыми опционально следует единственный error или единственный complete. После терминального уведомления больше ничего не испускается.

import { Observable, Subscription } from 'rxjs';

interface Quote { symbol: string; bid: number; ask: number; timestamp: number; }

// Producer function receives the subscriber (observer)
const quote$ = new Observable<Quote>((subscriber) => {
  subscriber.next({ symbol: 'EURUSD', bid: 1.0700, ask: 1.0702, timestamp: Date.now() });
  subscriber.next({ symbol: 'EURUSD', bid: 1.0701, ask: 1.0703, timestamp: Date.now() });
  subscriber.complete();                 // terminal — no values after this
  subscriber.next({ symbol: 'X' } as Quote); // ignored: stream already completed
});

const sub: Subscription = quote$.subscribe({
  next:     (q) => console.log('quote', q.bid),
  error:    (err) => console.error('stream error', err),
  complete: () => console.log('done'),
});

sub.unsubscribe(); // stop execution, release resources

Лениво: колбэк-продюсер выше не выполняется, пока не вызван .subscribe(). Объявление Observable не имеет побочных эффектов — именно подписка «спускает курок».

Холодные и горячие Observable

Холодный Observable — продюсер создаётся внутри Observable, поэтому каждый подписчик получает собственное независимое выполнение. Он не стартует до подписки, и каждая подписка заново запускает продюсер с нуля. Пример: http.get() — каждая подписка инициирует новый HTTP-запрос.

Горячий Observable — продюсер живёт вне Observable и является общим; все подписчики видят одно и то же выполнение, а значения испускаются независимо от того, слушает ли кто-то. Подписчики, присоединившиеся поздно, пропускают ранние значения. Пример: WebSocket-фид котировок, события мыши или Subject.

import { Observable, Subject } from 'rxjs';

// COLD — producer (Math.random) created INSIDE; each subscriber differs
const cold$ = new Observable<number>((sub) => sub.next(Math.random()));
cold$.subscribe((v) => console.log('A', v)); // e.g. 0.12
cold$.subscribe((v) => console.log('B', v)); // e.g. 0.87 — different execution

// HOT — producer (the Subject) lives OUTSIDE; subscribers share emissions
const hot$ = new Subject<number>();
hot$.subscribe((v) => console.log('A', v));
hot$.next(0.5);                               // A: 0.5
hot$.subscribe((v) => console.log('B', v));   // B subscribed AFTER 0.5
hot$.next(0.9);                               // A: 0.9, B: 0.9 (shared)
// B never sees 0.5 — it joined late

Торговый сценарий — фид котировок горячий

Рынок испускает котировки непрерывно; подписан ли компонент — неважно. WebSocket с мультикастом на каждый график и тикер изначально горячий. Поздно присоединившиеся компоненты видят только котировки с момента подписки.

import { Observable, share } from 'rxjs';

// A cold http.get() can be made hot/shared so N components reuse one request
const cold = this.http.get<Quote[]>('/api/quotes'); // cold: 3 subs = 3 requests
const shared$ = cold.pipe(share());                 // hot: 3 subs = 1 request

Подводный камень №1 у холодных Observable: двойная подписка на http.get() (например, один раз в TS и один раз через пайп async) инициирует запрос дважды. Используйте shareReplay(1), чтобы мультикастить единственный ответ.

Типы Subject (все четыре)

Subject — это одновременно и Observable, и Observer. Он делает мультикаст: вызов next() проталкивает значение всем текущим подписчикам одновременно. У него нет начального значения, и новые подписчики пропускают любые значения, испущенные до их подписки.

import { Subject } from 'rxjs';

const orders$ = new Subject<number>();
orders$.subscribe((v) => console.log('A', v));
orders$.next(1);                              // A: 1
orders$.next(2);                              // A: 2
orders$.subscribe((v) => console.log('B', v));// B missed 1 and 2
orders$.next(3);                              // A: 3, B: 3

BehaviorSubject(initialValue) — Subject, который хранит текущее значение. Он требует начального значения, и каждый новый подписчик при подписке немедленно получает последнее значение. Идеален для состояния, у которого всегда есть «текущий» ответ (выбранный инструмент, статус авторизации).

import { BehaviorSubject } from 'rxjs';

const selected$ = new BehaviorSubject<string>('EURUSD'); // initial value
selected$.subscribe((v) => console.log('A', v));         // A: EURUSD (immediately)
selected$.next('BTCUSD');                                // A: BTCUSD
selected$.subscribe((v) => console.log('B', v));         // B: BTCUSD (current value!)
console.log(selected$.value);                            // synchronous read: BTCUSD

ReplaySubject(bufferSize) — буферизует последние bufferSize значений и переигрывает их все каждому новому подписчику, как бы поздно он ни присоединился. Без размера буфера переигрывает всю историю. Полезен для «последних N тиков» или кэширования.

import { ReplaySubject } from 'rxjs';

const ticks$ = new ReplaySubject<number>(2);  // buffer last 2
ticks$.next(1);
ticks$.next(2);
ticks$.next(3);
ticks$.subscribe((v) => console.log('A', v)); // A: 2, A: 3 (last 2 replayed)
ticks$.next(4);                               // A: 4

AsyncSubject — испускает только последнее значение и только при вызове complete(). Все более ранние значения отбрасываются. Ведёт себя как Promise, разрешающийся в единственный финальный результат.

import { AsyncSubject } from 'rxjs';

const result$ = new AsyncSubject<number>();
result$.subscribe((v) => console.log('A', v));
result$.next(10);
result$.next(20);
result$.next(30);   // nothing emitted yet
result$.complete(); // NOW A: 30 (only the last value, on complete)

Эвристика senior-уровня: для состояния компонента/сервиса берите BehaviorSubject (или, в Angular 16+, Signal). Для шины событий/действий — обычный Subject. Для «переиграть последнее значение всем, кто присоединится» без начального зерна — ReplaySubject(1).

pipe() — как это работает

pipe() — метод на каждом Observable, который принимает последовательность операторов и применяет их слева направо как композицию функций. Каждый оператор принимает исходный Observable и возвращает новый; исходный никогда не мутируется. source$.pipe(f, g, h) эквивалентно h(g(f(source$))).

import { debounceTime, distinctUntilChanged, filter, switchMap } from 'rxjs/operators';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

// Instrument search: each operator returns a NEW Observable, composed in order
this.searchControl.valueChanges.pipe(
  debounceTime(300),                  // wait for 300ms of silence
  distinctUntilChanged(),             // skip if query unchanged
  filter((q): q is string => !!q && q.length >= 2),
  switchMap((q) => this.api.searchInstruments(q)), // cancel previous search
  takeUntilDestroyed(this.destroyRef),// auto-unsubscribe on destroy
).subscribe((results) => this.results.set(results));

Переиспользуемые цепочки pipe

Поскольку операторы — это просто функции, можно построить переиспользуемый конвейер с помощью отдельной функции pipe() из 'rxjs' и применить его к любому совместимому потоку.

import { pipe, Observable } from 'rxjs';
import { debounceTime, distinctUntilChanged, filter } from 'rxjs/operators';

// A reusable operator: takes string-emitting source, returns cleaned string stream
const searchPipeline = () => pipe(
  debounceTime<string>(300),
  distinctUntilChanged(),
  filter((q: string) => q.length >= 2),
);

// Apply the SAME pipeline to multiple independent streams
const instrumentSearch$ = this.instrumentInput$.pipe(searchPipeline());
const newsSearch$       = this.newsInput$.pipe(searchPipeline());

Возврат pipe(...) из фабрики создаёт переиспользуемый, тестируемый именованный оператор — самый чистый способ соблюсти DRY для повторяющихся цепочек по всей кодовой базе.

Операторы создания

of(...values) — испускает переданные аргументы по порядку, синхронно, затем завершается.

from(input) — преобразует массив, итерируемый объект, Promise или другой Observable в Observable, испускающий каждое значение.

fromEvent(target, eventName) — создаёт (горячий) Observable из DOM-события/события эмиттера, испуская каждый объект события.

interval(period) — испускает возрастающие целые числа (0, 1, 2…) каждые period мс. Сам по себе никогда не завершается.

timer(delay, period?) — испускает 0 через delay; если задан period, продолжает испускать 1, 2, 3… каждые period мс.

EMPTY — Observable, который не испускает значений и немедленно завершается. Полезен как no-op-фолбэк.

NEVER — Observable, который никогда не испускает и никогда не завершается. Редко используется за пределами тестов/заглушек.

throwError(() => err) — создаёт Observable, который немедленно завершается с ошибкой. Принимает фабричную функцию (RxJS 7+), чтобы ошибка создавалась лениво на каждую подписку.

defer(factory) — лениво создаёт Observable на каждого подписчика, вызывая factory в момент подписки, делая жадный источник холодным.

merge(...sources) — подписывается на все источники одновременно и испускает их значения вперемешку по мере поступления.

concat(...sources) — подписывается на источники по порядку; каждый стартует только после завершения предыдущего. Сохраняет порядок.

combineLatest([...sources]) — испускает массив последних значений каждого источника всякий раз, когда любой источник испускает (после того как все испустили хотя бы раз).

forkJoin([...sources]) — ждёт завершения каждого источника, затем испускает единственный массив их последних значений. Аналог Promise.all в RxJS.

zip([...sources]) — объединяет n-е значение каждого источника в массив; испускает n-й кортеж только когда каждый источник произвёл своё n-е значение.

import {
  of, from, fromEvent, interval, timer, EMPTY, NEVER,
  throwError, defer, merge, concat, combineLatest, forkJoin, zip,
} from 'rxjs';

of('EURUSD', 'BTCUSD');                       // → EURUSD, BTCUSD, complete
from([1, 2, 3]);                              // → 1, 2, 3
from(fetch('/api/quotes'));                   // Promise → Observable
fromEvent(window, 'resize');                  // hot stream of resize events
interval(1000);                               // 0, 1, 2… every second
timer(2000, 1000);                            // 0 after 2s, then 1, 2… each 1s
EMPTY;                                         // completes, emits nothing
NEVER;                                         // emits nothing, never completes
throwError(() => new Error('feed down'));      // errors on subscribe
defer(() => of(Date.now()));                   // re-evaluated per subscriber

// Combine three quote sources
merge(fx$, crypto$, equities$);               // interleaved as they arrive
concat(historical$, live$);                   // history first, THEN live feed
combineLatest([bid$, ask$]);                  // [latestBid, latestAsk] on any emit
forkJoin([instrument$, position$, quote$]);   // all complete → [i, p, q]
zip([symbol$, price$]);                        // [sym0, px0], [sym1, px1]…

Ловушка throwError: в RxJS 7+ передавайте фабрику throwError(() => err), а не саму ошибку. Передача значения напрямую устарела и вычисляется жадно.

Операторы преобразования

map(fn) — преобразует каждое испущенное значение функцией проекции и испускает результат. Полностью синхронный и не подписывается на внутренние Observable (для этого нужны flattening-операторы *Map).

import { map } from 'rxjs/operators';
quote$.pipe(map((q: Quote) => q.ask - q.bid)); // emit the spread per quote

Flattening-операторы (высшего порядка)

Эти операторы берут значение, отображают его во внутренний Observable, подписываются на него и «расплющивают» его эмиссии в выходной поток. Они различаются тем, как обрабатывают конкурентные внутренние Observable — это самое важное различие в RxJS.

switchMap(fn)

switchMap — отображает в внутренний Observable и отменяет (отписывается от) предыдущий внутренний всякий раз, когда поступает новое значение источника. Выживает только самый последний внутренний поток.

  • Когда использовать: автодополнение / поиск инструмента, изменения параметров маршрута — везде, где важен только новейший результат.
  • Подводный камень: не используйте для параллельных независимых запросов — он отменяет нужную вам работу «на лету».
import { switchMap } from 'rxjs/operators';
search$.pipe(
  switchMap((q: string) => this.api.searchInstruments(q)), // cancels stale search
);

mergeMap(fn) (он же flatMap)

mergeMap — отображает в внутренний Observable и сливает их все конкурентно, без отмены. Каждый внутренний работает до завершения параллельно.

  • Когда использовать: запустить N независимых запросов параллельно — загрузка нескольких файлов, запись во множество эндпоинтов.
  • Подводный камень: порядок вывода не гарантирован; побеждает тот внутренний, что испустит первым. Также может перегрузить сервер (ограничьте аргументом concurrent).
import { mergeMap } from 'rxjs/operators';
files$.pipe(
  mergeMap((file: File) => this.api.upload(file), 4), // up to 4 concurrent uploads
);

concatMap(fn)

concatMap — отображает в внутренний Observable и выполняет их строго последовательно, ставя каждое новое значение источника в очередь до завершения предыдущего внутреннего. Сохраняет порядок.

  • Когда использовать: упорядоченная очередь отправки ордеров, где каждый запрос должен завершиться до старта следующего.
  • Подводный камень: если внутренний Observable никогда не завершается (например, interval), очередь зависает навсегда — ничего после него не выполняется.
import { concatMap } from 'rxjs/operators';
orderQueue$.pipe(
  concatMap((order: Order) => this.api.placeOrder(order)), // one at a time, in order
);

exhaustMap(fn)

exhaustMap — отображает в внутренний Observable и игнорирует новые значения источника, пока текущий внутренний ещё активен. Текущее выполнение должно завершиться («исчерпаться»), прежде чем сможет стартовать новое.

  • Когда использовать: кнопки отправки / входа — игнорировать двойные клики, пока запрос в полёте.
  • Подводный камень: значения, поступающие во время активного внутреннего, полностью отбрасываются (не ставятся в очередь), что может ощущаться как неотзывчивость при неправильном применении.
import { exhaustMap } from 'rxjs/operators';
submitClick$.pipe(
  exhaustMap(() => this.api.placeOrder(this.form.value)), // ignore re-clicks
);

scan, buffer, pairwise, groupBy, expand

scan(accumulator, seed) — как Array.reduce, но испускает промежуточное накопленное значение после каждой эмиссии источника, а не только финальное. Идеален для нарастающих итогов.

import { scan } from 'rxjs/operators';

// Running P&L — emits the cumulative total on every trade
trade$.pipe(
  scan((pnl: number, t: { pnl: number }) => pnl + t.pnl, 0),
);

// Keep a rolling history of the last 100 ticks
tick$.pipe(
  scan((hist: number[], px: number) => [...hist, px].slice(-100), []),
);

buffer(closingNotifier$) — собирает испущенные значения в массив и испускает этот массив каждый раз, когда испускает Observable-уведомитель.

bufferTime(ms) — собирает значения в массивы, испуская каждую пачку каждые ms миллисекунд. Отлично подходит для пакетирования высокочастотных тиков.

pairwise() — испускает предыдущее и текущее значение вместе как кортеж [prev, curr] (ничего не выводит до второго значения). Идеален для вычисления дельт.

groupBy(keyFn) — разбивает источник на несколько внутренних Observable по ключу keyFn, по одному на каждый уникальный ключ.

expand(fn) — рекурсивно отображает каждое значение во внутренний Observable и подаёт его эмиссии обратно в fn — используется для рекурсивной/постраничной выборки.

import { buffer, bufferTime, pairwise, groupBy, expand, map, mergeMap, takeWhile } from 'rxjs/operators';

tick$.pipe(buffer(flush$));            // emit collected ticks when flush$ fires
tick$.pipe(bufferTime(1000));         // emit an array of ticks each second

price$.pipe(
  pairwise(),                          // [prev, curr]
  map(([prev, curr]) => curr - prev),  // per-tick delta
);

order$.pipe(
  groupBy((o: Order) => o.symbol),     // one inner stream per symbol
  mergeMap((group$) => group$.pipe(/* per-symbol logic */)),
);

// Paginated fetch: keep requesting next page until empty
this.api.getPage(0).pipe(
  expand((page) => page.hasNext ? this.api.getPage(page.next) : EMPTY),
  takeWhile((page) => page.items.length > 0, true),
);

Операторы фильтрации

filter(predicate) — испускает только значения, для которых предикат вернул true.

take(n) — испускает первые n значений, затем автоматически завершается.

takeUntil(notifier$) — испускает значения источника, пока notifier$ не испустит своё первое значение, затем завершается. Классический механизм отписки.

takeWhile(predicate) — испускает, пока предикат true; завершается, как только он вернёт false (опционально включив несоответствующее значение).

takeUntilDestroyed(destroyRef?) — оператор Angular 16+, завершающий поток при уничтожении компонента/директивы/сервиса. Вызванный внутри контекста внедрения, аргумента не требует; в остальных случаях передавайте DestroyRef.

skip(n) — игнорирует первые n значений, затем испускает остальные.

skipUntil(notifier$) — игнорирует значения источника, пока notifier$ не испустит, затем пропускает всё дальше.

debounceTime(ms) — ждёт ms тишины после значения, затем испускает самое последнее. Отбрасывает промежуточные значения. Идеален для полей поиска.

throttleTime(ms) — испускает первое значение, затем игнорирует последующие в течение ms миллисекунд. Идеален для прокрутки/изменения размера/ограничения частоты.

distinctUntilChanged(compareFn?) — испускает значение, только если оно отличается от предыдущего испущенного (по === или пользовательскому компаратору).

first(predicate?) — испускает первое значение (опционально соответствующее предикату), затем завершается; завершается с ошибкой EmptyError, если ничего не подошло.

last(predicate?) — испускает последнее значение (опционально соответствующее предикату) после завершения источника.

elementAt(n) — испускает только значение по индексу n, затем завершается.

import {
  filter, take, takeUntil, takeWhile, skip, skipUntil,
  debounceTime, throttleTime, distinctUntilChanged, first, last, elementAt,
} from 'rxjs/operators';

quote$.pipe(filter((q: Quote) => q.ask - q.bid < 0.0005)); // tight spreads only
quote$.pipe(take(1));                          // first quote, then complete
quote$.pipe(takeUntil(this.destroy$));         // until destroy$ fires
price$.pipe(takeWhile((p: number) => p < 100)); // while below 100
tick$.pipe(skip(1));                           // drop the first (stale) tick

// Search input: wait for silence, dedupe identical queries
this.search$.pipe(
  debounceTime(300),
  distinctUntilChanged(),
);

scroll$.pipe(throttleTime(200));               // at most one event / 200ms
quote$.pipe(first((q: Quote) => q.bid > 1.07)); // first crossing 1.07

debounce vs throttle: debounce ждёт паузу, затем испускает последнее значение (поле поиска). throttle испускает сразу, затем ограничивает частоту (обработчик прокрутки). Они решают противоположные концы одной задачи.

Операторы комбинирования

combineLatest([a$, b$]) — испускает массив последних значений каждого входа всякий раз, когда любой вход испускает, но только после того как каждый вход испустил хотя бы раз.

forkJoin([a$, b$]) — ждёт завершения всех входов, затем испускает единственный массив их последних значений. Observable-эквивалент Promise.all.

zip([a$, b$]) — спаривает значения по индексу: испускает [a0, b0], затем [a1, b1]… только когда каждый вход произвёл этот индекс.

withLatestFrom(other$) — для каждой эмиссии источника прикрепляет последнее значение из other$, не используя other$ как триггер. Эмиссии управляются только источником.

startWith(value) — добавляет одно или несколько синхронных значений перед началом испускания источника. Полезно для инициализации стартового состояния UI.

import { combineLatest, forkJoin, zip } from 'rxjs';
import { withLatestFrom, startWith, map } from 'rxjs/operators';

// Recompute a derived total whenever selected symbol OR its quote changes
combineLatest([selectedSymbol$, quote$]).pipe(
  map(([symbol, quote]) => ({ symbol, mid: (quote.bid + quote.ask) / 2 })),
);

// Load everything a dashboard needs, once, in parallel
forkJoin({
  instrument: this.api.getInstrument(id),
  position:   this.api.getPosition(id),
  quote:      this.api.getQuote(id),
}); // emits { instrument, position, quote } when all complete

zip([orderId$, fillPrice$]);                   // pair the nth id with nth fill

// On each submit click, grab the latest form value without re-triggering
submit$.pipe(withLatestFrom(formValue$));

connectionStatus$.pipe(startWith('connecting')); // seed UI before first event

Обработка ошибок

catchError((err, caught) => Observable) — перехватывает уведомление об ошибке и обязан вернуть новый Observable для продолжения. Верните of(fallback) для аккуратного восстановления или throwError(() => err) для повторного выброса.

retry(count) — при ошибке переподписывается на источник до count раз, прежде чем сдаться и пробросить ошибку.

retryWhen(notifier => Observable) — управляемый повтор, управляемый Observable-уведомителем. Устарел в RxJS 7; предпочтите retry({ count, delay }).

retry({ count, delay }) — современный, настраиваемый повтор: повторяет count раз с ожиданием delay мс (или функцией, возвращающей уведомитель) между попытками.

import { of, throwError } from 'rxjs';
import { catchError, retry } from 'rxjs/operators';

// 1. Recover with a fallback value — stream CONTINUES
this.api.getQuote(id).pipe(
  catchError((err: unknown) => {
    console.error('quote failed', err);
    return of({ symbol: id, bid: 0, ask: 0, timestamp: Date.now() } as Quote);
  }),
);

// 2. Rethrow after logging — stream ERRORS
this.api.placeOrder(order).pipe(
  catchError((err: unknown) => {
    this.logger.error(err);
    return throwError(() => err); // propagate to the subscriber's error callback
  }),
);

// 3. Modern retry with backoff (RxJS 7+)
this.api.getQuote(id).pipe(
  retry({ count: 3, delay: 2000 }), // up to 3 retries, 2s apart
);

// retryWhen is deprecated — do NOT reach for it in new code
// (use retry({ count, delay }) instead)

Критично: необработанная ошибка навсегда завершает поток — больше ни одного значения. На долгоживущем фиде всегда вызывайте catchError на внутреннем Observable (внутри switchMap/mergeMap), чтобы единичный неудавшийся запрос не убил всю подписку.

Утилитарные операторы

tap(observerOrFn) — выполняет побочные эффекты (логирование, аналитика, установка флага загрузки) для каждого уведомления, не изменяя проходящие значения.

delay(ms) — сдвигает во времени каждую эмиссию на ms миллисекунд.

timeout(ms) — завершается с ошибкой TimeoutError, если источник не испускает в течение ms миллисекунд после подписки (или между эмиссиями).

finalize(fn) — запускает колбэк при завершении потока по любой причине — complete или error или отписка. Идеален для скрытия спиннера.

shareReplay(bufferSize) — делает мультикаст источника всем подписчикам и переигрывает последние bufferSize значений поздним подписчикам. Превращает холодный HTTP-вызов в общий, закэшированный.

share() — делает мультикаст источника, чтобы подписчики разделяли одно выполнение, но без переигрывания — поздние подписчики пропускают прежние значения.

import { tap, delay, timeout, finalize, shareReplay, share, catchError } from 'rxjs/operators';
import { of } from 'rxjs';

this.loading.set(true);
this.api.getInstrument(id).pipe(
  tap((i) => console.log('loaded', i.symbol)), // side effect only
  timeout(5000),                               // error if > 5s with no response
  catchError(() => of(null)),
  finalize(() => this.loading.set(false)),      // ALWAYS hide spinner
).subscribe();

// Share ONE HTTP response across many components (cold → hot + cached)
readonly instruments$ = this.http.get<Instrument[]>('/api/instruments').pipe(
  shareReplay(1),                              // 1 request, replayed to all
);

// share() — multicast a live feed, no replay for late joiners
readonly liveQuotes$ = this.socket.quotes$.pipe(share());

shareReplay refCount: используйте shareReplay({ bufferSize: 1, refCount: true }), если хотите, чтобы базовый источник отписывался (а кэш сбрасывался) после ухода последнего подписчика — важно для долгоживущих фидов, чтобы не держать сокет открытым вечно.

firstValueFrom / lastValueFrom

firstValueFrom(observable$) — подписывается, разрешает Promise первым испущенным значением, затем автоматически отписывается. Отклоняется с EmptyError, если Observable завершается ничего не испустив. Современная замена устаревшему .toPromise().

lastValueFrom(observable$) — подписывается и разрешает Promise последним испущенным значением после завершения Observable. Также отклоняется с EmptyError, если ничего не было испущено.

Когда использовать: в guard-ах и резолверах маршрутов, желающих async/await, при взаимодействии с кодом на Promise или для однократных чтений (одиночный HTTP GET). Избегайте для долгоживущих потоков, которые никогда не завершаются (lastValueFrom зависнет).

import { firstValueFrom, lastValueFrom } from 'rxjs';
import { inject } from '@angular/core';
import { CanActivateFn, ResolveFn, Router } from '@angular/router';

// Functional auth guard using async/await via firstValueFrom
export const authGuard: CanActivateFn = async () => {
  const auth = inject(AuthService);
  const router = inject(Router);
  const isAuthed = await firstValueFrom(auth.isAuthenticated$); // first value, auto-unsub
  return isAuthed ? true : router.createUrlTree(['/login']);
};

// Resolver pre-fetching an instrument before the route activates
export const instrumentResolver: ResolveFn<Instrument> = (route) => {
  const api = inject(InstrumentService);
  const id = route.paramMap.get('id')!;
  return firstValueFrom(api.getInstrument(id)); // HTTP completes → resolves
};

// lastValueFrom — wait for a completing stream's final value
const finalPrice = await lastValueFrom(this.api.getQuote('EURUSD'));

Утечки памяти и отписка

Утечка памяти (RxJS) — подписка, которая никогда не уничтожается. Её продюсер (таймер, слушатель событий, сокет, HTTP) продолжает работать, а замыкание удерживает экземпляр компонента живым после того, как Angular его уничтожил, что приводит к утечке памяти и дублирующей/зомби-работе (например, обнаружение изменений на мёртвом компоненте).

1. takeUntil(destroy$) — классический паттерн

import { Subject } from 'rxjs';
import { takeUntil } from 'rxjs/operators';

export class QuotesComponent implements OnInit, OnDestroy {
  private readonly destroy$ = new Subject<void>();

  ngOnInit(): void {
    this.quotes.stream$.pipe(
      takeUntil(this.destroy$), // place LAST so everything above is torn down
    ).subscribe((q) => this.render(q));
  }

  ngOnDestroy(): void {
    this.destroy$.next();
    this.destroy$.complete();
  }
}

Работает везде, но многословен и подвержен ошибкам (забыли ngOnDestroy — получили утечку).

2. takeUntilDestroyed(destroyRef) — Angular 16+ (чище)

takeUntilDestroyed — автоматически завершает поток при уничтожении текущего контекста внедрения (компонент/директива/сервис), убирая весь шаблонный код с destroy$.

import { Component, DestroyRef, inject } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

export class QuotesComponent {
  private readonly destroyRef = inject(DestroyRef);

  constructor() {
    // In an injection context (constructor) you can omit the argument:
    this.quotes.stream$
      .pipe(takeUntilDestroyed()) // no destroyRef needed in constructor
      .subscribe((q) => this.render(q));
  }

  start(): void {
    // Outside the injection context, pass the DestroyRef explicitly:
    this.quotes.stream$
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe((q) => this.render(q));
  }
}

3. Пайп async в шаблоне — ЛУЧШИЙ

async-пайп — подписывается на Observable прямо в шаблоне, автоматически отписывается при уничтожении компонента и запускает обнаружение изменений на каждое новое значение (бесшовно работает с OnPush). Никакой ручной подписки, утечка невозможна.

@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    @if (quote$ | async; as quote) {
      <span>{{ quote.bid }} / {{ quote.ask }}</span>
    }
  `,
})
export class TickerComponent {
  readonly quote$ = this.quotes.stream$; // Angular manages the lifecycle
}

Предпочитайте этот способ всегда, когда возможно. Angular владеет жизненным циклом подписки, поэтому утечки структурно невозможны, а обнаружение изменений при OnPush обрабатывается за вас.

4. Ручной unsubscribe() — хрупкий

import { Subscription } from 'rxjs';

export class QuotesComponent implements OnDestroy {
  private sub?: Subscription;

  start(): void {
    this.sub = this.quotes.stream$.subscribe((q) => this.render(q));
  }

  ngOnDestroy(): void {
    this.sub?.unsubscribe(); // easy to forget; multiplies with many streams
  }
}

Приемлем для одной подписки, но не масштабируется — ручное управление множеством полей Subscription — это ровно тот путь, по которому проникают утечки.

Вопросы и ответы — вопросы с собеседования

1. В чём разница между холодным и горячим Observable?

Холодный Observable создаёт свой продюсер внутри, поэтому каждый подписчик получает независимое выполнение, стартующее при подписке — http.get() инициирует новый запрос на каждую подписку. Горячий Observable разделяет один продюсер, живущий вне; все подписчики видят одни и те же эмиссии, которые происходят независимо от подписчиков, поэтому поздно присоединившиеся пропускают ранние значения — например, WebSocket-фид котировок или Subject. Холодный можно превратить в горячий через share()/shareReplay().

2. switchMap vs mergeMap vs concatMap vs exhaustMap — когда какой использовать?

switchMap отменяет предыдущий внутренний — для автодополнения/поиска, где важен только последний. mergeMap запускает внутренние конкурентно без гарантии порядка — параллельные загрузки. concatMap ставит внутренние в очередь последовательно, сохраняя порядок — очередь отправки ордеров. exhaustMap игнорирует новые значения, пока внутренний активен — кнопка отправки/входа, игнорирующая двойные клики. Ключевая ловушка: switchMap для запроса сохранения может отменить сохранение; concatMap с незавершающимся внутренним замораживает очередь.

3. Сравните четыре типа Subject.

Subject: без начального значения, новые подписчики пропускают прежние эмиссии — шина событий. BehaviorSubject: требует начальное значение, переигрывает последнее значение новым подписчикам, открывает .value — текущее состояние. ReplaySubject(n): переигрывает последние n значений любому новому подписчику — история последних N. AsyncSubject: испускает только финальное значение и только при complete() — единственный результат. Для состояния в Angular обычно выбирают BehaviorSubject (или Signal).

4. Как предотвратить утечки памяти из-за подписок?

Четыре подхода: async-пайп (лучший — Angular авто-отписывается при уничтожении и обрабатывает OnPush CD); takeUntilDestroyed() (Angular 16+, чище всего для императивных подписок); takeUntil(destroy$) с Subject, испускаемым в ngOnDestroy (классический паттерн до 16-й версии, ставьте его последним в pipe); и ручной unsubscribe() (хрупкий, только для одной подписки). Утечка возникает потому, что продюсер удерживает уничтоженный компонент живым через замыкание подписки.

5. Что на самом деле делает pipe()?

pipe() компонует операторы слева направо как композицию функций: source$.pipe(f, g, h) равно h(g(f(source$))). Каждый оператор — функция, принимающая исходный Observable и возвращающая новый; исходный никогда не мутируется. Поскольку операторы — обычные функции, можно вынести переиспользуемую цепочку с помощью отдельной функции pipe() из 'rxjs' и применить её к нескольким потокам.

6. catchError — как одновременно восстановиться и пробросить ошибку, и где его размещать?

catchError обязан вернуть новый Observable. Чтобы восстановиться, верните of(fallback), чтобы поток продолжился со значением по умолчанию. Чтобы пробросить, верните throwError(() => err), чтобы сработал error-колбэк подписчика. Размещение важно: ошибка навсегда завершает поток, поэтому на долгоживущем фиде ставьте catchError на внутренний Observable внутри switchMap/mergeMap — так единичный неудавшийся запрос не убьёт внешнюю подписку.

7. firstValueFrom vs lastValueFrom vs старый toPromise()?

firstValueFrom разрешается первым значением, затем авто-отписывается; lastValueFrom разрешается последним значением после завершения источника. Оба бросают EmptyError, если ничего не испущено. Они заменяют устаревший .toPromise(), который неоднозначно разрешался в undefined для пустых потоков. Используйте их в guard-ах/резолверах, желающих async/await, и для однократных чтений; никогда не используйте lastValueFrom на потоке, который никогда не завершается — он зависнет.

8. Когда использовать combineLatest vs forkJoin vs zip?

combineLatest испускает последнее значение каждого источника на каждую эмиссию (после первой эмиссии всех) — живое производное состояние вроде mid-цены из bid$ и ask$. forkJoin ждёт завершения всех источников, затем испускает их последние значения один раз — параллельные разовые загрузки (как Promise.all). zip спаривает значения по индексу. Ловушка forkJoin: если какой-то источник никогда не завершается, он никогда не испустит, поэтому используйте его только с завершающимися источниками (HTTP) или добавьте take(1).

Тест

1. Какой оператор отменяет предыдущий внутренний Observable при поступлении нового значения?
switchMap отписывается от предыдущего внутреннего на каждое новое значение источника — идеален для автодополнения. mergeMap работает параллельно, concatMap ставит в очередь, exhaustMap игнорирует новые, пока активен.
2. Новый подписчик на BehaviorSubject немедленно получает…
BehaviorSubject хранит текущее значение и переигрывает его (1) каждому новому подписчику. «Все предыдущие» — это ReplaySubject; «только при complete» — AsyncSubject; «ничего до следующей» — обычный Subject.
3. Какой способ ЛУЧШЕ всего избегает утечек памяти из подписок для значения, привязанного в шаблоне?

4. forkJoin([a$, b$]) испускает…
forkJoin — это Observable-аналог Promise.all: он ждёт ЗАВЕРШЕНИЯ каждого источника, затем испускает массив их последних значений один раз. «Каждая эмиссия» — это combineLatest; «спаренные по индексу» — zip.
5. Что делает firstValueFrom(obs$), если obs$ завершается ничего не испустив?
firstValueFrom/lastValueFrom отклоняются с EmptyError на пустом потоке — в отличие от устаревшего toPromise(), который неоднозначно разрешался в undefined.
