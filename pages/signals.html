Angular Signals

Реактивный примитив, который меняет облик Angular: signal, computed, effect, мост интеграции с RxJS (toSignal/toObservable), сигнальные inputs и models, а также новое хранилище @ngrx/signals.

Что такое Signals

Signal (сигнал) — реактивный примитив, который хранит значение и уведомляет своих потребителей при каждом его изменении. Сигналы синхронны, у них всегда есть текущее значение, а чтобы прочитать сигнал, его вызывают как функцию (price()). Появились в Angular 16.

Граф реактивности — сигналы, вычисляемые значения и эффекты образуют граф зависимостей. Когда сигнал меняется, повторно вычисляются только те потребители, которые его действительно читают (точечная, fine-grained реактивность), а не пересчитывается всё дерево компонентов.

Writable signal (изменяемый сигнал) — сигнал, который можно изменить через .set() или .update().

Read-only signal (только для чтения) — производное значение (из computed или интеграции), которое можно читать, но нельзя задать напрямую.

Потребитель — всё, что читает сигнал: computed, effect или привязка в шаблоне. Чтение автоматически устанавливает зависимость.

Зачем нужны сигналы

  • Проще RxJS для локального состояния — никаких потоков, операторов и marble-диаграмм ради счётчика или переключателя.
  • Не нужно управлять подписками — отписываться не от чего; нет утечек памяти из-за забытых подписок.
  • Точечная реактивность — обновляются ровно те привязки, которые зависят от изменившегося сигнала, а не весь компонент.
  • Будущее change detection в Angular — zoneless-режим опирается на сигналы, чтобы точно знать, что нужно обновить.
  • Всегда синхронны и без «глитчей» — чтение всегда возвращает актуальное согласованное значение.
import { signal } from '@angular/core';

const price = signal<number>(100); // create a writable signal
console.log(price());              // read → 100 (call it like a function)
price.set(105);                    // write → consumers notified
console.log(price());              // 105

Чтение сигнала внутри реактивного контекста (шаблон, computed или effect) автоматически подписывает на него. Чтение в обычном императивном коде просто возвращает текущее значение.

Базовый API

signal<T>(initialValue) — изменяемый сигнал

signal<T>() — создаёт изменяемый сигнал с начальным значением initialValue. Возвращает WritableSignal<T> с тремя методами записи и одним чтением.

  • .set(newValue) — полностью заменяет текущее значение.
  • .update(fn) — вычисляет следующее значение на основе текущего: (current) => next.
  • .mutate(fn) — изменяет значение на месте. Объявлен устаревшим / удалён в новых версиях Angular — предпочитайте .update() с неизменяемой копией.
  • Вызов price() — читает текущее значение и (в реактивном контексте) регистрирует зависимость.
import { signal, WritableSignal } from '@angular/core';

interface Order { id: number; symbol: string; qty: number; }

const qty: WritableSignal<number> = signal(100);

qty.set(250);                         // replace
qty.update(current => current + 50);  // derive from current → 300
console.log(qty());                   // 300

// Immutable update for object/array signals (preferred over .mutate)
const orders = signal<Order[]>([]);
orders.update(list => [...list, { id: 1, symbol: 'EURUSD', qty: 100 }]);

.mutate устарел. Он менял ту же самую ссылку, что ненадёжно для change detection, и был удалён. Всегда создавайте новый массив/объект внутри .update().

computed(fn) — производный сигнал

computed() — создаёт только для чтения производный сигнал. Он выполняет fn, автоматически отслеживает каждый прочитанный внутри сигнал как зависимость и мемоизирует результат — пересчитывая его только при изменении одной из зависимостей. Ленивый: не вычисляется до первого чтения.

import { signal, computed } from '@angular/core';

const price = signal(100);
const tax = signal(0.2);

// auto-tracks price() and tax(); recomputes only when either changes
const total = computed(() => price() * (1 + tax()));

console.log(total()); // 120
price.set(200);
console.log(total()); // 240 — memoized recompute
// total.set(...) ❌ does not exist — computed signals are read-only

Поскольку computed мемоизирован, чтение total() десять раз выполнит функцию один раз — пока не изменится зависимость.

effect(fn) — реактивные побочные эффекты

effect() — регистрирует побочный эффект, который перезапускается при изменении любого прочитанного им сигнала. Он выполняется хотя бы раз, запускается после рендеринга и должен создаваться в контексте внедрения (конструктор или инициализатор поля). Возвращает EffectRef с методом .destroy(); также автоматически очищается при уничтожении его инжектора.

Базовый эффект — авто-отслеживание

import { Component, signal, effect } from '@angular/core';

@Component({ selector: 'app-ticker', template: '' })
export class TickerComponent {
  readonly price = signal(100);

  constructor() {
    // re-runs every time price() changes (after render)
    effect(() => console.log('price is now', this.price()));
  }
}

Эффект с onCleanup

onCleanup(fn) — регистрируется внутри эффекта; выполняется перед следующим запуском и при уничтожении. Идеален для очистки таймеров, интервалов и слушателей.

import { signal, effect } from '@angular/core';

const pollMs = signal(1000);

effect((onCleanup) => {
  const id = setInterval(() => console.log('poll quote'), pollMs());
  onCleanup(() => clearInterval(id)); // clears before re-run / on destroy
});

allowSignalWrites — запись сигнала внутри эффекта

allowSignalWrites — опция, разрешающая эффекту вызывать .set()/.update() у сигнала. По умолчанию запись внутри эффекта выбрасывает ошибку, чтобы не плодить скрытые каскады.

import { signal, effect } from '@angular/core';

const bid = signal(1.07);
const lastBid = signal(0);

effect(() => {
  // writing a DIFFERENT signal than the one we read
  lastBid.set(bid());
}, { allowSignalWrites: true });

untracked() — чтение без зависимости

untracked(fn) — читает сигналы внутри fn без регистрации их как зависимостей. Используйте, когда эффект должен реагировать на один сигнал, но лишь читать другой для контекста.

import { signal, effect, untracked } from '@angular/core';

const symbol = signal('EURUSD');
const requestId = signal(0);

effect(() => {
  const id = requestId();                 // depend on requestId
  const sym = untracked(() => symbol());  // read but do NOT depend
  console.log('request', id, 'for', sym);
});

Подводные камни:

  • Запись сигнала внутри эффекта без allowSignalWrites выбрасывает ошибку.
  • Чтение и запись одного и того же сигнала в одном эффекте создаёт бесконечный цикл (каждая запись перезапускает эффект).
  • Для производного состояния предпочитайте computed — прибегайте к эффекту только тогда, когда нужен настоящий побочный эффект (логирование, DOM, localStorage, аналитика).

toSignal(observable$) — Observable → Signal

toSignal() — подписывается на Observable и предоставляет его последнее значение как сигнал только для чтения. Должен вызываться в контексте внедрения; автоматически отписывается при уничтожении контекста. Передайте initialValue (или requireSync), чтобы определить значение до первой эмиссии.

import { Component, inject, Signal } from '@angular/core';
import { toSignal } from '@angular/core/rxjs-interop';

interface Quote { symbol: string; bid: number; ask: number; timestamp: number; }

@Component({ selector: 'app-quotes', template: '' })
export class QuotesComponent {
  private readonly ws = inject(WsService);

  // Observable<Quote[]> → Signal<Quote[]>
  readonly quotes: Signal<Quote[]> =
    toSignal(this.ws.quotes$, { initialValue: [] });
}

toSignal — это мост из потоковых данных (HTTP, WebSocket) в мир сигналов, чтобы шаблоны могли читать их без пайпа async.

toObservable(signal) — Signal → Observable

toObservable() — преобразует сигнал в Observable, который эмитит при каждом изменении сигнала (внутренне управляется эффектом). Полезно, когда нужны операторы RxJS (debounceTime, switchMap) поверх состояния-сигнала.

import { signal } from '@angular/core';
import { toObservable } from '@angular/core/rxjs-interop';
import { debounceTime, switchMap } from 'rxjs';

const search = signal('');

const results$ = toObservable(search).pipe(
  debounceTime(300),
  switchMap(term => this.api.searchInstruments(term)),
);

input() — сигнальные inputs (Angular 17+)

input() — объявляет input компонента как сигнал только для чтения, заменяя декоратор @Input(). input.required<T>() объявляет обязательный input без значения по умолчанию (компилятор требует, чтобы он был привязан).

import { Component, input, computed } from '@angular/core';

@Component({ selector: 'app-order-row', template: '{{ label() }}' })
export class OrderRowComponent {
  readonly symbol = input<string>('EURUSD');     // optional, with default
  readonly orderId = input.required<number>();   // required input

  // inputs are signals → compose them with computed
  readonly label = computed(() => `#${this.orderId()} ${this.symbol()}`);
}

Поскольку inputs — это сигналы, для реакции на их изменения больше не нужен ngOnChanges — используйте computed или effect.

model() — сигнал двусторонней привязки (Angular 17+)

model() — объявляет изменяемый сигнал с поддержкой двусторонней привязки ([(value)]), заменяя старую пару @Input() + @Output(). Запись в model одновременно обновляет локальное значение и эмитит изменение родителю.

import { Component, model } from '@angular/core';

@Component({
  selector: 'app-qty-input',
  template: `<input type="number"
            [value]="qty()"
            (input)="qty.set(+$any($event.target).value)">`,
})
export class QtyInputComponent {
  readonly qty = model<number>(1); // two-way: [(qty)] in the parent
}

// Parent usage:  <app-qty-input [(qty)]="orderQty" />

linkedSignal() — связанный изменяемый сигнал (Angular 19+)

linkedSignal() — изменяемый сигнал, значение которого производится от исходного сигнала, но может быть локально переопределено. Когда источник меняется, он пересчитывается (сбрасывая любое ручное переопределение). Закрывает разрыв между computed (только для чтения) и обычным изменяемым сигналом.

import { signal, linkedSignal } from '@angular/core';

interface Instrument { symbol: string; defaultQty: number; }

const instrument = signal<Instrument>({ symbol: 'EURUSD', defaultQty: 100 });

// derives from instrument, but the user can still override qty
const qty = linkedSignal(() => instrument().defaultQty);

qty.set(500);                                  // manual override → 500
instrument.set({ symbol: 'BTCUSD', defaultQty: 1 });
console.log(qty());                            // 1 — reset to new source

// With an explicit source + computation:
const selected = linkedSignal({
  source: instrument,
  computation: (instr) => instr.defaultQty,
});

Signals против RxJS — таблица выбора

Сигналы и RxJS дополняют друг друга, а не конкурируют. Сигналы отвечают за синхронное состояние; RxJS — за асинхронные потоки во времени. Выбирайте инструмент под форму задачи.

СценарийSignalsRxJS
Локальное состояние компонента✅ идеально❌ избыточно
HTTP-запрос❌✅ (одноразовый поток, операторы, retry)
Поток WebSocket❌✅ (непрерывный асинхронный поток)
Производное / вычисляемое значение✅ computedвозможно, но многословно (combineLatest+map)
Общее состояние между компонентамисигналы + сервисBehaviorSubject
Сложные асинхронные потоки (debounce, switchMap, retry)❌✅ операторы блистают
Простой счётчик / переключатель✅❌ избыточно

Правило большого пальца: если это «значение, которое меняется» — берите сигнал; если это «события, приходящие во времени» — берите Observable. Используйте toSignal/toObservable, чтобы пересечь границу.

// async source (RxJS) → synchronous state (signal) at the edge
readonly quotes = toSignal(this.ws.quotes$, { initialValue: [] });

// derived UI state stays in signal land
readonly bestBid = computed(() =>
  Math.max(0, ...this.quotes().map(q => q.bid)));

Signals + NgRx (@ngrx/signals)

SignalStore — лёгкое функциональное хранилище из @ngrx/signals, целиком построенное на сигналах. Вы собираете его из features: состояния, вычисляемых селекторов и методов. Без actions, reducers и шаблонного кода.

withState<T>(initial) — объявляет состояние хранилища; каждое свойство становится сигналом в хранилище.

withComputed(store => ...) — объявляет производные сигналы (селекторы), построенные из состояния через computed.

withMethods(store => ...) — объявляет методы, читающие/обновляющие состояние через patchState, включая асинхронные rxMethod.

patchState(store, partial) — иммутабельно обновляет один или несколько срезов состояния.

rxMethod<T>(pipe(...)) — оборачивает RxJS-конвейер в вызываемый метод хранилища, идеально для побочных эффектов HTTP/WebSocket.

import { signalStore, withState, withComputed, withMethods, patchState } from '@ngrx/signals';
import { rxMethod } from '@ngrx/signals/rxjs-interop';
import { computed, inject } from '@angular/core';
import { pipe, switchMap, tap } from 'rxjs';

interface Order { id: number; symbol: string; qty: number; }

interface OrderState {
  orders: Order[];
  loading: boolean;
}

export const OrderStore = signalStore(
  { providedIn: 'root' },
  withState<OrderState>({ orders: [], loading: false }),

  withComputed(({ orders }) => ({
    orderCount: computed(() => orders().length),
  })),

  withMethods((store) => ({
    loadOrders: rxMethod<void>(
      pipe(
        tap(() => patchState(store, { loading: true })),
        switchMap(() => inject(OrderService).getOrders()),
        tap((orders) => patchState(store, { orders, loading: false })),
      ),
    ),
  })),
);

Потребление — это чистые сигналы: store.orders(), store.orderCount(), store.loadOrders() — читаются прямо в шаблонах без пайпа async и без ручных подписок.

Вопросы и ответы — собеседование

1. Что такое сигнал и чем он отличается от BehaviorSubject?

Сигнал — это синхронный реактивный примитив, который всегда хранит текущее значение и уведомляет потребителей при изменении. Как и BehaviorSubject, у него есть «текущее значение», но сигнал читается вызовом функции (price()) вместо .getValue()/subscribe, не требует отписки, автоматически отслеживает зависимости (точечно) и интегрирован с change detection в Angular. BehaviorSubject — это асинхронный поток, на который нужно подписываться и который нужно освобождать. Используйте сигналы для состояния, а BehaviorSubject — когда вам действительно нужны операторы потоков.

2. Когда использовать computed, а когда effect?

Используйте computed, чтобы вывести новое значение из других сигналов — он только для чтения, мемоизирован и ленив. Используйте effect только для побочных эффектов, не порождающих значение: логирование, запись в localStorage, манипуляции с DOM, аналитика. Частая ошибка — использовать эффект для записи в другой сигнал, который должен был быть computed. Если ответ — «мне нужно значение», то это computed.

3. Почему запись сигнала внутри эффекта выбрасывает ошибку и как её разрешить?

По умолчанию Angular запрещает запись внутри эффектов, чтобы предотвратить скрытые реактивные каскады и случайные бесконечные циклы (чтение и запись одного и того же сигнала перезапускает эффект вечно). Когда есть законная необходимость записать другой сигнал, передайте { allowSignalWrites: true } вторым аргументом в effect(). Даже тогда предпочитайте computed или linkedSignal, если на самом деле вы просто выводите состояние.

4. Как завести поток WebSocket/HTTP в мир сигналов?

Используйте toSignal(observable$, { initialValue }) из @angular/core/rxjs-interop. Он подписывается на Observable, предоставляет последнюю эмиссию как сигнал только для чтения и автоматически отписывается при уничтожении контекста внедрения — поэтому не нужен ни пайп async, ни ручное освобождение. В обратную сторону (применить операторы RxJS к состоянию-сигналу) используйте toObservable(signal), например debounceTime + switchMap на поисковом сигнале.

5. Что заменило @Input()/@Output() и .mutate() в современных сигналах Angular?

Сигнальные inputs input() / input.required<T>() заменяют @Input() (и устраняют необходимость в ngOnChanges). model() заменяет пару двусторонней привязки @Input() + @Output(), давая изменяемый сигнал, пригодный для [(value)]. Метод записи .mutate() у сигнала объявлен устаревшим/удалён — вместо него используйте .update(), возвращающий новый массив/объект. В Angular 19 также добавили linkedSignal() — изменяемый сигнал, производный от источника.

Тест

1. Как прочитать текущее значение сигнала с именем price?
Сигналы читаются вызовом как функции: price(). Внутри реактивного контекста это также регистрирует зависимость.
2. Какое утверждение о computed() верно?
Вычисляемый сигнал только для чтения, автоматически отслеживает читаемые им сигналы и мемоизируется — пересчитываясь только при изменении зависимости.
3. Что произойдёт, если читать И писать один и тот же сигнал внутри одного effect?
Запись сигнала, который вы также читаете, перезапускает эффект бесконечно. По умолчанию запись внутри эффектов выбрасывает ошибку; даже с allowSignalWrites никогда не пишите тот же сигнал, который читаете.
4. Какой API преобразует Observable в сигнал только для чтения?
toSignal(observable$, { initialValue }) подписывается и предоставляет последнюю эмиссию как сигнал. toObservable делает обратное.
