Основы Angular

Фундаментальные темы Angular, которые разбирают на собеседованиях уровня senior: хуки жизненного цикла, обнаружение изменений и OnPush, standalone-компоненты, новый синтаксис управления потоком, пайпы, trackBy, запросы view/content, шаблоны и основные декораторы.

Хуки жизненного цикла

Хук жизненного цикла — метод, который Angular вызывает в определённый момент жизни компонента или директивы, от создания до уничтожения. Они выполняются в гарантированном порядке в каждом цикле обнаружения изменений.

Порядок первого выполнения: constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → ngOnDestroy.

constructor

constructor — конструктор класса, вызываемый JavaScript при создании экземпляра компонента, до того как Angular свяжет какие-либо входы или построит представление.

  • Когда срабатывает: первым, при создании экземпляра, до любого хука Angular.
  • Типичный сценарий: только внедрение зависимостей — присвоение внедрённых сервисов полям.
  • Подводный камень: нет DOM, нет значений @Input, нет HTTP. Входы всё ещё undefined, а @ViewChild/DOM недоступны.
@Component({ selector: 'app-orders', standalone: true, template: '' })
export class OrdersComponent {
  // DI only — never call HTTP or touch the DOM here
  constructor(private readonly http: HttpClient) {}
}

ngOnChanges(changes: SimpleChanges)

ngOnChanges — вызывается всякий раз, когда привязанное свойство @Input изменяется по ссылке, с картой предыдущих и текущих значений.

  • Когда срабатывает: перед ngOnInit при первом изменении (только если у компонента есть хотя бы один @Input), затем снова при каждом изменении ссылки @Input.
  • Типичный сценарий: реагирование на изменения входов — пересчёт производных значений при изменении привязанного Order.
  • Подводный камень: срабатывает только при изменении ссылки входа — мутация объекта/массива на месте его не вызовет.
@Input() order!: Order;

ngOnChanges(changes: SimpleChanges): void {
  const change = changes['order'];
  if (change && !change.firstChange) {
    console.log('prev', change.previousValue, 'curr', change.currentValue);
  }
}

ngOnInit

ngOnInit — вызывается один раз после того, как Angular установил начальные входы компонента и выполнил первый ngOnChanges.

  • Когда срабатывает: один раз, после первого ngOnChanges.
  • Типичный сценарий: начальная загрузка данных (HTTP), настройка подписок, чтение начальных значений @Input.
  • Подводный камень: DOM/представление ещё не готовы — используйте ngAfterViewInit для @ViewChild или работы с canvas.
ngOnInit(): void {
  this.sub = this.quoteService.stream(this.symbol)
    .subscribe((quote: Quote) => (this.quote = quote));
}

ngDoCheck

ngDoCheck — пользовательский хук обнаружения изменений, вызываемый в каждом цикле обнаружения изменений, что позволяет отлавливать изменения, которые Angular не может обнаружить самостоятельно.

  • Когда срабатывает: при каждом запуске обнаружения изменений, после ngOnChanges/ngOnInit.
  • Типичный сценарий: обнаружение глубоких мутаций, которые Angular пропускает (например, изменённый массив), через пользовательские KeyValueDiffers/IterableDiffers.
  • Подводный камень: выполняется крайне часто и затратен — избегайте без крайней необходимости и никогда не делайте здесь тяжёлую работу.
ngDoCheck(): void {
  // Runs on EVERY CD cycle — keep it trivial or avoid entirely
  if (this.orders.length !== this.lastCount) {
    this.lastCount = this.orders.length;
  }
}

Подводный камень: ngDoCheck срабатывает тысячи раз — тяжёлая логика здесь обрушит производительность.

ngAfterContentInit

ngAfterContentInit — вызывается один раз после того, как Angular проецирует внешний контент (ng-content) в компонент.

  • Когда срабатывает: один раз, после первой проверки спроецированного контента.
  • Типичный сценарий: доступ к спроецированным потомкам — @ContentChild/@ContentChildren теперь доступны.
  • Подводный камень: @ContentChild равен undefined до этого хука (в конструкторе/ngOnInit).
@ContentChild(HeaderComponent) header?: HeaderComponent;

ngAfterContentInit(): void {
  // Projected content is now available
  console.log(this.header?.title);
}

ngAfterContentChecked

ngAfterContentChecked — вызывается после каждой проверки обнаружения изменений спроецированного контента.

  • Когда срабатывает: после ngAfterContentInit и после каждой последующей проверки контента.
  • Типичный сценарий: реагирование на изменения спроецированного контента в каждом цикле.
  • Подводный камень: никогда не меняйте привязки здесь — рискуете получить ExpressionChangedAfterItHasBeenCheckedError, и это выполняется в каждом цикле.
ngAfterContentChecked(): void {
  // Runs after each content check — keep it side-effect free
}

ngAfterViewInit

ngAfterViewInit — вызывается один раз после того, как Angular инициализирует собственное представление компонента и дочерние представления.

  • Когда срабатывает: один раз, после полной отрисовки представления компонента (и дочерних представлений).
  • Типичный сценарий: элементы @ViewChild доступны, и DOM готов — идеально для инициализации графика на <canvas> или сторонней библиотеки.
  • Подводный камень: обновление привязанного свойства здесь может выбросить ExpressionChangedAfterItHasBeenCheckedError в dev-режиме; оберните в микрозадачу или вызовите detectChanges().
@ViewChild('chart') chartRef!: ElementRef<HTMLCanvasElement>;

ngAfterViewInit(): void {
  const ctx = this.chartRef.nativeElement.getContext('2d');
  this.renderCandles(ctx); // DOM is ready now
}

ngAfterViewChecked

ngAfterViewChecked — вызывается после каждой проверки обнаружения изменений представления компонента и дочерних представлений.

  • Когда срабатывает: после ngAfterViewInit и после каждой последующей проверки представления.
  • Типичный сценарий: реагирование, когда представление стабильно в каждом цикле (например, измерение размера отрисованного элемента).
  • Подводный камень: выполняется в каждом цикле; мутация привязок здесь вызывает ошибку изменения-после-проверки в dev-режиме.
ngAfterViewChecked(): void {
  // Fires after every view check — avoid binding mutations
}

ngOnDestroy

ngOnDestroy — вызывается один раз непосредственно перед тем, как Angular уничтожает компонент, для очистки.

  • Когда срабатывает: один раз, непосредственно перед удалением компонента из DOM.
  • Типичный сценарий: очистка — отписка от Observable, clearInterval, worker.terminate(), закрытие WebSocket.
  • Подводный камень: его пропуск вызывает утечки памяти; долгоживущие подписки удерживают компоненты живыми после навигации.
private sub?: Subscription;
private timer?: ReturnType<typeof setInterval>;
private worker?: Worker;

ngOnDestroy(): void {
  this.sub?.unsubscribe();
  if (this.timer) clearInterval(this.timer);
  this.worker?.terminate();
}

Утечка памяти: поток без отписки удерживает компонент (и его DOM) в памяти навсегда. Всегда выполняйте очистку в ngOnDestroy (или используйте takeUntilDestroyed() / пайп async).

Обнаружение изменений

Обнаружение изменений (Change Detection, CD) — процесс, с помощью которого Angular синхронизирует модель данных компонента с отрисованным DOM. После любого асинхронного события Angular обходит дерево компонентов и обновляет изменившиеся привязки.

Zone.js

Zone.js — библиотека, которая патчит (monkey-patch) все асинхронные браузерные API, чтобы Angular знал, когда асинхронная задача завершается, и мог автоматически запускать обнаружение изменений. Она патчит setTimeout/setInterval, Promise, fetch/XHR и DOM-события (click, input и т. д.).

Как это связано с основами JS: те же API цикла событий (макрозадачи, микрозадачи, DOM-события) — это именно то, что перехватывает Zone.js. Когда пропатченная задача завершается, Zone уведомляет Angular, и запускается CD.

Стратегия Default

Default (ChangeDetectionStrategy.Default) — после любого асинхронного события Angular выполняет обнаружение изменений на всём дереве компонентов, сверху вниз. Это безопасно (никогда не пропускает изменение), но медленно на больших деревьях, потому что каждый компонент проверяется при каждом событии.

Стратегия OnPush

OnPush (ChangeDetectionStrategy.OnPush) — Angular пропускает компонент (и его поддерево) во время CD, пока не произойдёт одно из четырёх конкретных условий. Это резко сокращает количество проверок на больших деревьях.

OnPush-компонент проверяется только когда происходит одно из этих ровно 4 условий-триггеров:

  1. Изменяется ссылка @Input — присваивается НОВЫЙ объект/массив, а не мутируется существующий.
  2. Срабатывает событие от компонента или одного из его потомков (например, обработчик клика в шаблоне).
  3. Пайп async, привязанный в шаблоне, испускает новое значение.
  4. Обнаружение изменений запущено вручную через markForCheck() или detectChanges().
@Component({
  selector: 'app-instruments',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: '...',
})
export class InstrumentsComponent {
  @Input() instruments: readonly Instrument[] = [];
}

Ручные триггеры

markForCheck() — (у ChangeDetectorRef) помечает этот компонент и всех его предков для проверки в следующем цикле обнаружения изменений. CD не запускается немедленно.

detectChanges() — (у ChangeDetectorRef) выполняет обнаружение изменений синхронно на этом компоненте и его потомках сейчас.

ApplicationRef.tick() — один раз выполняет обнаружение изменений по всему приложению, начиная с корня.

constructor(private readonly cdr: ChangeDetectorRef) {}

onTickOutsideAngular(quote: Quote): void {
  this.quote = quote;
  this.cdr.markForCheck();   // schedule a check for next cycle
  // this.cdr.detectChanges(); // or run CD on this subtree right now
}
APIОбластьВремя
markForCheck()Компонент + предкиСледующий цикл CD
detectChanges()Компонент + потомкиНемедленно (синхронно)
ApplicationRef.tick()Всё приложениеНемедленно (синхронно)

Пример из FinTech — 20 000 инструментов

Живая таблица инструментов, отрисовывающая 20 000 строк и получающая по несколько обновлений котировок в секунду. При стратегии Default Angular перепроверяет всё дерево на каждом тике — интерфейс подтормаживает. OnPush делает задачу решаемой, но только если вы соблюдаете равенство по ссылке.

Неправильно — мутация сохраняет ту же ссылку, OnPush её никогда не видит:

this.quotes.push(quote); // same array reference → OnPush misses the update

Правильно — новая ссылка (или сигнал), OnPush её обнаруживает:

this.quotes = [...this.quotes, quote]; // new array → CD runs
// or, even better, use a signal:
this.quotes.update(list => [...list, quote]);

Почему здесь OnPush обязателен: при 20 000 строк стратегия Default проверяет десятки тысяч привязок на каждом тике котировки. OnPush + неизменяемые обновления + trackBy/@for track сокращают это только до строк, которые действительно изменились.

Standalone-компоненты

Standalone-компонент — компонент, директива или пайп с пометкой standalone: true, который объявляет собственные зависимости в массиве imports, устраняя необходимость в NgModule. Standalone — это значение по умолчанию начиная с Angular 17+.

До (NgModule) и после (standalone)

// --- BEFORE: NgModule-based ---
@Component({ selector: 'app-orders', template: '...' })
export class OrdersComponent {}

@NgModule({
  declarations: [OrdersComponent],
  imports: [CommonModule, RouterModule],
  exports: [OrdersComponent],
})
export class OrdersModule {}
// --- AFTER: standalone ---
@Component({
  selector: 'app-orders',
  standalone: true,            // no NgModule needed
  imports: [RouterLink, FormatPricePipe], // declare deps directly
  template: '...',
})
export class OrdersComponent {}

bootstrapApplication()

bootstrapApplication(rootComponent, options) — функция запуска для standalone-приложения. Она загружает единственный корневой standalone-компонент и принимает провайдеры уровня приложения, заменяя старый platformBrowserDynamic().bootstrapModule(AppModule).

// main.ts
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes),
    provideHttpClient(),
  ],
}).catch((err: unknown) => console.error(err));

importProvidersFrom()

importProvidersFrom(...modules) — мост, который извлекает провайдеры из существующего NgModule, чтобы вы могли использовать старые библиотеки на основе модулей внутри списка провайдеров standalone-приложения.

bootstrapApplication(AppComponent, {
  providers: [
    importProvidersFrom(SomeLegacyModule), // reuse an old NgModule's providers
  ],
});

Массив imports в @Component

Standalone-компонент импортирует ровно то, что использует его шаблон — другие standalone-компоненты, директивы и пайпы. Глобального CommonModule нет; встроенному управлению потоком (@if/@for) импорт не нужен, но AsyncPipe, DatePipe и т. д. импортируются по отдельности (или через CommonModule).

@Component({
  selector: 'app-quote',
  standalone: true,
  imports: [AsyncPipe, CurrencyPipe, RouterLink],
  template: '...',
})
export class QuoteComponent {}

Новый синтаксис управления потоком (Angular 17+)

Встроенное управление потоком — блочный синтаксис (@if, @for, @switch, @defer), компилируемый в шаблон во время сборки. Он заменяет структурные директивы *ngIf, *ngFor, *ngSwitch более быстрыми, типобезопасными и не требующими импорта конструкциями.

@if / @else if / @else

@if — условно отрисовывает блок. Поддерживает псевдоним as для захвата вычисленного выражения и цепочки ветвей @else if / @else.

@if (quote$ | async; as quote) {
  <div class="bid">{{ quote.bid }} / {{ quote.ask }}</div>
} @else if (loading) {
  <app-spinner />
} @else {
  <p>No quote available</p>
}

Псевдоним as (quote$ | async; as quote) подписывается один раз и переиспользует значение во всём блоке — без повторного | async.

@for / @empty

@for — повторяет блок для каждого элемента коллекции. Он требует выражение track для идентификации и предоставляет контекстные переменные $index, $first, $last, $even, $odd и $count. Необязательный блок @empty отрисовывается, когда коллекция пуста.

@for (order of orders; track order.id) {
  <tr [class.even]="$even">
    <td>{{ $index + 1 }}</td>
    <td>{{ order.symbol }}</td>
    <td>{{ order.side }}</td>
  </tr>
} @empty {
  <tr><td colspan="3">No orders</td></tr>
}

track обязателен: в отличие от *ngFor, где trackBy был необязателен, @for не скомпилируется без track. Используйте стабильный уникальный ключ (order.id), никогда не $index для данных, которые переупорядочиваются.

@switch / @case / @default

@switch — выбирает один блок на основе совпадающего значения @case, с необязательным @default. Заменяет [ngSwitch] / *ngSwitchCase / *ngSwitchDefault.

@switch (instrument.type) {
  @case ('FOREX')  { <app-forex  [data]="instrument" /> }
  @case ('STOCK')  { <app-stock  [data]="instrument" /> }
  @case ('CRYPTO') { <app-crypto [data]="instrument" /> }
  @default { <p>Unknown instrument</p> }
}

@defer

@defer — лениво загружает компоненты/директивы/пайпы внутри своего блока (и их JS) только при срабатывании триггера. Он поддерживает под-блоки @placeholder (показывается до загрузки), @loading (показывается во время загрузки) и @error (показывается при сбое).

Триггеры @defer — on viewport (блок попадает в область видимости), on interaction (пользователь кликает/нажимает клавиши на плейсхолдере), on hover, on idle (браузер простаивает, по умолчанию), on timer(2s) (после задержки) и when <condition> (булево выражение становится истинным).

@defer (on viewport) {
  <app-heavy-chart [series]="candles" />
} @placeholder {
  <div class="chart-skeleton">Chart loads on scroll</div>
} @loading (minimum 500ms) {
  <app-spinner />
} @error {
  <p>Failed to load chart</p>
}

@loading (minimum 500ms) гарантирует, что спиннер показывается хотя бы 500мс, чтобы он не мелькал и не исчезал при быстрой загрузке.

Почему новый синтаксис лучше структурных директив

CLI для миграции

Angular поставляет автоматическую схематику, которая переписывает существующие шаблоны *ngIf/*ngFor/*ngSwitch на новый синтаксис.

ng generate @angular/core:control-flow

Пайпы Angular

Пайп — преобразование в шаблоне, применяемое оператором |, которое принимает входное значение (и необязательные аргументы) и возвращает форматированное/производное значение, не мутируя источник.

Встроенные пайпы

DatePipe — форматирует дату/число/строку в строку даты по шаблону или именованному формату.

{{ quote.timestamp | date:'dd/MM/yyyy' }}  <!-- 08/06/2026 -->

CurrencyPipe — форматирует число как локализованную строку валюты. Аргументы: код валюты, отображение и строка digitsInfo вида min.minFrac-maxFrac.

{{ order.price | currency:'USD':'symbol':'1.2-2' }}  <!-- $1,234.50 -->

DecimalPipe — форматирует число с группировкой и минимальным/максимальным числом дробных цифр (number:'minInt.minFrac-maxFrac').

{{ quote.bid | number:'1.4-5' }}  <!-- 1.07020 -->

PercentPipe — умножает на 100 и форматирует как процент.

{{ instrument.change | percent:'1.1-2' }}  <!-- 2.35% -->

AsyncPipe — подписывается на Observable или Promise в шаблоне, возвращает их последнее значение и автоматически отписывается при уничтожении. Он также вызывает markForCheck(), поэтому без проблем работает с OnPush.

@if (quote$ | async; as quote) { {{ quote.ask }} }

JsonPipe — сериализует значение через JSON.stringify. Быстрое средство для отладки.

<pre>{{ order | json }}</pre>

SlicePipe — возвращает подмножество массива или строки (как Array.slice), полезно для пагинации/превью.

@for (o of orders | slice:0:10; track o.id) { ... }

UpperCasePipe / LowerCasePipe / TitleCasePipe — преобразуют регистр строки в верхний регистр, нижний регистр или регистр заголовка (Title Case) соответственно.

{{ order.symbol | uppercase }}   <!-- EURUSD -->
{{ order.side | lowercase }}     <!-- buy -->
{{ instrument.type | titlecase }} <!-- Forex -->

KeyValuePipe — превращает объект или Map в итерируемую коллекцию пар { key, value } для использования с @for.

@for (entry of positionsBySymbol | keyvalue; track entry.key) {
  <div>{{ entry.key }}: {{ entry.value }}</div>
}

Пользовательский пайп — FormatPricePipe

Standalone, чистый пайп, который форматирует цену через Intl.NumberFormat, принимая валюту и количество дробных цифр.

@Pipe({
  name: 'formatPrice',
  standalone: true,
  pure: true, // default — recompute only when inputs change by reference
})
export class FormatPricePipe implements PipeTransform {
  transform(value: number, currency = 'USD', digits = 2): string {
    return new Intl.NumberFormat('en-US', {
      style: 'currency',
      currency,
      minimumFractionDigits: digits,
      maximumFractionDigits: digits,
    }).format(value);
  }
}
{{ instrument.price | formatPrice:'EUR':4 }}  <!-- €1,234.5000 -->

Чистые и нечистые пайпы

Чистый пайп (по умолчанию) — Angular повторно запускает transform() только когда изменяется ссылка входа (или примитивное значение). Дёшево и предсказуемо.

Нечистый пайп (pure: false) — transform() выполняется в каждом цикле обнаружения изменений, независимо от того, изменились ли входы. Необходим для изменяемых входов, но затратен.

АспектЧистыйНечистый
Повторный запуск когдаИзменяется ссылка входаВ каждом цикле CD
Производительность✅ Дёшево❌ Дорого
Обнаруживает мутацию на месте❌✅
По умолчанию✅❌

Правило FinTech: держите пайпы чистыми и подавайте им неизменяемые данные. При OnPush + неизменяемых обновлениях чистый пайп пересчитывается ровно тогда, когда нужно — никогда не на каждом тике.

trackBy

trackBy — функция, которая сообщает Angular, как идентифицировать элементы в списке, чтобы он мог переиспользовать существующие DOM-узлы для неизменившихся элементов вместо их уничтожения и пересоздания. В новом синтаксисе это обязательное выражение track у @for.

Без trackBy / trackС trackBy / track
Angular сравнивает по идентичности объектовAngular сравнивает по возвращённому вами ключу
Новый массив ⇒ уничтожает и пересоздаёт все DOM-узлыПерерисовываются только изменившиеся элементы
Теряет состояние фокуса/прокрутки/анимацииСохраняет существующие узлы и состояние
❌ Подтормаживает на больших списках✅ Плавно на больших списках
 
// Component — trackBy function for legacy *ngFor
trackByOrderId(index: number, order: Order): number {
  return order.id;
}
<!-- Legacy *ngFor -->
<tr *ngFor="let order of orders; trackBy: trackByOrderId">...</tr>

<!-- New control flow — track is mandatory -->
@for (order of orders; track order.id) {
  <tr>{{ order.symbol }}</tr>
}

FinTech — без компромиссов: таблица из 10 000 строк, получающая потоковые обновления. Без track каждая новая ссылка на массив перерисовывает все 10 000 строк на каждом тике. С track order.id Angular затрагивает только строки, данные которых действительно изменились.

ViewChild и ContentChild

@ViewChild(selector) — запрашивает один элемент, директиву или дочерний компонент, который находится в собственном шаблоне этого компонента (его представлении).

@ContentChild(selector) — запрашивает один спроецированный элемент/компонент, переданный через <ng-content> (спроецированный контент компонента, а не его собственный шаблон).

@ViewChildren / @ContentChildren — запрашивают все совпадающие элементы, возвращая QueryList<T> (observable-коллекция с итерацией, которая обновляется по мере изменения DOM).

@ContentChildСпроецированный ng-contentngAfterContentInit
@ViewChildrenСобственный шаблон (все)ngAfterViewInit
@ContentChildrenСпроецированный контент (все)ngAfterContentInit

{ static: true } vs { static: false }

{ static: true } — разрешает запрос до первого обнаружения изменений, поэтому результат доступен в ngOnInit. Работает только для запросов, которые не находятся внутри условных/циклических блоков.

{ static: false } — (по умолчанию) разрешает запрос после построения представления, поэтому результат доступен в ngAfterViewInit. Обязательно для элементов внутри @if/@for.

// static: true — usable in ngOnInit
@ViewChild('header', { static: true }) header!: ElementRef<HTMLElement>;

// static: false (default) — usable in ngAfterViewInit
@ViewChild('chart') chart!: ElementRef<HTMLCanvasElement>;

@ViewChildren(RowComponent) rows!: QueryList<RowComponent>;

ngAfterViewInit(): void {
  console.log('row count', this.rows.length);
  this.rows.changes.subscribe((list: QueryList<RowComponent>) => {
    console.log('rows changed', list.length);
  });
}

ng-template, ng-container, ngTemplateOutlet

<ng-template> — определяет фрагмент шаблона, который не отрисовывается, пока что-то явно его не инстанцирует (например, ветвь else у *ngIf или ngTemplateOutlet).

<ng-container> — логический группирующий элемент, который ничего не отрисовывает в DOM. Используйте его, чтобы применить структурную директиву или сгруппировать соседние элементы без добавления лишней обёртки.

ngTemplateOutlet — директива, которая отрисовывает указанный <ng-template> в заданном месте, опционально передавая ему объект контекста.

<!-- ng-template as an *ngIf else branch (not rendered until used) -->
<div *ngIf="quote; else loading">{{ quote.bid }}</div>
<ng-template #loading><app-spinner /></ng-template>

<!-- ng-container groups without an extra DOM element -->
<ng-container *ngIf="order">
  <td>{{ order.symbol }}</td>
  <td>{{ order.price }}</td>
</ng-container>

<!-- ngTemplateOutlet renders a template ref with context -->
<ng-template #row let-o="order">
  <tr><td>{{ o.symbol }}</td></tr>
</ng-template>
<ng-container *ngTemplateOutlet="row; context: { order: currentOrder }" />

Почему ng-container: внутри <table> нельзя обернуть ячейки в <div>, не сломав вёрстку — ng-container применяет директиву, не создавая элемента.

Справочник по декораторам

@Input() — помечает свойство класса как привязываемый вход, позволяя родителю передать значение через [prop]="...".

@Input({ required: true }) — делает вход обязательным; Angular сообщает об ошибке на этапе компиляции, если родитель его не передаёт.

@Input({ transform: fn }) — применяет функцию преобразования к привязанному значению перед присваиванием (например, booleanAttribute, numberAttribute или пользовательское приведение).

@Output() — предоставляет EventEmitter, который потомок использует для испускания пользовательских событий родителю через (event)="...".

@HostListener(event, args) — привязывает метод к DOM-событию на хост-элементе, опционально передавая данные события, такие как $event.

@HostBinding(property) — привязывает свойство/атрибут/класс/стиль хост-элемента к полю компонента, так что значение поля управляет хостом.

@Component({ selector: 'app-order-row', standalone: true, template: '...' })
export class OrderRowComponent {
  // Required input
  @Input({ required: true }) order!: Order;

  // Transformed input — coerce string "true"/presence to boolean
  @Input({ transform: booleanAttribute }) highlighted = false;

  // Numeric coercion
  @Input({ transform: numberAttribute }) rowIndex = 0;

  // Output — emit a typed event to the parent
  @Output() selected = new EventEmitter<Order>();

  // Host class bound to a field
  @HostBinding('class.active') get isActive(): boolean {
    return this.highlighted;
  }

  // Host event with $event payload
  @HostListener('click', ['$event'])
  onClick(event: MouseEvent): void {
    event.stopPropagation();
    this.selected.emit(this.order);
  }
}
<!-- Parent template -->
<app-order-row
  [order]="order"
  highlighted
  (selected)="onSelect($event)">
</app-order-row>

Вопросы и ответы — вопросы для собеседования

1. В каком порядке выполняются хуки жизненного цикла и где находится ngOnChanges?

constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → ngOnDestroy. ngOnChanges выполняется перед первым ngOnInit, но только если у компонента есть хотя бы один @Input, и снова при каждом последующем изменении ссылки входа.

2. Каковы точные условия, запускающие обнаружение изменений на OnPush-компоненте?

Четыре: (1) изменяется ссылка @Input (новый объект/массив, а не мутация), (2) срабатывает событие от компонента или потомка, (3) пайп async в его шаблоне испускает новое значение, или (4) он помечен вручную через markForCheck() / detectChanges(). Иначе Angular пропускает компонент и его поддерево.

3. Почему this.quotes.push(q) не обновляет OnPush-представление и как это исправить?

push мутирует массив на месте, поэтому его ссылка не меняется. OnPush сравнивает ссылки @Input, видит тот же массив и пропускает компонент. Исправьте, присвоив новую ссылку — this.quotes = [...this.quotes, q] — или используя сигнал (this.quotes.update(list => [...list, q])), который уведомляет CD при каждой установке.

4. В чём разница между markForCheck() и detectChanges()?

markForCheck() помечает компонент и всех его предков как «грязные», чтобы они были проверены в следующем цикле CD — он не запускает CD сейчас. detectChanges() выполняет обнаружение изменений синхронно, немедленно на компоненте и его потомках. Используйте markForCheck() с OnPush после обновления состояния вне поля зрения Angular; используйте detectChanges(), когда нужна немедленная синхронная отрисовка.

5. Почему track обязателен в @for и что ломается без хорошего ключа?

track даёт каждому элементу стабильную идентичность, чтобы Angular мог переиспользовать DOM-узлы для неизменившихся элементов вместо уничтожения и пересоздания всего списка. Без него (или с плохим ключом вроде $index для переупорядочиваемых данных) новая ссылка на массив заставляет Angular перерисовывать каждый узел, теряя состояние фокуса/прокрутки/анимации и обрушивая производительность на больших списках — недопустимо для таблицы из 10 000 строк.

6. Когда доступен @ViewChild и как { static: true } это меняет?

По умолчанию (static: false) @ViewChild разрешается после построения представления, поэтому впервые доступен в ngAfterViewInit. С { static: true } Angular разрешает его до первого обнаружения изменений, делая его доступным в ngOnInit — но только для запросов, которые не находятся внутри условного (@if) или циклического (@for) блока, который может ещё не существовать.

Квиз

1. Какой хук жизненного цикла — правильное место для инициализации <canvas> через @ViewChild?
Представление (и его элементы @ViewChild) готовы только в ngAfterViewInit. В конструкторе и ngOnInit DOM ещё не существует.
2. Что из этого НЕ запускает обнаружение изменений на OnPush-компоненте?
push() сохраняет ту же ссылку на массив. OnPush сравнивает ссылки, не видит изменений и пропускает компонент. Вместо этого присвойте новый массив.
3. Что делает markForCheck()?
markForCheck() помечает компонент + предков как «грязные» для СЛЕДУЮЩЕГО цикла. detectChanges() запускает сейчас; ApplicationRef.tick() охватывает всё приложение.
4. Какой триггер @defer загружает блок, когда он попадает в область видимости?
on viewport откладывает загрузку, пока блок не войдёт в область видимости. on idle (по умолчанию) ждёт простоя браузера, on hover — наведения указателя, on timer — задержки.
5. В чём разница между чистым и нечистым пайпом?
Чистый (по умолчанию) пересчитывается только при изменении ссылки/примитива на входе; нечистый (pure: false) пересчитывается в каждом цикле CD — затратно, поэтому предпочитайте чистый + неизменяемые данные.
