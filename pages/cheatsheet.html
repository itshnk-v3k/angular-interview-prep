3. switchMap vs mergeMap vs concatMap vs exhaustMap

ОператорПоведениеТипичное применение
switchMapОтменяет предыдущий внутренний при поступлении нового значения.Автодополнение / поиск при вводе, побеждает последний.
mergeMapЗапускает все внутренние Observable параллельно, без порядка.Независимые параллельные запросы.
concatMapСтавит внутренние в очередь, выполняет строго по одному по порядку.Последовательные записи / упорядоченные сохранения.
exhaustMapИгнорирует новые значения, пока выполняется внутренний.Кнопка входа / защита от двойной отправки.

4. Типы Subject

  • Subject — мультикаст, без начального значения; подписчики получают значения, эмитированные только после подписки.
  • BehaviorSubject — требует начальное значение, повторяет последнее значение каждому новому подписчику.
  • ReplaySubject — повторяет последние N (или все, в пределах временного окна) значений новым подписчикам.
  • AsyncSubject — эмитит только финальное значение и только после завершения источника.

5. Хуки жизненного цикла (по порядку)

  1. ngOnChanges — изменился input.
  2. ngOnInit — инициализация.
  3. ngDoCheck — кастомное обнаружение.
  4. ngAfterContentInit — проекция готова.
  5. ngAfterContentChecked — проекция проверена.
  6. ngAfterViewInit — представление готово.
  7. ngAfterViewChecked — представление проверено.
  8. ngOnDestroy — очистка.

Современные альтернативы: afterNextRender, afterRender (работа с DOM только в браузере).

6. Триггеры обнаружения изменений OnPush

  • Меняется ссылка @Input() (новый объект/массив, не мутация).
  • Срабатывает DOM-событие от компонента или его шаблона.
  • Пайп async (или сигнал), привязанный в шаблоне, эмитит новое значение.
  • Обнаружение изменений запускается вручную через ChangeDetectorRef.markForCheck().

7. Signals API

  • signal(v) — записываемое реактивное значение; читается через s(), обновляется через set / update.
  • computed(fn) — производный, мемоизированный, доступный только для чтения сигнал, пересчитывается лениво.
  • effect(fn) — выполняет побочный эффект при изменении любого читаемого сигнала.
  • toSignal(obs$) — преобразует Observable в сигнал (авто-отписка).
  • toObservable(sig) — преобразует сигнал в Observable.
  • input() — input компонента на основе сигналов (input.required<T>() для обязательного).
  • model() — двусторонне связываемый сигнал-input ([(value)]).
  • linkedSignal() — записываемый сигнал, сбрасывающийся из вычисляемого источника.

9. SOLID

  • S — Единственная ответственность: у класса/модуля должна быть одна причина для изменения.
  • O — Открытость/закрытость: открыт для расширения, закрыт для модификации.
  • L — Подстановка Лисков: подтипы должны быть применимы там, где ожидается базовый тип.
  • I — Разделение интерфейсов: лучше много мелких интерфейсов, чем один большой.
  • D — Инверсия зависимостей: зависьте от абстракций, а не от реализаций.

10. Частые вопросы на собеседовании

В1. == vs ===? О: === сравнивает значение+тип без приведения; == сначала приводит — всегда используйте ===.

В2. null vs undefined? О: undefined = не присвоено; null = намеренно пустое значение.

В3. Что такое замыкание? О: Функция, сохраняющая доступ к своей внешней лексической области даже после её завершения.

В4. let vs var? О: let имеет блочную область и TDZ; var имеет функциональную область и поднимается как undefined.

В5. Микрозадача vs макрозадача? О: Promise = микрозадачи (обрабатываются первыми); setTimeout = макрозадачи (по одной за тик цикла).

В6. Стрелочная vs обычная функция? О: У стрелочных нет собственного this/arguments; они связывают this лексически.

В7. interface vs type? О: Интерфейсы сливаются и расширяются (объекты); типы делают объединения/пересечения/примитивы.

В8. any vs unknown? О: any отключает проверки; unknown безопасен — нужно сузить перед использованием.

В9. readonly vs const? О: const фиксирует привязку; readonly фиксирует свойство на этапе компиляции.

В10. Компонент vs Директива? О: У компонента есть шаблон; директива добавляет поведение существующему элементу.

В11. Standalone vs NgModule? О: Standalone-компоненты объявляют свои imports, модуль не нужен (по умолчанию в v17+).

В12. Конструктор vs ngOnInit? О: Конструктор = настройка DI; ngOnInit = инициализация после установки input'ов.

В13. ViewChild vs ContentChild? О: ViewChild запрашивает собственный шаблон; ContentChild запрашивает проецируемый контент.

В14. Шаблонные vs реактивные формы? О: Шаблонные = на директивах в HTML; реактивные = явная модель в TS (тестируемые, масштабируемые).

В15. Subject vs BehaviorSubject? О: BehaviorSubject требует начальное значение и повторяет последнее новым подписчикам.

В16. Холодный vs горячий Observable? О: Холодный перезапускается для каждого подписчика (HTTP); горячий общий/мультикаст (Subject, события).

В17. Как избежать утечек памяти? О: Отписывайтесь через takeUntilDestroyed, пайп async или take(1).

В18. switchMap vs mergeMap? О: switchMap отменяет предыдущий внутренний; mergeMap запускает все параллельно.

В19. Что такое обнаружение изменений? О: Проверка Angular привязок для синхронизации модели с DOM (Zone.js или сигналы).

В20. Ленивая загрузка? О: Загрузка бандлов маршрутов по требованию через loadComponent / loadChildren для уменьшения начального бандла.

12. Типы guard'ов маршрутизатора

  • canActivate — разрешает/запрещает переход в маршрут.
  • canActivateChild — разрешает/запрещает переход в дочерние маршруты.
  • canDeactivate — разрешает/запрещает выход из маршрута (напр. предупреждение о несохранённых изменениях).
  • canMatch — решает, подходит ли определение маршрута вообще (заменяет canLoad; контролирует ленивые бандлы).
  • resolve — предзагрузка данных до активации маршрута.

13. Управление потоком в Angular 17+

  • @if (cond) { } @else if () { } @else { } — условный рендеринг (заменяет *ngIf).
  • @for (item of items; track item.id) { } — цикл; track обязателен (заменяет *ngFor).
  • @empty { } — запасной блок, показываемый когда коллекция @for пуста.
  • @switch (val) { @case (x) {} @default {} } — множественное ветвление (заменяет *ngSwitch).
  • @defer (on viewport) { } @placeholder { } @loading { } @error { } — ленивая загрузка блока шаблона.

15. Принципы ООП

  • Инкапсуляция — объединяет состояние и поведение, скрывает внутренности за публичным API (#private).
  • Абстракция — показывает, что объект делает, скрывает, как он это делает.
  • Наследование — порождает класс от другого для переиспользования и расширения поведения (extends).
  • Полиморфизм — один интерфейс, много реализаций; поведение определяется в момент вызова.
