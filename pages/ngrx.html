NgRx — управление состоянием

Реактивное управление состоянием в стиле Redux для Angular: однонаправленный поток данных, иммутабельное состояние, чистые редьюсеры, мемоизированные селекторы и обработка побочных эффектов — на сквозном примере FinTech-фичи Orders (заявки).

Что такое NgRx и когда его применять

NgRx — библиотека реактивного управления состоянием для Angular, основанная на паттерне Redux. Она обеспечивает три принципа: однонаправленный поток данных (данные движутся только в одном направлении), иммутабельное состояние (состояние никогда не мутируется, а только заменяется) и чистые функции (редьюсеры и селекторы не имеют побочных эффектов). Состояние хранится в едином наблюдаемом хранилище, построенном на RxJS.

Единый источник истины — всё состояние приложения хранится в одном иммутабельном дереве объектов внутри хранилища.

Однонаправленный поток данных — компоненты диспетчеризуют экшены, редьюсеры формируют новое состояние, селекторы его читают, и цикл повторяется. Данные никогда не текут в обратную сторону.

Redux DevTools — каждый экшен и каждый переход состояния записываются, что обеспечивает отладку с «путешествием во времени» и полную трассируемость.

Когда его применять

СценарийNgRx?Почему
Сложное состояние, разделяемое между многими компонентами✅Единое хранилище устраняет прокидывание пропсов и дублирование состояния
Большие команды✅Предсказуемые паттерны и трассируемость DevTools масштабируются на множество разработчиков
Сложные асинхронные потоки / оптимистичные обновления✅Эффекты изолируют побочные эффекты; экшены моделируют каждый шаг
Небольшие приложения с локальным состоянием❌Сервис на BehaviorSubject или Signals проще — NgRx избыточен

Эмпирическое правило: используйте NgRx, когда состояние разделяемое, долгоживущее и изменяется из многих мест. Для временного UI-состояния одного компонента подходящий инструмент — Signal или небольшой сервис.

Основные концепции

NgRx строится из пяти строительных блоков: Actions (экшены), Reducers (редьюсеры), Store (хранилище), Selectors (селекторы) и Effects (эффекты). Мы моделируем фичу Orders для торгового деска.

Action (экшен)

Action — простой объект со строкой type, описывающей что произошло, плюс необязательные свойства-полезная нагрузка. Экшены — единственный способ выразить намерение или сообщить о событии; их диспетчеризуют, а не вызывают напрямую. Соглашение '[Источник] Событие' делает их читаемыми в DevTools.

// orders.actions.ts
import { createAction, props } from '@ngrx/store';
import { Order } from './orders.state';

export const loadOrders = createAction('[Orders] Load');

export const loadOrdersSuccess = createAction(
  '[Orders] Load Success',
  props<{ orders: Order[] }>()
);

export const loadOrdersFailure = createAction(
  '[Orders] Load Failure',
  props<{ error: string }>()
);

props<T>() привязывает строго типизированную полезную нагрузку. loadOrdersSuccess({ orders }) полностью проверяется типами — передача неверной формы вызывает ошибку компиляции.

Reducer (редьюсер)

Reducer — чистая функция (state, action) => newState. У неё нет побочных эффектов (никакого HTTP, логирования, Date.now()) и она обязана возвращать совершенно новый объект — никогда не мутировать существующее состояние. При одинаковых входных данных она всегда даёт одинаковый результат.

// orders.reducer.ts
import { createReducer, on } from '@ngrx/store';
import { OrdersState, initialOrdersState } from './orders.state';
import { loadOrders, loadOrdersSuccess, loadOrdersFailure } from './orders.actions';

export const ordersReducer = createReducer(
  initialOrdersState,
  on(loadOrders, (state): OrdersState => ({
    ...state,
    loading: true,
    error: null,
  })),
  on(loadOrdersSuccess, (state, { orders }): OrdersState => ({
    ...state,
    orders,
    loading: false,
  })),
  on(loadOrdersFailure, (state, { error }): OrdersState => ({
    ...state,
    loading: false,
    error,
  }))
);

Подводный камень: state.orders.push(order) мутирует существующий массив и ломает обнаружение изменений, «путешествие во времени» в DevTools и мемоизированные селекторы. Всегда используйте spread: { ...state, orders: [...state.orders, order] }.

Store (хранилище)

Store — единое иммутабельное дерево состояния, представленное как RxJS Observable. Вы читаете из него через store.select(selector) и записываете, диспетчеризуя экшены через store.dispatch(action). Напрямую присваивать значение хранилищу нельзя.

import { Component, inject } from '@angular/core';
import { Store } from '@ngrx/store';
import { Observable } from 'rxjs';
import { Order } from './state/orders.state';
import { loadOrders } from './state/orders.actions';
import { selectAllOrders } from './state/orders.selectors';

@Component({ selector: 'app-orders', standalone: true, template: '' })
export class OrdersComponent {
  private readonly store = inject(Store);

  // READ — a live, memoized stream of orders
  readonly orders$: Observable<Order[]> = this.store.select(selectAllOrders);

  // WRITE — express intent; the effect + reducer do the work
  load(): void {
    this.store.dispatch(loadOrders());
  }
}

Selector (селектор)

Selector — чистая, мемоизированная функция, которая извлекает и выводит срез состояния. Она пересчитывается только при изменении своих входных срезов, поэтому вывод «дорогих» данных дёшев. createFeatureSelector берёт срез фичи верхнего уровня; createSelector комбинирует селекторы и добавляет проектор.

// orders.selectors.ts
import { createFeatureSelector, createSelector } from '@ngrx/store';
import { OrdersState } from './orders.state';
import { Order } from './orders.state';

export const selectOrdersState =
  createFeatureSelector<OrdersState>('orders');

export const selectAllOrders = createSelector(
  selectOrdersState,
  (state: OrdersState): Order[] => state.orders
);

export const selectPendingOrders = createSelector(
  selectAllOrders,
  (orders: Order[]): Order[] =>
    orders.filter((o) => o.status === 'pending')
);

export const selectOrdersLoading = createSelector(
  selectOrdersState,
  (state: OrdersState): boolean => state.loading
);

Мемоизация: selectPendingOrders повторно выполняет filter только тогда, когда selectAllOrders возвращает новую ссылку. Если состояние изменилось в другом месте, переиспользуется кэшированный результат — большое преимущество для OnPush-компонентов.

Effect (эффект)

Effect — изолированный обработчик побочных эффектов (HTTP, WebSocket, навигация, хранилище). Он слушает поток экшенов, выполняет асинхронную работу и диспетчеризует новый экшен с результатом. Контракт простой: экшены на входе → экшены на выходе. Редьюсеры остаются чистыми, потому что вся «нечистота» живёт здесь.

// orders.effects.ts
import { inject } from '@angular/core';
import { Actions, createEffect, ofType } from '@ngrx/effects';
import { of } from 'rxjs';
import { switchMap, map, catchError } from 'rxjs/operators';
import { OrderService } from './order.service';
import { loadOrders, loadOrdersSuccess, loadOrdersFailure } from './orders.actions';

export class OrdersEffects {
  private readonly actions$ = inject(Actions);
  private readonly orderService = inject(OrderService);

  loadOrders$ = createEffect(() =>
    this.actions$.pipe(
      ofType(loadOrders),
      switchMap(() =>
        this.orderService.getAll().pipe(
          map((orders) => loadOrdersSuccess({ orders })),
          catchError((error: Error) =>
            of(loadOrdersFailure({ error: error.message }))
          )
        )
      )
    )
  );
}

Выбор оператора уплощения имеет значение: switchMap отменяет предыдущий запрос (отлично для перезагрузок/автодополнения), concatMap ставит их в очередь по порядку (отлично для записи), mergeMap выполняет все параллельно, exhaustMap игнорирует новые, пока занят (отлично для логина/отправки).

Диаграмма потока данных

NgRx — это замкнутый однонаправленный цикл. Экшен входит в цикл, проходит через эффекты и редьюсеры, и результирующее состояние проецируется обратно во вью селекторами.

 ┌────────────────────────────────────────────────────────────────┐
 │                                                                  │
 │   ┌─────────────┐   dispatch(action)    ┌──────────────────┐     │
 │   │  Component  │ ────────────────────► │      Store       │     │
 │   └─────────────┘                       └──────────────────┘     │
 │          ▲                                  │         │          │
 │          │ selector                action   │         │ action   │
 │          │ (derived data)                   ▼         ▼          │
 │   ┌─────────────┐                  ┌──────────┐  ┌──────────┐    │
 │   │  Selector   │ ◄── new State ── │ Reducer  │  │  Effect  │    │
 │   │ (memoized)  │                  │ (pure)   │  │ (async)  │    │
 │   └─────────────┘                  └──────────┘  └──────────┘    │
 │                                         ▲              │         │
 │                                         │  Success /   │ HTTP /  │
 │                                         └── Failure ───┘ WS      │
 │                                            action                │
 └────────────────────────────────────────────────────────────────┘

Читайте цикл как нумерованные шаги:

  1. Компонент диспетчеризует Action — например store.dispatch(loadOrders()).
  2. Effect выполняет асинхронную работу — перехватывает loadOrders, вызывает HTTP/WebSocket-сервис.
  3. Effect диспетчеризует Action успеха или неудачи — loadOrdersSuccess({ orders }) или loadOrdersFailure({ error }).
  4. Reducer обновляет состояние — чистая функция возвращает новый иммутабельный объект состояния.
  5. Selector выводит данные — мемоизированная проекция (например, ожидающие заявки) пересчитывается только при изменении входных данных.
  6. Компонент перерисовывается — выбранный поток эмитит, срабатывает обнаружение изменений OnPush, вью обновляется.

Почему это масштабируется: поскольку каждое изменение состояния — явный сериализуемый экшен, вся история воспроизводима в DevTools, и рассуждать о том, «как состояние пришло к этому?», становится тривиально.

Структура папок состояния фичи

Срез фичи держит все артефакты NgRx вместе и рядом с обслуживаемой фичей, благодаря чему срез лениво загружаем и самодостаточен.

Регистрация

NgRx регистрируется через standalone-провайдеры — либо глобально (root), либо лениво (на маршрут фичи).

Корневая регистрация — регистрирует всё хранилище и эффекты один раз при старте приложения.

// main.ts / app.config.ts (root)
import { provideStore } from '@ngrx/store';
import { provideEffects } from '@ngrx/effects';
import { ordersReducer } from './features/orders/state/orders.reducer';
import { OrdersEffects } from './features/orders/state/orders.effects';

export const appConfig = {
  providers: [
    provideStore({ orders: ordersReducer }),
    provideEffects([OrdersEffects]),
  ],
};

Ленивая (фичевая) регистрация — регистрирует один срез фичи только при загрузке его маршрута, через provideState.

// orders.routes.ts (lazy feature)
import { Routes } from '@angular/router';
import { provideState } from '@ngrx/store';
import { provideEffects } from '@ngrx/effects';
import { ordersReducer } from './state/orders.reducer';
import { OrdersEffects } from './state/orders.effects';

export const ordersRoutes: Routes = [
  {
    path: '',
    providers: [
      provideState('orders', ordersReducer),
      provideEffects([OrdersEffects]),
    ],
    loadComponent: () =>
      import('./orders.component').then((m) => m.OrdersComponent),
  },
];

provideStore vs provideState: используйте provideStore один раз в корне для глобальной формы состояния, а provideState внутри ленивых маршрутов, чтобы подключать срезы фич по требованию — сохраняя начальный бандл небольшим.

EntityAdapter (@ngrx/entity)

EntityAdapter — утилита из @ngrx/entity для управления нормализованными коллекциями объектов, индексированными по ID. Вместо массива состояние хранит { ids: string[], entities: Record<string, Order> }, что даёт поиск за O(1) и CRUD-помощники без шаблонного кода.

getInitialState() — возвращает пустую форму { ids, entities } с местом для дополнительных полей.

addOne(entity, state) — добавляет одну сущность (ничего не делает, если ID уже существует).

addMany(entities, state) — добавляет несколько сущностей за раз.

setAll(entities, state) — заменяет всю коллекцию (идеально для свежей загрузки).

updateOne({ id, changes }, state) — поверхностно сливает частичные изменения в одну сущность.

removeOne(id, state) — удаляет одну сущность по ID.

getSelectors() — возвращает готовые мемоизированные селекторы: selectAll, selectEntities, selectIds, selectTotal.

// orders.state.ts (entity version)
import { EntityState, EntityAdapter, createEntityAdapter } from '@ngrx/entity';
import { Order } from './orders.state';

export interface OrdersState extends EntityState<Order> {
  loading: boolean;
  error: string | null;
}

export const ordersAdapter: EntityAdapter<Order> =
  createEntityAdapter<Order>({
    selectId: (order: Order) => order.id,
    sortComparer: (a: Order, b: Order) => a.symbol.localeCompare(b.symbol),
  });

export const initialOrdersState: OrdersState =
  ordersAdapter.getInitialState({ loading: false, error: null });
// orders.reducer.ts (entity version)
import { createReducer, on } from '@ngrx/store';
import { ordersAdapter, initialOrdersState, OrdersState } from './orders.state';
import { loadOrdersSuccess, updateOrderStatus, removeOrder } from './orders.actions';

export const ordersReducer = createReducer(
  initialOrdersState,
  on(loadOrdersSuccess, (state, { orders }): OrdersState =>
    ordersAdapter.setAll(orders, { ...state, loading: false })
  ),
  on(updateOrderStatus, (state, { id, status }): OrdersState =>
    ordersAdapter.updateOne({ id, changes: { status } }, state)
  ),
  on(removeOrder, (state, { id }): OrdersState =>
    ordersAdapter.removeOne(id, state)
  )
);
// orders.selectors.ts — reuse the adapter's selectors
import { createFeatureSelector } from '@ngrx/store';
import { ordersAdapter, OrdersState } from './orders.state';

export const selectOrdersState =
  createFeatureSelector<OrdersState>('orders');

const { selectAll, selectEntities, selectTotal } =
  ordersAdapter.getSelectors();

export const selectAllOrders =
  createSelectorWrapper(selectOrdersState, selectAll);
// In practice:
// export const selectAllOrders = createSelector(selectOrdersState, selectAll);

Зачем нормализовать: обновление одной заявки выполняется за O(1) (entities[id]) вместо обхода массива, и при этом никогда нет риска дублирующихся записей.

@ngrx/signals (SignalStore)

SignalStore — современная, лёгкая альтернатива NgRx, построенная на Angular Signals вместо RxJS и Redux-церемоний. Состояние — это сигнал; вы собираете хранилище из фич: withState (начальное состояние), withComputed (производные сигналы, как селекторы) и withMethods (обновления и асинхронная логика, как редьюсеры и эффекты вместе).

withState(initial) — объявляет реактивное состояние в виде сигналов.

withComputed(store => ...) — объявляет мемоизированные производные сигналы из состояния.

withMethods(store => ...) — объявляет методы, которые обновляют состояние через patchState и выполняют побочные эффекты.

// orders.store.ts
import { computed, inject } from '@angular/core';
import { signalStore, withState, withComputed, withMethods, patchState } from '@ngrx/signals';
import { OrderService } from './order.service';
import { Order } from './orders.state';

interface OrdersState {
  orders: Order[];
  loading: boolean;
}

export const OrdersStore = signalStore(
  { providedIn: 'root' },
  withState<OrdersState>({ orders: [], loading: false }),
  withComputed(({ orders }) => ({
    pendingOrders: computed(() =>
      orders().filter((o) => o.status === 'pending')
    ),
    orderCount: computed(() => orders().length),
  })),
  withMethods((store, orderService = inject(OrderService)) => ({
    async load(): Promise<void> {
      patchState(store, { loading: true });
      const orders = await orderService.getAllAsync();
      patchState(store, { orders, loading: false });
    },
  }))
);

Когда выбирать SignalStore: он убирает шаблонный код экшенов, редьюсеров и эффектов и нативно интегрируется с Signals — идеален для новых приложений и состояния, локального для фичи. Классический @ngrx/store по-прежнему хорош для очень больших приложений, которым нужна строгая трассируемость экшенов и полный лог экшенов в DevTools.

Вопросы и ответы — вопросы с собеседований

1. Почему редьюсер обязан быть чистой функцией?

Чистота гарантирует, что одна и та же пара (state, action) всегда даёт одно и то же новое состояние без побочных эффектов. Это делает переходы состояния детерминированными и воспроизводимыми, что как раз и обеспечивает «путешествие во времени» в Redux DevTools. Это также означает, что редьюсеры никогда не запускают HTTP и не мутируют существующий объект — поэтому мемоизированные селекторы и обнаружение изменений OnPush (которые сравнивают ссылки) работают корректно. Побочные эффекты — в эффектах, а не в редьюсерах.

2. В чём разница между селектором и прямым чтением состояния?

Селектор — мемоизированная, композируемая, чистая функция. Он кэширует свой последний результат и пересчитывает только при изменении входных срезов, поэтому вывод «дорогих» данных (например, фильтрация ожидающих заявок) дёшев и эмитит стабильную ссылку, когда ничего значимого не изменилось. Прямое чтение состояния пропускает мемоизацию, пересчитывает при каждой эмиссии и связывает компоненты с формой состояния. Селекторы также централизуют логику вывода, делая её тестируемой и переиспользуемой.

3. Какова роль эффекта и почему побочные эффекты держат вне редьюсеров?

Эффект слушает поток экшенов и выполняет побочные эффекты — HTTP-вызовы, подписки на WebSocket, навигацию, хранилище — затем диспетчеризует результирующий экшен (экшены на входе → экшены на выходе). Удержание «нечистоты» здесь позволяет редьюсерам оставаться чистыми и детерминированными. Например, loadOrders$ перехватывает loadOrders, вызывает сервис и мапит ответ в loadOrdersSuccess или loadOrdersFailure. Выбор switchMap/concatMap/exhaustMap определяет, как обрабатываются конкурентные запросы.

4. Когда НЕ стоит использовать NgRx?

Для небольших приложений или состояния, локального для одного компонента и короткоживущего, NgRx добавляет шаблонный код и косвенность без выгоды. Сервис на базе BehaviorSubject или Angular Signals проще. NgRx оправдывает свою сложность, когда состояние разделяется между многими компонентами, изменяется из многих мест, включает сложные асинхронные/оптимистичные потоки или когда большая команда выигрывает от предсказуемых паттернов и трассируемости DevTools. Для современных приложений, которым нужно меньше церемоний, @ngrx/signals — золотая середина.

5. Какую проблему решает @ngrx/entity?

Он управляет нормализованными коллекциями, индексированными по ID — состояние становится { ids: string[], entities: Record<string, Order> } вместо сырого массива. Это даёт поиск и обновления за O(1), предотвращает дублирующиеся записи и убирает CRUD-шаблон через методы адаптера (addOne, setAll, updateOne, removeOne). getSelectors() предоставляет готовые мемоизированные селекторы (selectAll, selectEntities, selectIds, selectTotal). Обновление одной заявки по ID гораздо дешевле, чем обход массива.

Тест

1. Какое утверждение о редьюсере верно?
Редьюсеры — чистые функции (state, action) => newState без побочных эффектов. HTTP и диспетчеризация принадлежат эффектам; состояние нужно заменять, а не мутировать.
2. В чём основное преимущество селектора?
Селекторы — мемоизированные чистые функции. Они кэшируют последний результат и пересчитывают только при изменении входного среза — дешёвый вывод и стабильные ссылки для OnPush.
3. В loadOrders$ какой оператор отменяет предыдущий выполняющийся запрос при поступлении нового loadOrders?
switchMap отписывается от предыдущего внутреннего observable при поступлении нового экшена — идеально для перезагрузок. concatMap ставит в очередь, mergeMap распараллеливает, exhaustMap игнорирует новые, пока занят.
4. Какой провайдер подключает срез фичи лениво внутри маршрута?
provideStore регистрирует корневое хранилище один раз; provideState подключает срез фичи в providers ленивого маршрута, в паре с provideEffects([OrdersEffects]).
