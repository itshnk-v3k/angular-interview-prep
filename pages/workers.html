ServiceWorker & WebWorkerServiceWorker и WebWorker

Two completely different background scripts with similar names: ServiceWorker proxies the network for caching and offline support, while WebWorker moves CPU-heavy computations (RSI, MACD) off the UI thread. Neither one has access to the DOM.Два совершенно разных фоновых скрипта с похожим названием: ServiceWorker проксирует сеть для кэширования и работы офлайн, а WebWorker выносит ресурсоёмкие вычисления (RSI, MACD) из потока UI. Ни тот, ни другой не имеют доступа к DOM.

ServiceWorker vs WebWorker — comparison

Both run on a thread separate from the main UI, and both are written in TypeScript, but they solve opposite problems. Understanding which one to choose is a common signal of senior-level expertise.

ServiceWorker — a background script that sits between the application and the network, intercepting fetch requests in order to serve cached responses, provide offline mode and deliver push notifications.

WebWorker — a background script that runs JavaScript on its own thread purely for computation and communicates with the page via message passing. It has no network-proxy role.

Mnemonic: ServiceWorker serves the network; WebWorker works the CPU. One survives a page reload to provide offline mode; the other dies together with its tab.

ServiceWorker in Angular

Angular ServiceWorker (ngsw) — Angular's ready-made service worker (ngsw-worker.js) that turns the app into an installable PWA. It is controlled declaratively through the ngsw-config.json file rather than hand-written fetch handlers, caching assets and API data and taking over the management of version updates.

Setup overview

Add the PWA package via the CLI, which generates ngsw-config.json, a manifest and the registration call. In a standalone app, registration is done through provideServiceWorker; in an NgModule app you use ServiceWorkerModule.register('ngsw-worker.js'). Registration is disabled in development mode and enabled in production builds.

// Scaffold everything (adds @angular/service-worker + ngsw-config.json)
//   ng add @angular/pwa

// --- Standalone bootstrap: main.ts ---
import { bootstrapApplication } from '@angular/platform-browser';
import { provideServiceWorker } from '@angular/service-worker';
import { isDevMode } from '@angular/core';
import { AppComponent } from './app/app.component';

bootstrapApplication(AppComponent, {
  providers: [
    provideServiceWorker('ngsw-worker.js', {
      enabled: !isDevMode(),            // only in production builds
      registrationStrategy: 'registerWhenStable:30000',
    }),
  ],
});
// --- Legacy NgModule equivalent: app.module.ts ---
import { ServiceWorkerModule } from '@angular/service-worker';
import { isDevMode, NgModule } from '@angular/core';

@NgModule({
  imports: [
    ServiceWorkerModule.register('ngsw-worker.js', {
      enabled: !isDevMode(),
    }),
  ],
})
export class AppModule {}

Caching strategies in ngsw-config.json

performance (cache-first) — serve from cache and skip the network as long as a fresh copy exists in the cache. Fast and convenient for offline; ideal for instrument metadata that changes rarely.

freshness (network-first) — try the network first and fall back to the cache only on failure or timeout. The best choice for data that must be up to date, such as live quotes, while still degrading gracefully when offline.

Update flow with SwUpdate

SwUpdate — an Angular service for managing the service worker lifecycle. It exposes versionUpdates (an Observable of version events), checkForUpdate() for on-demand checks, and activateUpdate() for switching to a freshly downloaded version.

VersionReady event — a versionUpdates event whose type equals 'VERSION_READY', signalling that a new build has fully downloaded and is ready to activate. This is the moment you prompt the user to reload.

import { Injectable, inject } from '@angular/core';
import { SwUpdate, VersionReadyEvent } from '@angular/service-worker';
import { filter } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class AppUpdateService {
  private readonly swUpdate = inject(SwUpdate);

  init(): void {
    if (!this.swUpdate.isEnabled) {
      return;                              // no SW in dev or unsupported browser
    }

    // 1. React when a new version has finished downloading.
    this.swUpdate.versionUpdates
      .pipe(filter((e): e is VersionReadyEvent => e.type === 'VERSION_READY'))
      .subscribe((event: VersionReadyEvent) => {
        console.log('Update ready', event.latestVersion.hash);
        const reload = confirm('A new version is available. Reload now?');
        if (reload) {
          this.swUpdate.activateUpdate().then(() => document.location.reload());
        }
      });
  }

  // 2. Manually poll for updates (e.g. every few minutes).
  async checkForUpdate(): Promise<boolean> {
    return this.swUpdate.checkForUpdate();
  }
}

Pitfall: never reload the page automatically in the middle of trading. In a FinTech app an unexpected reload can interrupt the submission of a pending Order — always ask and let the trader choose when to activate the new version.

WebWorker in Angular

WebWorker — a separate JavaScript context on its own OS thread with no access to the DOM, window or the Angular injector. It communicates with the page only by passing serializable messages via postMessage and onmessage, which makes it ideal for keeping CPU-heavy indicator calculations off the rendering thread.

postMessage(data) — sends a structured-cloneable message to the other side of the channel (worker → page or page → worker).

onmessage / the message event — fires on the receiving side, with the data available in event.data.

Generating the worker

The CLI sets up the TypeScript configuration and creates a sample worker file for you:

// ng generate web-worker rsi
// → creates src/app/rsi.worker.ts and updates tsconfig.worker.json

Worker file — RSI calculation

The worker waits for a message, runs the heavy loop and sends the result back. The /// <reference> directive pulls in the correct types of the WebWorker library.

/// <reference lib="webworker" />
import { RsiRequest, RsiResponse } from './models/trading';

function calculateRsi(prices: readonly number[], period: number): number {
  if (prices.length <= period) {
    return 0;
  }
  let gains = 0;
  let losses = 0;
  for (let i = 1; i <= period; i++) {
    const delta = prices[i] - prices[i - 1];
    if (delta >= 0) {
      gains += delta;
    } else {
      losses -= delta;
    }
  }
  let avgGain = gains / period;
  let avgLoss = losses / period;

  for (let i = period + 1; i < prices.length; i++) {
    const delta = prices[i] - prices[i - 1];
    const gain = delta > 0 ? delta : 0;
    const loss = delta < 0 ? -delta : 0;
    avgGain = (avgGain * (period - 1) + gain) / period;
    avgLoss = (avgLoss * (period - 1) + loss) / period;
  }

  if (avgLoss === 0) {
    return 100;
  }
  const rs = avgGain / avgLoss;
  return 100 - 100 / (1 + rs);
}

addEventListener('message', ({ data }: MessageEvent<RsiRequest>) => {
  const rsi = calculateRsi(data.prices, data.period);
  const response: RsiResponse = { symbol: data.symbol, rsi };
  postMessage(response);
});

The component side

Create the worker using import.meta.url (the only form the Angular CLI bundles correctly), send the request, handle the typed result, catch errors via onerror and always call terminate() on destroy to free the thread.

import { Component, OnDestroy, signal } from '@angular/core';
import { RsiRequest, RsiResponse } from './models/trading';

@Component({
  selector: 'app-rsi-panel',
  standalone: true,
  template: `<p>RSI: {{ rsi() ?? '—' }}</p>`,
})
export class RsiPanelComponent implements OnDestroy {
  readonly rsi = signal<number | null>(null);
  private worker?: Worker;

  compute(symbol: string, prices: readonly number[]): void {
    if (typeof Worker === 'undefined') {
      return;                              // SSR / unsupported environment
    }

    // import.meta.url lets the CLI resolve & bundle the worker.
    this.worker = new Worker(new URL('./rsi.worker', import.meta.url));

    this.worker.onmessage = ({ data }: MessageEvent<RsiResponse>) => {
      this.rsi.set(data.rsi);
    };

    this.worker.onerror = (err: ErrorEvent) => {
      console.error('RSI worker failed:', err.message);
      this.rsi.set(null);
    };

    const request: RsiRequest = { symbol, prices, period: 14 };
    this.worker.postMessage(request);
  }

  ngOnDestroy(): void {
    this.worker?.terminate();              // kill the thread, avoid leaks
  }
}

Right: the heavy RSI/MACD loops run in the worker, so the fast stream of Quote updates never blocks scrolling or chart rendering on the UI thread.

Wrong: running the same loop directly in a component method freezes change detection — a backtest over 50k bars will block the tab until it finishes.

Pitfall: messages are copied by structured clone, not shared. You cannot pass a class instance with methods, a DOM node or a function — only plain data. For huge Float64Array price buffers, transfer the ArrayBuffer itself to avoid copying.

Q&A — Interview Questions

1. What is the key difference between ServiceWorker and WebWorker?

They use the same threading model but solve different problems. ServiceWorker is a network proxy: it intercepts fetch requests to cache responses, provide offline mode and handle push — it lives independently of any page and survives reloads. WebWorker is a pure-computation thread: you send it data, it does CPU-heavy work (e.g. RSI/MACD over price history) and sends the result back, and it is tied to the page that created it. Neither has DOM access. In Angular the former uses @angular/service-worker, while the latter uses new Worker().

2. When should you choose performance versus freshness in ngsw-config.json?

performance is cache-first: serve from cache and skip the network while the copy is fresh — ideal for rarely changing data such as the Instrument catalog, giving instant loads and offline support. freshness is network-first: try the network (within a timeout) and fall back to the cache only on failure — the right choice for live Quote data, where showing a stale price is unacceptable, while still degrading gracefully offline.

3. How do you safely notify users of a new app version?

Inject SwUpdate and subscribe to versionUpdates, filtering for the VERSION_READY event, which fires as soon as a new build has fully downloaded. Then prompt the user to update; on confirmation call activateUpdate() and reload the page. Never reload automatically — in a trading app a forced reload can interrupt a pending Order, so the trader should choose the moment. Always check swUpdate.isEnabled, since the SW is disabled in dev and unsupported browsers.

4. Why must you call terminate() on a WebWorker, and what data can you pass to it?

A WebWorker is a real OS thread that keeps running until you call terminate() (or it finishes on its own). Forgetting to do this in ngOnDestroy will leak threads on every component re-creation. Messages are copied by structured clone, so you can only pass serializable data — plain objects, arrays, typed arrays, Map, Set, Date — but not class instances with methods, DOM nodes or functions. For large price buffers you can transfer an ArrayBuffer to move ownership and avoid copying.

Quiz

1. Which task belongs to a WebWorker, not a ServiceWorker?
A WebWorker performs CPU-heavy computation off the UI thread. Caching, intercepting fetch and push are all ServiceWorker responsibilities.
2. Which caching strategy is best for live Quote data?
Live quotes must be up to date, so freshness tries the network first and falls back to the cache only on failure. performance (cache-first) would risk serving a stale price.
3. Which SwUpdate.versionUpdates event signals that the downloaded build is ready to activate?
VERSION_READY means a new build has fully downloaded and can be activated — the moment to prompt the user to reload. VERSION_DETECTED only means a new build has been detected.

ServiceWorker против WebWorker — сравнение

Оба выполняются в потоке, отдельном от основного UI, и оба пишутся на TypeScript, но решают противоположные задачи. Понимание того, какой из них выбрать, — частый признак senior-уровня.

ServiceWorker — фоновый скрипт, который находится между приложением и сетью, перехватывая запросы fetch, чтобы отдавать кэшированные ответы, обеспечивать офлайн-режим и доставлять push-уведомления.

WebWorker — фоновый скрипт, выполняющий JavaScript в собственном потоке исключительно ради вычислений и общающийся со страницей через передачу сообщений. У него нет роли сетевого прокси.

НазначениеСетевой прокси / кэшированиеРесурсоёмкие вычисления
Доступ к DOM❌❌
Жизненный циклНезависим от страницыПривязан к странице
Применение в трейдингеКэш данных инструментов, офлайнРасчёт RSI/MACD
Интеграция с Angular@angular/service-workernew Worker()

Мнемоника: ServiceWorker обслуживает сеть; WebWorker работает с CPU. Один переживает перезагрузку страницы, чтобы обеспечить офлайн-режим; другой умирает вместе со своей вкладкой.

ServiceWorker в Angular

Angular ServiceWorker (ngsw) — готовый service worker от Angular (ngsw-worker.js), превращающий приложение в устанавливаемое PWA. Он управляется декларативно через файл ngsw-config.json, а не написанными вручную обработчиками fetch, кэшируя ассеты и данные API и беря на себя управление обновлениями версий.

Стратегии кэширования в ngsw-config.json

performance (cache-first) — отдавать из кэша и пропускать сеть, пока в кэше есть свежая копия. Быстро и удобно для офлайна; идеально для метаданных инструментов, которые меняются редко.

freshness (network-first) — сначала пробовать сеть и обращаться к кэшу только при сбое или таймауте. Лучший выбор для данных, которые должны быть актуальными, например живых котировок, при этом сохраняя плавную деградацию в офлайне.

Поток обновлений с SwUpdate

SwUpdate — сервис Angular для управления жизненным циклом service worker. Он предоставляет versionUpdates (Observable событий версий), checkForUpdate() для проверки по запросу и activateUpdate() для переключения на свежескачанную версию.

Событие VersionReady — событие versionUpdates, чей type равен 'VERSION_READY', сигнализирующее, что новая сборка полностью скачана и готова к активации. Именно в этот момент вы предлагаете пользователю перезагрузиться.

import { Injectable, inject } from '@angular/core';
import { SwUpdate, VersionReadyEvent } from '@angular/service-worker';
import { filter } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class AppUpdateService {
  private readonly swUpdate = inject(SwUpdate);

  init(): void {
    if (!this.swUpdate.isEnabled) {
      return;                              // no SW in dev or unsupported browser

    // 1. React when a new version has finished downloading.
    this.swUpdate.versionUpdates
      .pipe(filter((e): e is VersionReadyEvent => e.type === 'VERSION_READY'))
      .subscribe((event: VersionReadyEvent) => {
        console.log('Update ready', event.latestVersion.hash);
        const reload = confirm('A new version is available. Reload now?');

Подводный камень: никогда не перезагружайте страницу автоматически в разгар торговли. В FinTech-приложении неожиданная перезагрузка может прервать отправку незавершённого Order — всегда спрашивайте и позволяйте трейдеру самому выбрать момент активации новой версии.

WebWorker в Angular

WebWorker — отдельный контекст JavaScript в собственном OS-потоке без доступа к DOM, window или инжектору Angular. Он общается со страницей только через передачу сериализуемых сообщений с помощью postMessage и onmessage, что делает его идеальным для того, чтобы держать ресурсоёмкие расчёты индикаторов вне потока рендеринга.

postMessage(data) — отправляет structured-cloneable сообщение на другую сторону канала (воркер → страница или страница → воркер).

onmessage / событие message — срабатывает на принимающей стороне, а данные лежат в event.data.

Файл воркера — расчёт RSI

Воркер ждёт сообщение, выполняет тяжёлый цикл и отправляет результат обратно. Директива /// <reference> подключает к нему корректные типы библиотеки WebWorker.

    return 0;
  }

  let losses = 0;
  for (let i = 1; i <= period; i++) {
    const delta = prices[i] - prices[i - 1];
    if (delta >= 0) {
      gains += delta;
    } else {
      losses -= delta;
    }
  }
  let avgGain = gains / period;
  let avgLoss = losses / period;

  for (let i = period + 1; i < prices.length; i++) {
    const delta = prices[i] - prices[i - 1];
    const gain = delta > 0 ? delta : 0;
    const loss = delta < 0 ? -delta : 0;

    avgLoss = (avgLoss * (period - 1) + loss) / period;

  if (avgLoss === 0) {
    return 100;
  }
  const rs = avgGain / avgLoss;
  return 100 - 100 / (1 + rs);
}

Сторона компонента

Создайте воркер в форме с import.meta.url (единственной, которую Angular CLI собирает корректно), отправьте запрос, обработайте типизированный результат, ловите ошибки через onerror и всегда вызывайте terminate() при уничтожении, чтобы освободить поток.

import { Component, OnDestroy, signal } from '@angular/core';
import { RsiRequest, RsiResponse } from './models/trading';

@Component({
  selector: 'app-rsi-panel',
  standalone: true,
  template: `<p>RSI: {{ rsi() ?? '—' }}</p>`,
})
export class RsiPanelComponent implements OnDestroy {
  readonly rsi = signal<number | null>(null);
  private worker?: Worker;

  compute(symbol: string, prices: readonly number[]): void {
    if (typeof Worker === 'undefined') {
      return;                              // SSR / unsupported environment
    }

    // import.meta.url lets the CLI resolve & bundle the worker.
    this.worker = new Worker(new URL('./rsi.worker', import.meta.url));

    this.worker.onmessage = ({ data }: MessageEvent<RsiResponse>) => {
      this.rsi.set(data.rsi);
    };

    this.worker.onerror = (err: ErrorEvent) => {
      console.error('RSI worker failed:', err.message);
      this.rsi.set(null);
    };

    const request: RsiRequest = { symbol, prices, period: 14 };
    this.worker.postMessage(request);
  }

  ngOnDestroy(): void {
    this.worker?.terminate();              // kill the thread, avoid leaks
  }
}

Правильно: тяжёлые циклы RSI/MACD выполняются в воркере, поэтому быстрый поток обновлений Quote никогда не блокирует прокрутку или отрисовку графика в потоке UI.

Неправильно: запуск того же цикла прямо в методе компонента замораживает обнаружение изменений — бэктест на 50 тыс. баров заблокирует вкладку до завершения.

Подводный камень: сообщения копируются по structured clone, а не разделяются. Нельзя передать экземпляр класса с методами, DOM-узел или функцию — только простые данные. Для огромных буферов цен Float64Array передавайте сам ArrayBuffer, чтобы избежать копирования.

Вопросы и ответы — Вопросы с собеседований

1. В чём ключевое различие между ServiceWorker и WebWorker?

Они используют одну модель потоков, но решают разные задачи. ServiceWorker — сетевой прокси: он перехватывает запросы fetch, чтобы кэшировать ответы, обеспечивать офлайн-режим и обрабатывать push — он живёт независимо от любой страницы и переживает перезагрузки. WebWorker — поток чистых вычислений: вы отправляете ему данные, он выполняет ресурсоёмкую работу (например, RSI/MACD по истории цен) и отправляет результат обратно, и он привязан к создавшей его странице. Ни один из них не имеет доступа к DOM. В Angular для первого используется @angular/service-worker, а для второго — new Worker().

2. Когда выбирать performance, а когда freshness в ngsw-config.json?

performance — cache-first: отдавать из кэша и пропускать сеть, пока копия свежая — идеально для редко меняющихся данных вроде каталога Instrument, что даёт мгновенную загрузку и поддержку офлайна. freshness — network-first: пробовать сеть (в пределах таймаута) и обращаться к кэшу только при сбое — правильный выбор для живых данных Quote, где показ устаревшей цены недопустим, при этом сохраняя плавную деградацию в офлайне.

3. Как безопасно уведомить пользователей о новой версии приложения?

Внедрите SwUpdate и подпишитесь на versionUpdates, отфильтровав событие VERSION_READY, которое срабатывает, как только новая сборка полностью скачана. Затем предложите пользователю обновиться; при подтверждении вызовите activateUpdate() и перезагрузите страницу. Никогда не перезагружайте автоматически — в торговом приложении принудительная перезагрузка может прервать незавершённый Order, поэтому момент должен выбирать трейдер. Всегда проверяйте swUpdate.isEnabled, поскольку SW отключён в dev и неподдерживаемых браузерах.

4. Почему нужно вызывать terminate() у WebWorker и какие данные ему можно передавать?

WebWorker — это настоящий OS-поток, который продолжает работать, пока вы не вызовете terminate() (или он сам не завершится). Забыв сделать это в ngOnDestroy, вы будете утекать потоки при каждом пересоздании компонента. Сообщения копируются по structured clone, поэтому передавать можно только сериализуемые данные — простые объекты, массивы, типизированные массивы, Map, Set, Date — но не экземпляры классов с методами, DOM-узлы или функции. Для больших буферов цен можно передать ArrayBuffer, чтобы перенести владение и избежать копирования.

Квиз

1. Какая задача относится к WebWorker, а не к ServiceWorker?
WebWorker выполняет ресурсоёмкие вычисления вне потока UI. Кэширование, перехват fetch и push — всё это обязанности ServiceWorker.
2. Какая стратегия кэширования лучше всего подходит для живых данных Quote?
Живые котировки должны быть актуальными, поэтому freshness сначала пробует сеть и обращается к кэшу только при сбое. performance (cache-first) рисковал бы отдать устаревшую цену.
3. Какое событие SwUpdate.versionUpdates сигнализирует, что скачанная сборка готова к активации?
VERSION_READY означает, что новая сборка полностью скачана и может быть активирована — момент, чтобы предложить пользователю перезагрузиться. VERSION_DETECTED означает лишь то, что новая сборка обнаружена.
