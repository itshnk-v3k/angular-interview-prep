Архитектура Angular приложений

Как senior-инженеры структурируют крупные Angular-приложения: умные и глупые компоненты, MVVM, масштабируемая структура папок, паттерны проектирования GoF в том виде, в котором они реально встречаются во фреймворке, принципы ООП, а также паттерны Facade / Repository / Barrel, применяемые в продакшен-кодовых базах FinTech.

Умные и глупые компоненты

Самое важное разделение в дизайне компонентов Angular. Оно удерживает работу с состоянием в нескольких компонентах-контейнерах и делает остальное дерево UI чистым, переиспользуемым и тривиальным для тестирования.

Умный (контейнерный) компонент — знает о состоянии приложения. Он внедряет сервисы, читает из стора, диспатчит экшены / обрабатывает события и передаёт простые данные вниз дочерним компонентам через инпуты. Ему принадлежит логика «что происходит».

Глупый (презентационный) компонент — ничего не знает о сервисах или состоянии. Он лишь получает данные через @Input(), испускает намерения пользователя через @Output() и рендерит. Он чистый, переиспользуемый, дружелюбный к OnPush и легко тестируется изолированно.

АспектУмный / контейнерГлупый / презентационный
Внедряет сервисы✅❌
Знает о состоянии / сторе✅❌
Взаимодействует черезСервисы, стор, роутер@Input() / @Output()
Переиспользуем между фичами❌ (привязан к фиче)✅ (легко переиспользуем)
Обнаружение измененийDefault или OnPushOnPush (чистые инпуты)
Просто юнит-тестироватьНужны замоканные сервисы✅ тривиально

Ниже OrdersContainerComponent — это умный компонент: он внедряет фасад, владеет потоком orders$ и превращает события дочерних компонентов в экшены. OrderRowComponent — глупый: чистые инпуты/аутпуты и OnPush.

// order.model.ts — shared FinTech domain types
export type OrderSide = 'BUY' | 'SELL';
export type OrderStatus = 'pending' | 'filled' | 'cancelled';

export interface Order {
  readonly id: string;
  readonly symbol: string;
  readonly side: OrderSide;
  readonly qty: number;
  readonly price: number;
  readonly status: OrderStatus;
}
// orders-container.component.ts — SMART
import { Component, inject } from '@angular/core';
import { Observable } from 'rxjs';
import { OrderFacade } from './order.facade';
import { Order } from './order.model';

@Component({
  selector: 'app-orders-container',
  standalone: true,
  imports: [OrderRowComponent, AsyncPipe],
  template: `
    <app-order-row
      *ngFor="let order of orders$ | async; trackBy: trackById"
      [order]="order"
      (cancel)="onCancel($event)">
    </app-order-row>
  `,
})
export class OrdersContainerComponent {
  private readonly facade = inject(OrderFacade);

  readonly orders$: Observable<Order[]> = this.facade.orders$;

  ngOnInit(): void {
    this.facade.loadOrders();        // dispatches, owns the state
  }

  onCancel(orderId: string): void {
    this.facade.cancelOrder(orderId); // turns child event into an action
  }

  trackById(_: number, order: Order): string {
    return order.id;
  }
}
// order-row.component.ts — DUMB (OnPush, pure)
import {
  ChangeDetectionStrategy, Component, EventEmitter, Input, Output,
} from '@angular/core';
import { Order } from './order.model';

@Component({
  selector: 'app-order-row',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <span>{{ order.symbol }}</span>
    <span>{{ order.side }} {{ order.qty }} @ {{ order.price }}</span>
    <span>{{ order.status }}</span>
    <button (click)="cancel.emit(order.id)">Cancel</button>
  `,
})
export class OrderRowComponent {
  @Input({ required: true }) order!: Order;     // data in
  @Output() cancel = new EventEmitter<string>(); // intent out
  // No services, no store — pure rendering.
}

Правило большого пальца: если компонент внедряет сервис — он умный. Держите умных компонентов мало и делайте их тонкими; выносите всё визуальное в переиспользуемые глупые компоненты, управляемые инпутами.

MVVM в Angular

MVVM (Model-View-ViewModel) — архитектурный паттерн, который отделяет пользовательский интерфейс от бизнес-логики через три слоя, связанные привязкой данных, так что View не знает напрямую, откуда берутся его данные.

Model — данные плюс бизнес-логика. В Angular это ваши сервисы, интерфейсы и доменные типы (Order, Quote, сервис OrderService, который их получает).

View — отрендеренный UI. В Angular это HTML-шаблон компонента, включая привязки и структурные директивы.

ViewModel — связующее звено, которое предоставляет данные Model и команды для View и реагирует на ввод пользователя. В Angular это класс компонента: он хранит реактивное состояние и методы, которые вызывает шаблон.

Angular по своей природе MVVM: класс компонента и есть ViewModel, а движок привязок Angular — механизм синхронизации. Привязки свойств/интерполяции проталкивают данные Model во View; привязки событий проталкивают действия пользователя обратно; [(ngModel)] делает и то и другое сразу (двусторонняя привязка).

// MODEL — data + business logic
export interface Instrument {
  readonly symbol: string;
  readonly tickSize: number;
}

@Injectable({ providedIn: 'root' })
export class InstrumentService {            // Model layer
  getBySymbol(symbol: string): Observable<Instrument> { /* ... */ }
}

// VIEWMODEL — the component class binds Model to View
@Component({
  selector: 'app-instrument-search',
  template: `
    <!-- VIEW: template HTML -->
    <input [(ngModel)]="query" placeholder="Symbol" />   <!-- two-way sync -->
    <p>Tick size: {{ instrument?.tickSize }}</p>        <!-- Model → View -->
    <button (click)="search()">Load</button>             <!-- View → ViewModel -->
  `,
})
export class InstrumentSearchComponent {     // ViewModel
  private readonly service = inject(InstrumentService);
  query = '';                                // bound state
  instrument: Instrument | null = null;

  search(): void {                           // command exposed to the View
    this.service.getBySymbol(this.query)
      .subscribe(i => (this.instrument = i));
  }
}

Двусторонняя привязка [(ngModel)] — синтаксис Angular «банан в коробке», сокращение для привязки свойства [ngModel] плюс привязки события (ngModelChange). Она автоматически синхронизирует поле ввода View и поле ViewModel — механизм синхронизации в самом сердце MVVM.

Чем MVVM отличается от MVC

АспектMVCMVVM
Третий слойController (оркеструет запрос → ответ)ViewModel (предоставляет состояние + команды)
Связь View ↔ логикаController вручную обновляет ViewАвтоматическая двусторонняя привязка данных
Осведомлённость о ViewController знает о ViewViewModel не ссылается на View
Типичное место обитанияСерверные фреймворки (Rails, Spring MVC)Богатые клиентские UI (Angular, WPF)
Шаблонный код для синхронизации UIВысокий (ручная разводка)Низкий (привязка делает это)

Ключевая мысль: в MVC Controller активно лезет во View; в MVVM ViewModel просто предоставляет данные, а движок привязок Angular выполняет синхронизацию. ViewModel никогда не импортирует и не трогает DOM, поэтому он так хорошо тестируется.

Структура Feature / Core / Shared

Общепринятая масштабируемая структура для корпоративного Angular-приложения на standalone-API. Она разделяет синглтоны (Core), переиспользуемые части (Shared) и бизнес-домены (Features), так что приложение растёт по фичам без перепутывания.

src/
└── app/
    ├── core/                      # app-wide SINGLETONS (loaded once)
    │   ├── auth/
    │   │   ├── auth.service.ts
    │   │   └── auth.guard.ts
    │   ├── interceptors/
    │   │   ├── auth.interceptor.ts
    │   │   └── error.interceptor.ts
    │   └── guards/
    │       └── role.guard.ts
    ├── shared/                    # REUSABLE, stateless building blocks
    │   ├── components/
    │   │   ├── order-row/
    │   │   └── price-badge/
    │   ├── pipes/
    │   │   └── currency-precision.pipe.ts
    │   └── directives/
    │       └── highlight-change.directive.ts
    ├── features/                  # BUSINESS DOMAINS (lazy-loaded)
    │   ├── orders/
    │   │   ├── orders-container.component.ts
    │   │   ├── order-row.component.ts
    │   │   ├── order.facade.ts
    │   │   ├── order.service.ts
    │   │   └── orders.routes.ts
    │   └── charts/
    │       ├── base-chart.component.ts
    │       ├── candlestick-chart.component.ts
    │       └── line-chart.component.ts
    ├── app.component.ts
    ├── app.config.ts              # providers, router, interceptors
    └── app.routes.ts              # top-level (lazy) routes

core/ — общеприложенческие синглтоны, которые должны существовать ровно один раз: аутентификация, гарды маршрутов, HTTP-интерцепторы, глобальная обработка ошибок, конфигурация уровня приложения. Они помечены providedIn: 'root' и логически никогда не импортируются фичами более одного раза.

shared/ — не имеющие состояния переиспользуемые презентационные строительные блоки без бизнес-логики: глупые компоненты, пайпы и директивы. Всё, что могло бы появиться в двух разных фичах, живёт здесь. Этот слой никогда не должен зависеть от фичи.

features/ — самодостаточные бизнес-домены (orders, charts). Каждый владеет своими умными/глупыми компонентами, сервисами, фасадом и собственными лениво загружаемыми маршрутами. Фичи могут зависеть от core и shared, но не друг от друга.

app.config.ts — конфигурация standalone-бутстрапа: провайдеры приложения, настройка роутера и разводка интерцепторов / HTTP-клиента, передаваемая в bootstrapApplication.

Правило направления зависимостей: features → shared → core. Никогда не позволяйте shared импортировать из фичи и никогда не позволяйте одной фиче импортировать из другой — направляйте через общую абстракцию или стор.

Паттерны GoF в Angular

Паттерны проектирования «банды четырёх» в Angular не академичны — фреймворк построен из них. Их распознавание — частый сигнал на senior-собеседовании.

Observer

Observer — определяет зависимость «один ко многим», так что при изменении состояния одного объекта все его зависимые автоматически уведомляются. В Angular он проявляется как RxJS Observables / Subjects: подписчики реагируют на каждое испущенное значение.

const quote$ = new Subject<Quote>();           // subject = observable
quote$.subscribe(q => console.log('A', q.bid)); // observer 1
quote$.subscribe(q => console.log('B', q.ask)); // observer 2
quote$.next({ symbol: 'EURUSD', bid: 1.07, ask: 1.0702 }); // notifies all

Factory

Factory — делегирует создание объекта функции вместо конструктора, так что вызывающий код отвязан от конкретного класса. В Angular он проявляется как провайдеры DI useFactory, которые строят зависимость из других внедрённых значений.

export const ORDER_REPO = new InjectionToken<IOrderRepository>('OrderRepo');

providers: [
  {
    provide: ORDER_REPO,
    useFactory: (http: HttpClient) => new HttpOrderRepository(http),
    deps: [HttpClient],          // injected into the factory
  },
];

Decorator

Decorator — динамически добавляет объекту дополнительное поведение/метаданные. @Component, @Injectable, @Pipe в Angular — это декораторы, которые аннотируют класс метаданными, читаемыми компилятором.

@Injectable({ providedIn: 'root' })   // decorator adds DI metadata
export class OrderService {}

@Pipe({ name: 'pnl' })                 // decorator adds pipe metadata
export class PnlPipe implements PipeTransform {
  transform(value: number): string { return value.toFixed(2); }
}

Strategy

Strategy — инкапсулирует взаимозаменяемые алгоритмы за общим интерфейсом, выбираемые во время выполнения. ChangeDetectionStrategy в Angular переключает алгоритм обнаружения изменений (Default или OnPush) для каждого компонента.

@Component({
  selector: 'app-order-row',
  changeDetection: ChangeDetectionStrategy.OnPush, // pick the strategy
  template: `...`,
})
export class OrderRowComponent {}

Command

Command — инкапсулирует запрос в самостоятельный объект, несущий всю информацию для его выполнения. В Angular это NgRx Actions: типизированный объект, описывающий «что произошло», диспатчится, чтобы быть обработанным позже редьюсерами/эффектами.

export const cancelOrder = createAction(
  '[Orders] Cancel Order',
  props<{ orderId: string }>(),    // the command's payload
);

store.dispatch(cancelOrder({ orderId: 'ord-42' })); // request as an object

Proxy

Proxy — суррогат, который контролирует доступ к другому объекту, перехватывая вызовы. HttpInterceptor в Angular проксирует каждый HTTP-запрос, имея возможность проинспектировать, изменить или замкнуть его до того, как он достигнет сервера.

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const cloned = req.clone({               // proxy: modify the request
    setHeaders: { Authorization: 'Bearer token' },
  });
  return next(cloned);                      // forward to the real handler
};

Chain of Responsibility

Chain of Responsibility — передаёт запрос по цепочке обработчиков, каждый из которых решает, обработать его и/или передать дальше. Цепочка интерцепторов в Angular — именно это: каждый интерцептор вызывает next(), чтобы передать запрос следующему обработчику.

// Order matters: each link calls next() to pass control along the chain.
provideHttpClient(
  withInterceptors([
    authInterceptor,    // 1. adds token, then next()
    loggingInterceptor, // 2. logs, then next()
    errorInterceptor,   // 3. catches errors, then next()
  ]),
);

Singleton

Singleton — гарантирует, что у класса ровно один экземпляр с глобальной точкой доступа. providedIn: 'root' в Angular создаёт один общий экземпляр на всё приложение через корневой инжектор.

@Injectable({ providedIn: 'root' }) // one instance, app-wide
export class QuoteService {
  private readonly quotes$ = new BehaviorSubject<Quote[]>([]);
}
// Every component that injects QuoteService gets the SAME instance.

Template Method

Template Method — определяет скелет алгоритма, откладывая конкретные шаги на подклассы/хуки. Хуки жизненного цикла в Angular (ngOnInit, ngOnChanges, ngOnDestroy…) — это точки-хуки, которые фреймворк вызывает в фиксированном порядке.

// Angular runs a fixed sequence; you fill in the steps:
export class ChartComponent implements OnInit, OnDestroy {
  ngOnInit(): void { /* framework calls this step on init */ }
  ngOnDestroy(): void { /* and this step on teardown */ }
}

Facade

Facade — упрощённый единый интерфейс перед сложной подсистемой. В Angular это сервис, оборачивающий стор NgRx / несколько сервисов, так что компоненты зависят от одного чистого API. Подробно рассмотрено в следующем разделе.

Принципы ООП в Angular

Четыре столпа объектно-ориентированного программирования, каждый показан так, как он реально проявляется в дизайне сервисов и компонентов Angular.

Encapsulation (Инкапсуляция)

Encapsulation — объединение состояния с методами, которые с ним работают, и сокрытие внутреннего представления за публичным API. В Angular держите состояние сервиса private/protected и предоставляйте только API для чтения.

@Injectable({ providedIn: 'root' })
export class QuoteStore {
  // private internal state — nobody can next() into it from outside
  private readonly quotes$ = new BehaviorSubject<Quote[]>([]);

  // public read-only API — the only way in
  getQuotes(): Observable<Quote[]> {
    return this.quotes$.asObservable();
  }

  setQuotes(quotes: Quote[]): void {  // controlled mutation
    this.quotes$.next(quotes);
  }
}

Abstraction (Абстракция)

Abstraction — раскрытие того, что нечто делает, а не как. Определяйте интерфейсы как контракты и зависите от абстракции, а не от конкретного класса — так реализацию можно свободно подменять.

// Contract — the abstraction
export interface IOrderRepository {
  getOrders(): Observable<Order[]>;
  addOrder(order: Order): Observable<Order>;
}

// Consumers depend on IOrderRepository, NOT HttpOrderService.
@Injectable({ providedIn: 'root' })
export class OrderFacade {
  // injected against the abstraction (see Repository section)
  constructor(private readonly repo: IOrderRepository) {}
}

Inheritance (Наследование)

Inheritance — класс наследует состояние и поведение от базового класса через extends, переиспользуя общую логику. Полезно для базового компонента/сервиса, который разделяют несколько специализированных.

// Base holds shared chart plumbing
@Directive()
export abstract class BaseChartComponent {
  protected readonly ctx = inject(ChartContextService);
  protected destroy(): void { /* shared teardown */ }
  abstract render(data: Quote[]): void;   // each subtype implements
}

export class CandlestickChartComponent extends BaseChartComponent {
  render(data: Quote[]): void { /* candlestick drawing */ }
}

export class LineChartComponent extends BaseChartComponent {
  render(data: Quote[]): void { /* line drawing */ }
}

Осторожно: предпочитайте композицию глубоким деревьям наследования. Прибегайте к extends только когда подтипы разделяют действительно общее, стабильное поведение.

Polymorphism (Полиморфизм)

Polymorphism — множество типов, реализующих один интерфейс, можно использовать взаимозаменяемо; вызывающий код работает против контракта, а во время выполнения запускается нужное конкретное поведение.

export interface IChart {
  render(data: Quote[]): void;
}

class CandlestickChart implements IChart {
  render(data: Quote[]): void { /* ... */ }
}
class LineChart implements IChart {
  render(data: Quote[]): void { /* ... */ }
}

// Used interchangeably — same call, different behaviour.
const charts: IChart[] = [new CandlestickChart(), new LineChart()];
charts.forEach(chart => chart.render(quotes));

Паттерн Facade

Facade — сервис, который оборачивает сложное управление состоянием (стор NgRx, эффекты, несколько сервисов) и предоставляет маленький, раскрывающий намерения API. Компоненты общаются только с фасадом: они читают готовые потоки и вызывают методы-действия, никогда не касаясь стора напрямую.

Зачем: компоненты остаются свободными от импортов и селекторов NgRx, стор можно рефакторить без правки UI, а публичная поверхность легко тестируется и мокается.

// order.facade.ts — wraps the store behind a clean API
import { Injectable, inject } from '@angular/core';
import { Store } from '@ngrx/store';
import { Observable } from 'rxjs';
import { Order } from './order.model';
import * as OrderActions from './order.actions';
import * as OrderSelectors from './order.selectors';

@Injectable({ providedIn: 'root' })
export class OrderFacade {
  private readonly store = inject(Store);

  // --- exposed state (selectors hidden behind simple streams) ---
  readonly orders$: Observable<Order[]> =
    this.store.select(OrderSelectors.selectAllOrders);

  readonly loading$: Observable<boolean> =
    this.store.select(OrderSelectors.selectOrdersLoading);

  // --- exposed commands (dispatch hidden) ---
  loadOrders(): void {
    this.store.dispatch(OrderActions.loadOrders());
  }

  addOrder(order: Order): void {
    this.store.dispatch(OrderActions.addOrder({ order }));
  }

  cancelOrder(orderId: string): void {
    this.store.dispatch(OrderActions.cancelOrder({ orderId }));
  }
}

Умный компонент из первого раздела потребляет этот фасад и никогда не импортирует Store, селекторы или экшены — оставаясь тонким, а стор — заменяемым.

Паттерн Repository

Repository — абстрагирует доступ к данным за интерфейсом, так что потребители зависят от IOrderRepository, а не от конкретного HTTP-сервиса. Реализацию (REST, GraphQL, WebSocket, in-memory мок) можно подменить без изменения хотя бы одного потребителя.

// 1. The contract (abstraction)
export interface IOrderRepository {
  getOrders(): Observable<Order[]>;
  getPosition(symbol: string): Observable<Position>;
  addOrder(order: Order): Observable<Order>;
}

// Position — FinTech domain type
export interface Position {
  readonly symbol: string;
  readonly netQty: number;
  readonly avgPrice: number;
}
// 2. A concrete implementation
@Injectable()
export class HttpOrderRepository implements IOrderRepository {
  private readonly http = inject(HttpClient);
  private readonly base = '/api/orders';

  getOrders(): Observable<Order[]> {
    return this.http.get<Order[]>(this.base);
  }

  getPosition(symbol: string): Observable<Position> {
    return this.http.get<Position>(`/api/positions/${symbol}`);
  }

  addOrder(order: Order): Observable<Order> {
    return this.http.post<Order>(this.base, order);
  }
}
// 3. DI token wiring — bind the abstraction to an implementation
export const ORDER_REPOSITORY =
  new InjectionToken<IOrderRepository>('OrderRepository');

// app.config.ts
providers: [
  { provide: ORDER_REPOSITORY, useClass: HttpOrderRepository },
];

// Consumer depends ONLY on the token / interface:
@Injectable({ providedIn: 'root' })
export class OrderService {
  private readonly repo = inject(ORDER_REPOSITORY);
  orders(): Observable<Order[]> { return this.repo.getOrders(); }
}

Тест на подмену: направьте токен на useClass: MockOrderRepository в тестах или useClass: WsOrderRepository для стриминга — без изменений у потребителей. В этом и вся ценность паттерна.

Barrel-экспорты (index.ts)

Barrel — файл index.ts, который реэкспортирует публичные члены папки/модуля, давая потребителям один чистый путь импорта и скрывая внутреннюю структуру файлов.

// features/orders/index.ts — the barrel (public API)
export { OrderFacade } from './order.facade';
export { OrdersContainerComponent } from './orders-container.component';
export type { Order, OrderSide, OrderStatus } from './order.model';
// internal helpers are intentionally NOT exported

// Consumer — one tidy import instead of many deep paths
import { OrderFacade, Order } from '@app/features/orders';

Будьте внимательны: слишком широкие barrel-файлы могут навредить tree-shaking и создать циклические импорты. Держите их на уровне настоящих публичных границ (фича, общая библиотека), а не каждой папки.

Монорепозиторий с Nx

Nx — система сборки и инструментарий для монорепозиториев, который размещает множество приложений и библиотек в одном рабочем пространстве, с графом проектов, генераторами и кэшированием вычислений, заточенными под корпоративный Angular.

Workspace (рабочее пространство) — единый репозиторий, содержащий все проекты, с общим инструментарием, правилами линтинга и ограничениями зависимостей.

apps/ — развёртываемые приложения (например, trading-web, admin). Это тонкие оболочки, которые компонуют библиотеки.

libs/ — основная масса кода: библиотеки feature, data-access, UI и util, которые импортируют приложения и другие библиотеки. Nx обеспечивает, какие библиотеки могут зависеть от каких.

# Scaffold a new lib or app with a generator
nx generate @nx/angular:library orders-data-access

# Build a single project (and only what it needs)
nx build trading-web

# Run tasks ONLY for projects affected by your changes
nx affected --target=test
nx affected --target=build --base=main
КомандаЧто делает
nx generateСкаффолдит приложения, библиотеки, компоненты через генераторы (единообразная структура)
nx buildСобирает проект; переиспользует кэшированный вывод, когда входные данные не изменились
nx affectedЗапускает таргет только для проектов, затронутых текущим диффом

Почему для корпоративного Angular: единый источник истины для общих библиотек, принудительные границы модулей между командами, кардинально более быстрый CI за счёт запусков только затронутого и кэширования вычислений, и единообразный скаффолдинг по десяткам проектов.

Вопросы и ответы — Вопросы с собеседования

1. В чём разница между умным и глупым компонентом?

Умный (контейнерный) компонент знает о состоянии приложения: он внедряет сервисы или фасад, читает стор, диспатчит экшены и передаёт простые данные вниз. Глупый (презентационный) компонент ничего не знает о сервисах — он лишь принимает данные через @Input(), испускает намерения пользователя через @Output() и рендерит. Глупые компоненты чистые, переиспользуемые, дружелюбные к OnPush и тривиальны для тестирования; умных компонентов должно быть мало, и они должны быть тонкими.

2. Почему Angular описывают как MVVM-фреймворк и как реализуется ViewModel?

Angular разделяет Model (сервисы, интерфейсы, доменные типы), View (HTML-шаблон) и ViewModel (класс компонента). Класс компонента — это ViewModel: он предоставляет шаблону состояние и методы-команды. Движок привязок Angular — механизм синхронизации: привязки интерполяции/свойств проталкивают данные Model во View, привязки событий проталкивают намерения обратно, а [(ngModel)] делает и то и другое. В отличие от MVC, ViewModel никогда не ссылается на View; разводку выполняет привязка данных, поэтому он так хорошо тестируется.

3. Какую проблему решает сервис Facade по сравнению с использованием NgRx напрямую в компонентах?

Он скрывает сложность стора за маленьким, раскрывающим намерения API. Компоненты потребляют готовые потоки (orders$) и вызывают методы (loadOrders(), addOrder()) без импорта Store, селекторов или экшенов. Это держит компоненты тонкими и не зависящими от фреймворка, позволяет рефакторить или заменять управление состоянием без правки UI и делает фасад лёгким для мокинга в тестах.

4. Как паттерн Repository улучшает тестируемость и гибкость?

Потребители зависят от интерфейса (IOrderRepository), привязанного к конкретной реализации через DI-токен, а не от конкретного HTTP-сервиса. Поскольку зависимость — это абстракция, вы можете подменять реализации — HttpOrderRepository в продакшене, MockOrderRepository в тестах, WsOrderRepository для стриминга — изменив один провайдер, без изменений у потребителей. Это принцип инверсии зависимостей, применённый к доступу к данным.

5. Назовите три паттерна GoF, которые использует сам фреймворк Angular, и где.

Подходят многие примеры: Observer в RxJS Observables/Subjects; Decorator в @Component/@Injectable/@Pipe; Singleton через providedIn: 'root'; Strategy в ChangeDetectionStrategy; Proxy и Chain of Responsibility в цепочке HTTP-интерцепторов; Factory в DI useFactory; Command в NgRx Actions; и Template Method в хуках жизненного цикла.

Квиз

1. Какое утверждение корректно описывает глупый (презентационный) компонент?
Глупые компоненты чистые: данные внутрь через @Input, намерения наружу через @Output, без сервисов или стора. Это делает их переиспользуемыми, дружелюбными к OnPush и тривиальными для тестирования. Состоянием владеют умные/контейнерные компоненты.
2. В отображении MVVM в Angular что играет роль ViewModel?
Класс компонента — это ViewModel: он предоставляет состояние и методы-команды, к которым привязывается шаблон (View). Model — это сервисы/интерфейсы; двусторонняя привязка [(ngModel)] — механизм синхронизации.
3. Цепочка HTTP-интерцепторов Angular — пример какого паттерна(ов) GoF?
Каждый интерцептор — это Proxy, который может проинспектировать/изменить/замкнуть запрос, а упорядоченный список, где каждый вызывает next(), — это Chain of Responsibility.
