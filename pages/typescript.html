TypeScript

Система типов, на которой работает Angular: дженерики, утилитарные типы, размеченные объединения, защитники типов, декораторы и инструментарий keyof / mapped-типов.

Дженерики

Дженерик (обобщённый тип) — параметр типа (заполнитель вроде T), который подставляется в момент использования, позволяя одному определению безопасно работать с множеством конкретных типов без потери информации о типе.

Дженерики заменяют any: вместо того чтобы выбрасывать информацию о типе, вы проносите её сквозь код. Вызывающая сторона решает, какой конкретный тип использовать, а компилятор проверяет каждое использование по нему.

// The canonical identity function — T is inferred from the argument
function identity<T>(value: T): T {
  return value;
}

const a = identity<number>(42);   // a: number
const b = identity('EURUSD');      // b: string (T inferred as string)

// Without generics you lose the type:
function bad(value: any): any { return value; } // ❌ no safety

Ограничения дженериков

Ограничение дженерика — выражение T extends X, которое ограничивает допустимые типы параметра, чтобы тело функции могло безопасно полагаться на члены X.

<T extends object> — допускает только объектные типы, блокируя примитивы.

<K extends keyof T> — ограничивает K литеральными ключами T, обеспечивая типобезопасный доступ к свойствам.

interface Order {
  id: number;
  symbol: string;
  side: 'BUY' | 'SELL';
  quantity: number;
  price: number;
  status: 'NEW' | 'FILLED' | 'CANCELLED';
}

// Constrain to objects only
function logKeys<T extends object>(obj: T): (keyof T)[] {
  return Object.keys(obj) as (keyof T)[];
}

// K must be a real key of T → return type is precise
function getProp<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const order: Order = { id: 1, symbol: 'EURUSD', side: 'BUY', quantity: 100, price: 1.07, status: 'NEW' };
const side = getProp(order, 'side');   // side: 'BUY' | 'SELL'
// getProp(order, 'foo');              // ❌ compile error — 'foo' is not a key

Значения по умолчанию для дженериков

Значение дженерика по умолчанию — резервный тип (<T = string>), используемый, когда вызывающая сторона не передаёт аргумент типа или компилятор не может его вывести.

interface ApiResponse<T = unknown> {
  data: T;
  status: number;
}

const r1: ApiResponse = { data: 'ok', status: 200 };          // T defaults to unknown
const r2: ApiResponse<Order[]> = { data: [], status: 200 };    // explicit T

Дженерики в интерфейсах и классах

// Generic interface
interface Repository<T> {
  getById(id: number): T | undefined;
  add(item: T): void;
}

// Generic class implementing it
class InMemoryStore<T extends { id: number }> implements Repository<T> {
  private items = new Map<number, T>();
  getById(id: number): T | undefined { return this.items.get(id); }
  add(item: T): void { this.items.set(item.id, item); }
}

const orders = new InMemoryStore<Order>();
orders.add(order);
const found = orders.getById(1); // found: Order | undefined

Дженерики в Angular

Весь публичный API Angular — обобщённый, и именно здесь концепция окупается каждый день.

Observable<T> — поток, излучаемые значения которого типизированы как T.

HttpClient.get<User[]>() — аргумент дженерика сообщает HttpClient форму тела ответа, чтобы возвращаемый Observable был типизирован.

signal<number>(0) — записываемый сигнал, тип значения которого — number.

BehaviorSubject<Order[]> — субъект, который хранит и переигрывает текущее значение типа Order[].

import { HttpClient } from '@angular/common/http';
import { Observable, BehaviorSubject } from 'rxjs';
import { signal, inject } from '@angular/core';

class OrderService {
  private http = inject(HttpClient);

  // The <Order[]> flows all the way to the subscriber
  getOrders(): Observable<Order[]> {
    return this.http.get<Order[]>('/api/orders');
  }

  // Holds current value, typed Order[]
  private orders$ = new BehaviorSubject<Order[]>([]);

  // A signal of number, fully typed
  count = signal<number>(0);
}

Сигнал на собеседовании: всегда передавайте дженерик в http.get<T>(). Если его забыть, ответ станет Object, и каждый последующий .map() потеряет типобезопасность.

Утилитарные типы

Утилитарный тип — встроенный обобщённый тип, который преобразует существующий тип в новый (делает поля необязательными, выбирает ключи, выводит типы возврата и т. д.), так что вам не приходится вручную писать производные формы.

Partial<T>

Partial<T> — делает каждое свойство T необязательным.

Когда использовать: DTO для обновления и PATCH-запросы, где вызывающая сторона отправляет только изменённые поля.

function updateOrder(id: number, changes: Partial<Order>): void { /* ... */ }
updateOrder(1, { price: 1.085, status: 'FILLED' }); // only the fields that changed

Required<T>

Required<T> — делает каждое свойство T обязательным (убирает ?).

Когда использовать: проверка, что объект конфигурации, собранный из необязательных частей, теперь полностью заполнен.

interface ChartConfig { theme?: string; gridlines?: boolean; }
function render(cfg: Required<ChartConfig>): void { /* theme & gridlines guaranteed */ }

Readonly<T>

Readonly<T> — помечает каждое свойство T как readonly, блокируя переприсваивание.

Когда использовать: неизменяемые объекты состояния (NgRx), где мутация — это баг.

const frozenOrder: Readonly<Order> = order;
// frozenOrder.price = 2; // ❌ Cannot assign to 'price' because it is read-only

Pick<T, K>

Pick<T, K> — создаёт тип, содержащий только ключи K из T.

Когда использовать: облегчённое представление/сводка более крупной модели.

type OrderSummary = Pick<Order, 'id' | 'symbol' | 'status'>;
const summary: OrderSummary = { id: 1, symbol: 'EURUSD', status: 'NEW' };

Omit<T, K>

Omit<T, K> — создаёт тип со всеми ключами T кроме K.

Когда использовать: DTO для создания, который опускает поля, генерируемые сервером, например id.

// Client sends everything except the server-assigned id
type CreateOrder = Omit<Order, 'id'>;
const draft: CreateOrder = { symbol: 'BTCUSD', side: 'BUY', quantity: 2, price: 65000, status: 'NEW' };

Record<K, V>

Record<K, V> — строит объектный тип, ключи которого — K, а все значения — V.

Когда использовать: словари / справочные карты по идентификатору или символу.

interface Quote { symbol: string; bid: number; ask: number; timestamp: number; }

// A live price map keyed by symbol
const priceMap: Record<string, Quote> = {
  EURUSD: { symbol: 'EURUSD', bid: 1.07, ask: 1.0702, timestamp: Date.now() },
  BTCUSD: { symbol: 'BTCUSD', bid: 64990, ask: 65010, timestamp: Date.now() },
};
const eur = priceMap['EURUSD']; // Quote

NonNullable<T>

NonNullable<T> — удаляет null и undefined из T.

Когда использовать: после защитника типа, чтобы выразить, что значение больше не может быть нулевым.

type MaybeQuote = Quote | null | undefined;
type DefiniteQuote = NonNullable<MaybeQuote>; // Quote

ReturnType<F>

ReturnType<F> — извлекает тип возврата функционального типа F.

Когда использовать: вывод типа из фабрики/селектора вместо его дублирования.

function buildPosition(symbol: string, qty: number) {
  return { symbol, quantity: qty, avgPrice: 0, unrealizedPnl: 0 };
}
type Position = ReturnType<typeof buildPosition>; // { symbol; quantity; avgPrice; unrealizedPnl }

Parameters<F>

Parameters<F> — извлекает типы параметров функционального типа F в виде кортежа.

Когда использовать: обёртывание/декорирование функции с сохранением её точного списка аргументов.

type BuildArgs = Parameters<typeof buildPosition>; // [symbol: string, qty: number]

function logged(...args: BuildArgs): Position {
  console.log('building', args);
  return buildPosition(...args);
}

Exclude<T, U>

Exclude<T, U> — удаляет из объединения T члены, присваиваемые к U.

Когда использовать: сужение объединения статусов до подмножества состояний.

type Status = 'NEW' | 'FILLED' | 'CANCELLED';
type ActiveStatus = Exclude<Status, 'CANCELLED'>; // 'NEW' | 'FILLED'

Extract<T, U>

Extract<T, U> — оставляет из объединения T только члены, присваиваемые к U. Противоположность Exclude.

Когда использовать: извлечение одной формы из размеченного объединения по его тегу.

type Side = 'BUY' | 'SELL' | 'HOLD';
type Tradable = Extract<Side, 'BUY' | 'SELL'>; // 'BUY' | 'SELL'

InstanceType<T>

InstanceType<T> — извлекает тип экземпляра, создаваемого конструкторным типом T.

Когда использовать: типизация значения, созданного из ссылки на класс (например, класса DI-токена).

class OrderBook { orders: Order[] = []; }
type Book = InstanceType<typeof OrderBook>; // OrderBook
const b: Book = new OrderBook();

type против interface

interface — именованный контракт, описывающий форму объекта или класса, который поддерживает слияние объявлений и extends.

псевдоним типа (type alias) — имя для любого типа, включая объединения, пересечения, кортежи, примитивы и mapped-типы.

// interface — object/class contracts, can be merged & extended
interface Instrument {
  symbol: string;
  type: 'FOREX' | 'STOCK' | 'CRYPTO';
  price: number;
  change: number;
}
interface Instrument { exchange?: string; } // declaration merging — now also has exchange

Правило большого пальца: используйте interface для моделей и контрактов классов (расширяемый, поддерживает слияние, понятнее ошибки); используйте type для объединений, пересечений, кортежей и псевдонимов утилитарных типов.

Размеченные объединения

Размеченное объединение — объединение объектных типов, которые все разделяют общее литеральное свойство-«тег», что позволяет компилятору сузить тип до одного конкретного члена, как только вы проверите тег.

Общее литеральное поле (здесь type) — это дискриминант. switch по нему сужает каждую ветвь до одного члена, так что payload корректно типизирован в каждом случае — идеально для обработки WebSocket-сообщений в торговой платформе.

interface Quote { symbol: string; bid: number; ask: number; timestamp: number; }
interface Order { id: number; symbol: string; side: 'BUY' | 'SELL'; quantity: number; price: number; status: string; }

// Each member shares a literal `type` discriminant
type ApiEvent =
  | { type: 'quote'; payload: Quote }
  | { type: 'order'; payload: Order }
  | { type: 'error'; message: string };

function handle(event: ApiEvent): void {
  switch (event.type) {
    case 'quote':
      // event narrowed to { type: 'quote'; payload: Quote }
      updateTicker(event.payload.symbol, event.payload.bid);
      break;
    case 'order':
      // event.payload is Order here
      console.log('order filled', event.payload.id);
      break;
    case 'error':
      // no payload on this member — only message
      console.error(event.message);
      break;
  }
}

// Real usage: parse incoming WebSocket frames
socket.onmessage = (e: MessageEvent) => handle(JSON.parse(e.data) as ApiEvent);

За что это любят сеньорские команды: добавление нового варианта сообщения заставляет каждый switch его обработать (с исчерпывающим never по умолчанию — см. ниже), превращая баг времени выполнения в ошибку компиляции.

never, unknown, any

any — полностью отключает проверку типов; разрешена любая операция, а ошибки типов подавляются. Избегайте его — это дыра в системе типов.

unknown — типобезопасный аналог any: он принимает любое значение, но не даёт вам делать с ним ничего, пока вы не сузите его защитником типа.

never — тип без возможных значений; используется для функций, которые никогда не возвращают управление (бросают исключение/зацикливаются), и для доказательства исчерпывающести switch.

Почему any — это плохо: он распространяется незаметно — один any может отключить проверку типов по всей цепочке вызовов.

const raw: any = JSON.parse('{}');
raw.foo.bar.baz();        // compiles, then explodes at runtime — no safety

// unknown forces you to check first
const safe: unknown = JSON.parse('{}');
// safe.foo;              // ❌ compile error — must narrow
if (typeof safe === 'object' && safe !== null && 'symbol' in safe) {
  console.log((safe as Quote).symbol); // ✅ only after narrowing
}

Исчерпывающие проверки с never

Паттерн assertNever: в ветке default присвойте значение к never. Если добавлен новый член объединения, но он не обработан, присваивание не скомпилируется.

function assertNever(x: never): never {
  throw new Error('Unhandled case: ' + JSON.stringify(x));
}

function describe(event: ApiEvent): string {
  switch (event.type) {
    case 'quote': return 'quote ' + event.payload.symbol;
    case 'order': return 'order ' + event.payload.id;
    case 'error': return 'error ' + event.message;
    default:
      // If a new ApiEvent variant is added, `event` is no longer `never`
      // here and this line fails to compile — forcing you to handle it.
      return assertNever(event);
  }
}

Ход сеньора: сочетайте размеченные объединения с assertNever, чтобы компилятор гарантировал обработку каждого случая — проверка исчерпывающести бесплатно.

Декораторы

Декоратор — функция с префиксом @, которая получает метаданные о классе, методе, свойстве или параметре, который она аннотирует, и может читать или изменять его поведение.

Angular читает метаданные декораторов на этапе сборки/во время выполнения, чтобы связать компоненты, внедрение, входы и привязки DOM.

Декораторы классов

@Component — объявляет класс компонентом с шаблоном, селектором и стилями.

@Directive — объявляет класс директивой, которая добавляет поведение к существующему элементу.

@Injectable — помечает класс как доступный для внедрения зависимостей (и где он предоставляется).

@Pipe — объявляет класс пайпом шаблона, который преобразует значение для отображения.

@NgModule — объявляет NgModule, группируя объявления, импорты и провайдеры (устаревшее в standalone-приложениях).

@Component({
  selector: 'app-ticker',
  standalone: true,
  template: `{{ symbol }}: {{ price | number }}`,
})
export class TickerComponent {
  symbol = 'EURUSD';
  price = 1.07;
}

@Directive({ selector: '[appHighlight]', standalone: true })
export class HighlightDirective {}

@Injectable({ providedIn: 'root' })
export class OrderService {}

@Pipe({ name: 'pnl', standalone: true })
export class PnlPipe {
  transform(value: number): string { return (value >= 0 ? '+' : '') + value.toFixed(2); }
}

@NgModule({ declarations: [], imports: [], providers: [] })
export class LegacyModule {}

Декораторы свойств и методов

@Input — помечает свойство как привязываемый вход, который может установить родитель.

@Output — предоставляет EventEmitter, который дочерний компонент использует для излучения событий родителю.

@ViewChild — запрашивает один элемент/компонент/директиву из собственного шаблона компонента.

@ContentChild — запрашивает один спроецированный элемент из ng-content компонента.

@ViewChildren — запрашивает все совпадающие элементы во view как QueryList.

@ContentChildren — запрашивает все совпадающие спроецированные элементы как QueryList.

@HostListener — привязывает метод к DOM-событию на хост-элементе.

@HostBinding — привязывает свойство/атрибут/класс хост-элемента к полю компонента.

@Component({ selector: 'app-order-row', standalone: true, template: `...` })
export class OrderRowComponent {
  @Input() order!: Order;
  @Output() cancel = new EventEmitter<number>();

  @ViewChild('priceCell') priceCell!: ElementRef<HTMLElement>;
  @ViewChildren(TickerComponent) tickers!: QueryList<TickerComponent>;
  @ContentChild('footer') footer?: ElementRef<HTMLElement>;
  @ContentChildren(OrderRowComponent) rows!: QueryList<OrderRowComponent>;

  @HostBinding('class.filled') get isFilled() { return this.order.status === 'FILLED'; }

  @HostListener('click', ['$event'])
  onClick(e: MouseEvent): void {
    this.cancel.emit(this.order.id);
  }
}

Обычные декораторы классов и методов TS

Вне Angular декораторы — это просто функции. Декоратор класса получает конструктор; декоратор метода получает цель, ключ и дескриптор свойства и может обернуть исходный метод.

// Method decorator that logs every call
function log(_target: object, key: string, desc: PropertyDescriptor): void {
  const original = desc.value as (...args: unknown[]) => unknown;
  desc.value = function (...args: unknown[]): unknown {
    console.log('call', key, args);
    return original.apply(this, args);
  };
}

class PriceCalc {
  @log
  spread(q: Quote): number { return q.ask - q.bid; }
}

Angular использует устаревшую (экспериментальную) семантику декораторов, включаемую через experimentalDecorators в tsconfig.json. Новые декораторы TC39 Stage-3 отличаются по сигнатуре.

Защитники типов

Защитник типа (type guard) — проверка во время выполнения, которую понимает компилятор, сужающая значение до более конкретного типа внутри защищённого блока.

Защитник typeof — сужает примитивы (string, number, boolean, symbol, undefined, function, bigint).

Защитник instanceof — сужает до класса, проверяя цепочку прототипов.

Защитник in — сужает, проверяя, существует ли свойство в объекте.

Пользовательский предикат типа — функция, возвращающая value is SomeType, которая учит компилятор, как сужать тип.

// 1. typeof — primitives
function format(price: number | string): string {
  if (typeof price === 'number') return price.toFixed(2); // price: number here
  return price.trim();                                     // price: string here
}

// 2. instanceof — classes
function report(err: unknown): void {
  if (err instanceof Error) console.error(err.message);   // err: Error
}

// 3. in — object property presence (discriminate two shapes)
type Result = { data: Order } | { error: string };
function show(r: Result): void {
  if ('error' in r) console.error(r.error);               // r: { error: string }
  else console.log(r.data.id);                            // r: { data: Order }
}

// 4. Custom type predicate — `x is Order`
function isOrder(x: unknown): x is Order {
  return typeof x === 'object' && x !== null
    && 'id' in x && 'symbol' in x && 'side' in x;
}

function process(payload: unknown): void {
  if (isOrder(payload)) {
    // payload is now typed Order across this block
    console.log(payload.symbol, payload.side);
  }
}

Сигнал на собеседовании: пользовательский предикат возвращает x is Order, а не boolean. С обычным boolean компилятор не сужает тип после вызова.

keyof, typeof, Mapped-типы

keyof T — порождает объединение литералов ключей свойств T.

typeof variable — в позиции типа выводит статический тип из значения времени выполнения или константы.

Mapped-тип — итерирует по ключам типа ({ [K in keyof T]: ... }), чтобы преобразовать каждое свойство сразу.

keyof

type OrderKey = keyof Order; // 'id' | 'symbol' | 'side' | 'quantity' | 'price' | 'status'

function pluck<T, K extends keyof T>(rows: T[], key: K): T[K][] {
  return rows.map(r => r[key]);
}
const symbols = pluck(orders, 'symbol'); // string[]

typeof

// Derive a type from a runtime config constant — single source of truth
const defaultQuote = { symbol: 'EURUSD', bid: 0, ask: 0, timestamp: 0 };
type Quote = typeof defaultQuote; // { symbol: string; bid: number; ask: number; timestamp: number }

// Combine with keyof for literal-union enums
const SIDES = { buy: 'BUY', sell: 'SELL' } as const;
type SideValue = (typeof SIDES)[keyof typeof SIDES]; // 'BUY' | 'SELL'

Mapped-типы

// Make every field optional (this is literally how Partial is defined)
type MyPartial<T> = { [K in keyof T]?: T[K] };

// Make every field readonly
type Immutable<T> = { readonly [K in keyof T]: T[K] };

// Transform value types: wrap each field of Quote in a signal
import { Signal } from '@angular/core';
type Signalify<T> = { [K in keyof T]: Signal<T[K]> };
type QuoteSignals = Signalify<Quote>;
// { symbol: Signal<string>; bid: Signal<number>; ask: Signal<number>; timestamp: Signal<number> }

Большинство утилитарных типов (Partial, Readonly, Record, Pick) сами являются mapped-типами — понимание mapped-типов означает, что вы можете читать исходный код стандартной библиотеки.

Вопросы и ответы — Вопросы с собеседований

1. В чём разница между unknown и any?

Оба принимают любое значение, но any отключает проверку типов — вы можете вызывать/обращаться к чему угодно без ошибки, поэтому баги просачиваются в рантайм. unknown типобезопасен: компилятор запрещает любую операцию над ним, пока вы не сузите его защитником (typeof, instanceof, in или предикатом). Используйте unknown для значений неопределённой формы (например, JSON.parse, пойманные ошибки) и оставьте any для демонстрации, почему он опасен.

2. Когда вы бы использовали interface вместо type?

Используйте interface для форм объектов/моделей и контрактов классов: он поддерживает слияние объявлений, extends и выдаёт более понятные сообщения об ошибках, а соглашения Angular предпочитают его для моделей. Используйте type, когда вам нужно объединение, пересечение, кортеж, псевдоним примитива или mapped/утилитарный тип — возможности, которых нет у interface. Оба могут описывать формы объектов, поэтому для простых моделей это в основном вопрос стиля.

3. Как размеченное объединение обеспечивает исчерпывающую проверку?

Каждый член объединения разделяет литеральный дискриминант (например, type: 'quote'). switch по этому полю сужает каждую ветвь до одного члена. В ветке default, если каждый случай обработан, тип значения — never; передача его в assertNever(x: never) компилируется. Добавьте новый вариант и забудьте случай, и значение больше не будет never, так что вызов не скомпилируется — компилятор заставляет вас обработать его.

4. Что возвращает пользовательский защитник типа и почему не просто boolean?

Он возвращает предикат типа value is SomeType. Эта сигнатура сообщает компилятору, что когда функция возвращает true, аргумент можно сузить до SomeType в вызывающем блоке. Если бы он возвращал обычный boolean, вызов бы отработал, но компилятор не сузил бы переменную после него, так что вам всё равно пришлось бы делать приведение типа.

5. Зачем передавать дженерик в HttpClient.get<T>()?

get<Order[]>('/api/orders') типизирует возвращаемый Observable<Order[]>, так что каждый подписчик, map и привязка в шаблоне проверяются по реальной форме. Если его опустить, ответ по умолчанию станет Object, что стирает типобезопасность и IntelliSense для остальной части цепочки. Учтите, что это утверждение времени компиляции, а не валидация во время выполнения — сервер всё ещё может прислать другую форму.

Тест

1. Какой утилитарный тип создаёт DTO для обновления, где каждое поле Order необязательно?
Partial<T> делает каждое свойство необязательным — идеально для PATCH/обновляющих запросов. Required делает противоположное; Readonly блокирует переприсваивание; Pick выбирает подмножество.
2. Что может выразить псевдоним type, чего не может interface?
Только `type` может быть псевдонимом типов-объединений. Интерфейсы обрабатывают формы объектов и расширение и уникально поддерживают слияние объявлений — но не могут быть объединением.
3. Каков тип x в ветке default исчерпывающего switch по размеченному объединению?
Если каждый случай обработан, не остаётся возможных значений, поэтому суженный тип — `never`. Передача его в assertNever(x: never) компилируется только пока switch остаётся исчерпывающим.
4. Что порождает keyof Order для Order { id; symbol; side; quantity; price; status }?
keyof T порождает объединение литералов ключей свойств T — здесь шесть имён ключей — а не типы их значений.
