Auth: Login, Logout & TokensАутентификация: вход, выход и токены

Login/logout flows, JWT access & refresh tokens, where to store them, attaching tokens via an interceptor, silent refresh with rotation, and clean logout. Потоки входа и выхода, JWT access- и refresh-токены, где их хранить, добавление токена через интерцептор, «тихое» обновление с ротацией и корректный выход.

Authentication flow

Authentication — the process of proving who the user is (login), after which the server issues tokens the client attaches to every protected request.

A typical token-based (stateless) flow on a trading platform:

  1. User submits credentials → POST /auth/login.
  2. Server returns an access token (short-lived) and a refresh token (long-lived).
  3. Client stores the tokens and attaches the access token to every API call (Authorization: Bearer …).
  4. When the access token expires, the client silently exchanges the refresh token for a new pair.
  5. On logout the client discards the tokens and (ideally) revokes the refresh token server-side.

Stateless: a JWT is self-contained — the server validates the signature and expiry without a session lookup, which scales well for real-time systems.

Access vs Refresh tokens

A JWT has three parts — header.payload.signature. The payload carries claims such as sub (user id), exp (expiry), and roles. Never trust the payload without verifying the signature, and never put secrets in it — it is only Base64-encoded, not encrypted.

Pitfall: putting a long-lived refresh token in localStorage means a single XSS can exfiltrate it and impersonate the user indefinitely. Prefer an httpOnly, Secure, SameSite cookie for the refresh token and keep the access token in memory.

Login

Login — exchange credentials for tokens, then persist the auth state (current user + access token) for the app.

interface AuthTokens { accessToken: string; refreshToken: string; }
interface Credentials { email: string; password: string; }

@Injectable({ providedIn: 'root' })
export class AuthService {
  private http = inject(HttpClient);
  private router = inject(Router);

  // Access token kept in memory; survives via silent refresh, not storage
  private accessToken: string | null = null;
  readonly user = signal<User | null>(null);

  login(creds: Credentials): Observable<User> {
    return this.http.post<{ tokens: AuthTokens; user: User }>('/auth/login', creds).pipe(
      tap(({ tokens, user }) => {
        this.accessToken = tokens.accessToken; // in memory
        this.user.set(user);
        // refreshToken is set by the server as an httpOnly cookie — JS never sees it
      }),
      map(({ user }) => user),
    );
  }

  getAccessToken(): string | null { return this.accessToken; }
  isAuthenticated(): boolean { return this.accessToken !== null; }
}

Attaching the token (interceptor)

Auth interceptor — a functional HttpInterceptorFn that clones every outgoing request and adds the Authorization header.

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).getAccessToken();
  // Don't attach the token to the login/refresh calls themselves
  const isAuthCall = req.url.includes('/auth/');
  const authReq = token && !isAuthCall
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;
  return next(authReq);
};

// Register:
// provideHttpClient(withInterceptors([authInterceptor, refreshInterceptor]))

Pitfall: HttpRequest is immutable — you must req.clone(). Mutating req.headers directly does nothing.

Silent refresh & rotation

Silent refresh — when a request fails with 401, exchange the refresh token for a new access token and retry the original request, without the user noticing.

Rotation — each refresh returns a new refresh token and invalidates the old one, so a leaked refresh token is usable only once.

The hard part is concurrency: if five requests fail with 401 at once, you must refresh once and let the others wait for the result.

@Injectable({ providedIn: 'root' })
export class TokenRefreshService {
  private http = inject(HttpClient);
  private auth = inject(AuthService);
  // Shared in-flight refresh so concurrent 401s wait for one call
  private refresh$: Observable<string> | null = null;

  refreshToken(): Observable<string> {
    if (!this.refresh$) {
      this.refresh$ = this.http.post<{ accessToken: string }>('/auth/refresh', {}).pipe(
        map(res => res.accessToken),
        tap(token => this.auth.setAccessToken(token)),
        shareReplay(1),                 // share the single result
        finalize(() => (this.refresh$ = null)), // allow the next cycle
      );
    }
    return this.refresh$;
  }
}

export const refreshInterceptor: HttpInterceptorFn = (req, next) => {
  const refresher = inject(TokenRefreshService);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401 && !req.url.includes('/auth/')) {
        return refresher.refreshToken().pipe(
          switchMap(token => next(req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }))),
          catchError(e => { inject(AuthService).logout(); return throwError(() => e); }),
        );
      }
      return throwError(() => err);
    }),
  );
};

Why shareReplay(1): it multicasts the single refresh call to all waiting requests, so the refresh endpoint is hit once even under a burst of parallel 401s.

Logout & cleanup

Logout — discard local auth state and revoke the refresh token server-side so it cannot be reused.

logout(): void {
  // Ask the server to revoke the refresh token / clear the httpOnly cookie
  this.http.post('/auth/logout', {}).subscribe({ complete: () => {} });
  this.accessToken = null;
  this.user.set(null);
  this.router.navigateByUrl('/login');
}

Pitfall: clearing the client token alone does not log the user out everywhere — a stolen refresh token stays valid until the server revokes it. Always revoke server-side and rely on rotation.

Security notes

  • XSS is the main threat to in-browser tokens — sanitize output, use Angular's built-in escaping, and avoid bypassSecurityTrust* unless you must.
  • CSRF matters when tokens live in cookies — use SameSite=Strict/Lax and an anti-CSRF token for state-changing requests.
  • Always serve over HTTPS; mark cookies Secure.
  • Keep access tokens short-lived; never log them.

Q&A

1. Why use two tokens instead of one long-lived token?

A short-lived access token limits the damage if it leaks (it expires in minutes), while a long-lived refresh token — kept in a more protected place (httpOnly cookie) and rotated on each use — lets the user stay logged in without re-entering credentials. One long-lived token would combine wide exposure with long validity, the worst of both.

2. Where should the refresh token live and why?

In an httpOnly, Secure, SameSite cookie. JavaScript cannot read httpOnly cookies, so an XSS cannot exfiltrate the refresh token. The access token, by contrast, is best kept in memory and re-obtained via silent refresh.

3. How do you avoid multiple refresh calls when several requests get 401 at once?

Share a single in-flight refresh Observable (e.g. cached with shareReplay(1) and reset on finalize). All concurrent 401s subscribe to the same refresh; once it emits the new token, each retries its original request.

4. What is token rotation?

Each refresh issues a brand-new refresh token and invalidates the previous one. If an attacker replays an old refresh token after it has been rotated, the server detects reuse and can revoke the whole token family — limiting the value of a stolen token to a single use.

5. Why must the interceptor clone the request?

HttpRequest is immutable. You cannot mutate its headers in place; req.clone({ setHeaders }) returns a new request with the Authorization header added, which you pass to next().

Quiz

1. Which token is sent with every protected API request?
The short-lived access token goes in the Authorization header of each request; the refresh token is sent only to the refresh endpoint.
2. Safest place for a refresh token in the browser?
JS cannot read httpOnly cookies, so XSS cannot steal the refresh token. Pair with Secure + SameSite.
3. On a 401, the refresh interceptor should…
Exchange the refresh token for a new access token, retry the original request, and only log out if the refresh itself fails.
4. What does token rotation protect against?
Each refresh invalidates the previous refresh token, so a stolen one is good for at most a single use and reuse is detectable.

Поток аутентификации

Аутентификация — процесс подтверждения личности пользователя (вход), после которого сервер выдаёт токены, прикрепляемые клиентом к каждому защищённому запросу.

Типичный поток на основе токенов (stateless) на торговой платформе:

  1. Пользователь отправляет учётные данные → POST /auth/login.
  2. Сервер возвращает access-токен (короткоживущий) и refresh-токен (долгоживущий).
  3. Клиент сохраняет токены и добавляет access-токен к каждому вызову API (Authorization: Bearer …).
  4. Когда access-токен истекает, клиент «тихо» обменивает refresh-токен на новую пару.
  5. При выходе клиент удаляет токены и (в идеале) отзывает refresh-токен на сервере.

Stateless: JWT самодостаточен — сервер проверяет подпись и срок действия без обращения к сессии, что хорошо масштабируется для real-time систем.

JWT состоит из трёх частей — header.payload.signature. В payload лежат claims: sub (id пользователя), exp (срок), роли. Никогда не доверяйте payload без проверки подписи и не кладите туда секреты — это лишь Base64, а не шифрование.

Добавление токена (интерцептор)

Auth-интерцептор — функциональный HttpInterceptorFn, который клонирует каждый исходящий запрос и добавляет заголовок Authorization.

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).getAccessToken();
  // Don't attach the token to the login/refresh calls themselves
  const isAuthCall = req.url.includes('/auth/');
  const authReq = token && !isAuthCall
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;
  return next(authReq);
};

// Register:
// provideHttpClient(withInterceptors([authInterceptor, refreshInterceptor]))

Подводный камень: HttpRequest неизменяем — нужно вызывать req.clone(). Прямое изменение req.headers ничего не даёт.

Тихое обновление и ротация

Тихое обновление (silent refresh) — когда запрос завершается с 401, клиент обменивает refresh-токен на новый access-токен и повторяет исходный запрос незаметно для пользователя.

Ротация — каждое обновление возвращает новый refresh-токен и аннулирует старый, поэтому утёкший refresh-токен можно использовать лишь один раз.

Сложность — в параллельности: если пять запросов одновременно получили 401, обновить токен нужно один раз, а остальные должны дождаться результата.

@Injectable({ providedIn: 'root' })
export class TokenRefreshService {
  private http = inject(HttpClient);
  private auth = inject(AuthService);
  // Shared in-flight refresh so concurrent 401s wait for one call
  private refresh$: Observable<string> | null = null;

  refreshToken(): Observable<string> {
    if (!this.refresh$) {
      this.refresh$ = this.http.post<{ accessToken: string }>('/auth/refresh', {}).pipe(
        map(res => res.accessToken),
        tap(token => this.auth.setAccessToken(token)),
        shareReplay(1),                 // share the single result
        finalize(() => (this.refresh$ = null)), // allow the next cycle
      );
    }
    return this.refresh$;
  }
}

export const refreshInterceptor: HttpInterceptorFn = (req, next) => {
  const refresher = inject(TokenRefreshService);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401 && !req.url.includes('/auth/')) {
        return refresher.refreshToken().pipe(
          switchMap(token => next(req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }))),
          catchError(e => { inject(AuthService).logout(); return throwError(() => e); }),
        );
      }
      return throwError(() => err);
    }),
  );
};

Зачем shareReplay(1): он мультикастит единственный вызов обновления всем ожидающим запросам, поэтому эндпоинт обновления вызывается один раз даже при всплеске параллельных 401.

Выход и очистка

Logout — удалить локальное состояние аутентификации и отозвать refresh-токен на сервере, чтобы его нельзя было переиспользовать.

logout(): void {
  // Ask the server to revoke the refresh token / clear the httpOnly cookie
  this.http.post('/auth/logout', {}).subscribe({ complete: () => {} });
  this.accessToken = null;
  this.user.set(null);
  this.router.navigateByUrl('/login');
}

Подводный камень: очистка только клиентского токена не разлогинивает пользователя везде — украденный refresh-токен остаётся валидным, пока сервер его не отзовёт. Всегда отзывайте на сервере и полагайтесь на ротацию.

Заметки по безопасности

  • XSS — главная угроза для токенов в браузере: экранируйте вывод, используйте встроенную защиту Angular и избегайте bypassSecurityTrust* без крайней необходимости.
  • CSRF важен, когда токены в cookie — используйте SameSite=Strict/Lax и анти-CSRF токен для запросов, меняющих состояние.
  • Всегда отдавайте по HTTPS; помечайте cookie флагом Secure.
  • Держите access-токены короткоживущими; никогда не логируйте их.

Вопросы и ответы

1. Зачем два токена вместо одного долгоживущего?

Короткоживущий access-токен ограничивает ущерб при утечке (истекает за минуты), а долгоживущий refresh-токен — в более защищённом месте (httpOnly-cookie) и с ротацией при каждом использовании — позволяет оставаться в системе без повторного ввода пароля. Один долгоживущий токен сочетал бы широкую открытость с долгим сроком — худший вариант.

2. Где должен жить refresh-токен и почему?

В httpOnly-cookie с флагами Secure и SameSite. JavaScript не может прочитать httpOnly-cookie, поэтому XSS не уведёт refresh-токен. Access-токен же лучше держать в памяти и заново получать тихим обновлением.

3. Как избежать нескольких вызовов обновления, когда сразу несколько запросов получают 401?

Разделять один «в полёте» Observable обновления (например, закэшированный через shareReplay(1) и сбрасываемый в finalize). Все параллельные 401 подписываются на одно обновление; как только оно отдаёт новый токен, каждый повторяет свой исходный запрос.

4. Что такое ротация токенов?

Каждое обновление выдаёт совершенно новый refresh-токен и аннулирует предыдущий. Если злоумышленник повторно использует старый refresh-токен после ротации, сервер замечает повтор и может отозвать всё «семейство» токенов — ценность украденного токена ограничивается одним использованием.

5. Почему интерцептор обязан клонировать запрос?

HttpRequest неизменяем. Изменить заголовки на месте нельзя; req.clone({ setHeaders }) возвращает новый запрос с добавленным заголовком Authorization, который передаётся в next().

Тест

1. Какой токен отправляется с каждым защищённым запросом к API?
Короткоживущий access-токен идёт в заголовке Authorization каждого запроса; refresh-токен отправляется только на эндпоинт обновления.
2. Самое безопасное место для refresh-токена в браузере?
JS не может читать httpOnly-cookie, поэтому XSS не украдёт refresh-токен. В паре с Secure + SameSite.
3. При 401 refresh-интерцептор должен…
Обменять refresh-токен на новый access-токен, повторить исходный запрос и разлогинивать только если само обновление не удалось.
4. От чего защищает ротация токенов?
Каждое обновление аннулирует предыдущий refresh-токен, поэтому украденный годен максимум для одного использования, а повтор обнаруживается.
