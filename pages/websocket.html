WebSocket для trading платформы

Двунаправленный потоковый обмен данными в реальном времени для торговых интерфейсов: нативный API, RxJS webSocket(), продакшен-сервис переподключения, а также как сохранить плавность таблицы на 20 тыс. инструментов с помощью батчинга, зон и web worker'ов.

Основы WebSocket

WebSocket — протокол, обеспечивающий полнодуплексную связь по одному TCP-соединению. В отличие от модели «запрос-ответ» HTTP, он сохраняет постоянное соединение после начального рукопожатия, поэтому и клиент, и сервер могут отправлять сообщения в любой момент без поллинга.

Полный дуплекс — обе стороны могут передавать данные одновременно по одному соединению, в отличие от одноразового цикла «запрос → ответ» в HTTP.

Рукопожатие (handshake) — соединение открывается как HTTP-запрос с заголовком Upgrade: websocket, затем протокол переключается на WebSocket-фреймы, а TCP-сокет остаётся открытым.

ws:// vs wss:// — ws:// работает без шифрования, wss:// шифруется через TLS. В продакшен торговых системах всегда используйте wss://.

Состояния готовности (ready states)

Свойство readyState сообщает текущую фазу жизненного цикла соединения:

СостояниеЗначениеСмысл
CONNECTING0Сокет создан, рукопожатие ещё не завершено.
OPEN1Соединение установлено — можно вызывать send() и получать данные.
CLOSING2Идёт закрывающее рукопожатие.
CLOSED3Соединение закрыто или не удалось открыть.

Нативный WebSocket API

Нативный WebSocket — встроенный в браузер API. Вы создаёте его с URL, навешиваете обработчики событий (onopen, onmessage, onclose, onerror) и используете send() / close().

interface Quote {
  symbol: string;
  bid: number;
  ask: number;
  timestamp: number;
}

const socket = new WebSocket('wss://stream.exchange.example/marketdata');

// Fired once the handshake completes and readyState becomes OPEN (1)
socket.onopen = (): void => {
  // Only safe to send after OPEN
  socket.send(JSON.stringify({ action: 'subscribe', symbol: 'BTCUSD' }));
};

// Fired for every inbound frame — data arrives as a string for text frames
socket.onmessage = (event: MessageEvent<string>): void => {
  const quote = JSON.parse(event.data) as Quote;
  console.log(quote.symbol, quote.bid, quote.ask);
};

// Fired when the connection closes — inspect code & reason
socket.onclose = (event: CloseEvent): void => {
  console.warn(`closed: code=${event.code} reason=${event.reason} clean=${event.wasClean}`);
};

// Fired on transport error (no detail for security reasons) — close usually follows
socket.onerror = (): void => {
  console.error('socket error');
};

// Graceful shutdown — 1000 is the normal-closure code
function disconnect(): void {
  socket.send(JSON.stringify({ action: 'unsubscribe', symbol: 'BTCUSD' }));
  socket.close(1000, 'client navigating away');
}

Подводный камень: вызов send() до того, как readyState === OPEN, выбрасывает InvalidStateError. Всегда дожидайтесь onopen (или буферизуйте исходящие сообщения до этого момента).

Коды закрытия: 1000 = нормальное, 1001 = уход (going away), 1006 = аномальное (без close-фрейма — обычно обрыв сети). Любой код, отличный от 1000, как правило означает, что нужно переподключиться.

RxJS webSocket()

webSocket() — RxJS Subject, оборачивающий нативный WebSocket. Он предоставляет интерфейс Observable для входящих сообщений (подпишитесь, чтобы получать) и интерфейс Observer для исходящих сообщений (.next() для отправки). Сериализация/десериализация JSON по умолчанию выполняется за вас.

Ленивое подключение — сокет открывается только при первом subscribe() и закрывается, когда отписывается последний подписчик.

Мультиплексирование — несколько логических потоков (подписки по символам) могут разделять один физический сокет через оператор multiplex().

import { webSocket, WebSocketSubject } from 'rxjs/webSocket';
import { retry } from 'rxjs';

interface TradeEvent {
  symbol: string;
  price: number;
  size: number;
  timestamp: number;
}

// Typed subject — Observable<TradeEvent> out, accepts outgoing messages in
const connection$: WebSocketSubject<TradeEvent> =
  webSocket<TradeEvent>('wss://stream.exchange.example/trades');

// Send a subscribe message through the Observer interface
connection$.next({ symbol: 'EURUSD', price: 0, size: 0, timestamp: Date.now() });

// Subscribe to receive; retry reconnects 3s after an error/close
connection$
  .pipe(
    retry({ delay: 3000 }), // re-subscribe → reopens the socket after 3s
  )
  .subscribe((event: TradeEvent) => this.handleEvent(event));

Предпочитайте retry({ delay }): более старый оператор retryWhen устарел. Используйте retry({ delay: 3000 }) для фиксированной задержки или retry({ delay: (err, count) => timer(backoff(count)) }) для кастомного/экспоненциального backoff.

Примечание: retry переподписывается на источник, что заново запускает фабрику webSocket() и открывает новый сокет. Любые сообщения подписки нужно отправить заново при переподключении (см. сервис ниже).

Сервис переподключения

Продакшен-сервису рыночных данных нужно больше, чем просто голый сокет: автоматическое переподключение с экспоненциальным backoff, повтор подписок (повторная отправка подписок после переподключения), типобезопасная обработка сообщений через размеченное объединение и детерминированная очистка с takeUntilDestroyed.

Экспоненциальный backoff — каждая неудачная попытка переподключения ждёт дольше предыдущей (base · 2^attempt), с ограничением сверху, часто со случайным джиттером, чтобы избежать лавины одновременных переподключений (thundering herd).

Размеченное объединение (discriminated union) — объединение объектных типов с общим литеральным полем type, позволяющее TypeScript сужать тип до точной формы внутри switch — без any.

takeUntilDestroyed — оператор Angular, завершающий поток при уничтожении контекста инъекции (компонента/сервиса), автоматически отписываясь без ручного учёта Subscription.

import { Injectable, DestroyRef, inject } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { webSocket, WebSocketSubject } from 'rxjs/webSocket';
import { Observable, Subject, retry, timer, filter, map } from 'rxjs';

// ---- Domain types (no `any` anywhere) ----
interface Quote {
  symbol: string;
  bid: number;
  ask: number;
  timestamp: number;
}

// Outbound commands the client sends to the server
type OutboundMessage =
  | { type: 'subscribe'; symbol: string }
  | { type: 'unsubscribe'; symbol: string }
  | { type: 'heartbeat' };

// Inbound messages — a discriminated union keyed on `type`
type InboundMessage =
  | { type: 'quote'; payload: Quote }
  | { type: 'ack'; symbol: string }
  | { type: 'error'; code: number; message: string };

@Injectable({ providedIn: 'root' })
export class MarketDataService {
  private readonly destroyRef = inject(DestroyRef);

  /** The underlying RxJS WebSocket subject (recreated on each reconnect). */
  private socket$: WebSocketSubject<InboundMessage | OutboundMessage> | null = null;

  /** Symbols we are currently subscribed to — replayed after a reconnect. */
  private readonly activeSymbols = new Set<string>();

  /** Single multicast stream of all decoded inbound messages. */
  private readonly messages$ = new Subject<InboundMessage>();

  private readonly url = 'wss://stream.exchange.example/marketdata';

  constructor() {
    this.connect();
  }

  /** Opens the socket and wires reconnection with exponential backoff. */
  private connect(): void {
    this.socket$ = webSocket<InboundMessage | OutboundMessage>({
      url: this.url,
      // On (re)open, replay every active subscription so we don't miss data.
      openObserver: {
        next: () => this.activeSymbols.forEach((symbol) =>
          this.socket$?.next({ type: 'subscribe', symbol })),
      },
    });

    this.socket$
      .pipe(
        // Reconnect with exponential backoff + jitter, capped at 30s.
        retry({
          delay: (_error: unknown, attempt: number) => {
            const backoff = Math.min(1000 * 2 ** attempt, 30_000);
            const jitter = Math.random() * 1000;
            return timer(backoff + jitter);
          },
        }),
        // Auto-cleanup when the service/injection context is destroyed.
        takeUntilDestroyed(this.destroyRef),
      )
      .subscribe({
        next: (msg) => this.messages$.next(msg as InboundMessage),
        error: (err: unknown) => console.error('market data stream failed', err),
      });
  }

  /** Subscribe to a symbol; remembers it for replay after reconnects. */
  subscribe(symbol: string): void {
    this.activeSymbols.add(symbol);
    this.socket$?.next({ type: 'subscribe', symbol });
  }

  /** Unsubscribe from a symbol. */
  unsubscribe(symbol: string): void {
    this.activeSymbols.delete(symbol);
    this.socket$?.next({ type: 'unsubscribe', symbol });
  }

  /** Type-safe stream of quotes for one symbol (uses the discriminant). */
  quotes(symbol: string): Observable<Quote> {
    return this.messages$.pipe(
      // `msg.type === 'quote'` narrows `msg` to the quote variant — no cast.
      filter((msg): msg is Extract<InboundMessage, { type: 'quote' }> =>
        msg.type === 'quote'),
      map((msg) => msg.payload),
      filter((quote) => quote.symbol === symbol),
    );
  }
}

Почему это готово к продакшену: сокет переоткрывается автоматически с ограниченным backoff, подписки повторяются при каждом переподключении, поэтому ни один символ не теряется незаметно, каждое сообщение сужается по своему дискриминанту (ноль any), а takeUntilDestroyed разбирает всё при уничтожении сервиса.

Добавьте на уровне приложения heartbeat ({ type: 'heartbeat' } по интервалу) плюс таймаут на стороне сервера, чтобы обнаруживать полуоткрытые TCP-соединения, для которых onclose никогда не срабатывает.

Производительность для 20 тыс. инструментов

Живая торговая таблица может получать десятки тысяч тиков в секунду. Рендеринг каждого тика по отдельности расплавил бы основной поток. Стратегия — уменьшить количество обновлений, рендерить только то, что видно, и держать тяжёлую работу и обнаружение изменений вне критического пути.

bufferTime(100) — батчинг обновлений

bufferTime — собирает эмиссии в массив и выдаёт батч с фиксированным интервалом, поэтому UI обновляется один раз за окно, а не на каждый тик.

import { bufferTime, filter } from 'rxjs';

// Apply ~10 batched updates per second instead of thousands of ticks
quotes$
  .pipe(
    bufferTime(100),                  // collect 100ms of quotes
    filter((batch: Quote[]) => batch.length > 0),
  )
  .subscribe((batch: Quote[]) => this.applyBatch(batch));

throttleTime(50, …) — ограничение частоты по символу

throttleTime — выдаёт значение, затем игнорирует последующие значения в течение заданной длительности. С { leading: true, trailing: true } вы получаете и первый тик сразу, и последний тик каждого окна.

import { throttleTime, asyncScheduler } from 'rxjs';

// Cap one symbol's stream to ~20 updates/sec, keeping first & last
quoteForSymbol$
  .pipe(
    throttleTime(50, asyncScheduler, { leading: true, trailing: true }),
  )
  .subscribe((quote: Quote) => this.render(quote));

distinctUntilChanged(…) — пропуск холостых обновлений

distinctUntilChanged — подавляет эмиссию, если она равна предыдущей (по компаратору, который вы предоставляете), избегая лишних рендеров, когда ничего значимого не изменилось.

import { distinctUntilChanged } from 'rxjs';

interface PricePoint { symbol: string; price: number; }

priceStream$
  .pipe(
    distinctUntilChanged((a: PricePoint, b: PricePoint) => a.price === b.price),
  )
  .subscribe((p: PricePoint) => this.flashCell(p));

Виртуальный скролл и Web Worker

Виртуальный скролл — рендерит только строки, видимые во вьюпорте в данный момент (плюс небольшой буфер), переиспользуя DOM-узлы при прокрутке. С 20 тыс. инструментов в DOM одновременно существует лишь ~30 строк (Angular CDK cdk-virtual-scroll-viewport).

Web Worker — выполняет JavaScript в фоновом потоке. Вынесите парсинг сообщений и расчёт индикаторов (VWAP, скользящие средние, P&L) в worker, чтобы основной поток оставался свободным для рендеринга.

ngZone.runOutsideAngular()

ngZone.runOutsideAngular(fn) — выполняет код вне патча Zone.js Angular. Асинхронная работа, запланированная внутри fn, не запускает обнаружение изменений. Это предотвращает полный цикл CD на каждое отдельное WebSocket-сообщение. Вызывайте ngZone.run() только в тот момент, когда UI действительно нужно обновить.

Проблема: Zone.js патчит коллбэки WebSocket. Без вмешательства каждое входящее сообщение запускает обнаружение изменений по всему приложению — при тысячах тиков/сек UI зависает.

import { Component, NgZone, OnInit, inject } from '@angular/core';

interface Quote {
  symbol: string;
  bid: number;
  ask: number;
  timestamp: number;
}

@Component({ selector: 'app-blotter', template: '...' })
export class BlotterComponent implements OnInit {
  private readonly ngZone = inject(NgZone);
  private readonly ws = inject(MarketDataService);

  /** Hot cache updated outside Angular — no CD on write. */
  private readonly latestQuotes = new Map<string, Quote>();

  /** Bound in the template — only reassigned inside ngZone.run(). */
  displayQuote: Quote | null = null;

  ngOnInit(): void {
    // Subscribe OUTSIDE Angular: incoming ticks won't trigger change detection.
    this.ngZone.runOutsideAngular(() => {
      this.ws.quotes('EURUSD').subscribe((quote: Quote) => {
        // Cheap, CD-free bookkeeping for all ticks.
        this.latestQuotes.set(quote.symbol, quote);

        // Re-enter Angular ONLY when the UI genuinely needs to refresh.
        if (this.shouldUpdateUI(quote)) {
          this.ngZone.run(() => {
            this.displayQuote = quote; // assignment inside run() → CD runs
          });
        }
      });
    });
  }

  /** Throttle UI: e.g. update at most ~10x/sec or on meaningful moves. */
  private shouldUpdateUI(quote: Quote): boolean {
    const prev = this.displayQuote;
    return !prev || quote.bid !== prev.bid || quote.ask !== prev.ask;
  }
}

Результат: тысячи тиков/сек дёшево поглощаются в Map с нулевым обнаружением изменений; UI перерисовывается только при вызове ngZone.run(). Дополните это сигналом или OnPush для финального минимального рендера.

В современном zoneless Angular нет зоны, из которой нужно выходить — используйте signal() и обновляйте их напрямую; только чтения сигналов в шаблоне запускают точечное обновление.

Вопросы и ответы — Собеседование

1. Чем WebSocket отличается от HTTP-поллинга?

HTTP-поллинг постоянно открывает новые циклы «запрос-ответ» по интервалу — высокая задержка, расточительные заголовки, и сервер не может пушить. WebSocket выполняет одно HTTP-рукопожатие Upgrade, затем держит одно TCP-соединение открытым для полнодуплексного обмена: обе стороны отправляют фреймы в любой момент с минимальными накладными расходами. Для рыночных данных вы получаете субмиллисекундный пуш вместо задержки поллинга.

2. Почему в Angular предпочтительнее RxJS webSocket(), а не нативный API?

webSocket() возвращает Subject: Observable для входящих сообщений и Observer (.next()) для исходящих. Это открывает весь конвейер операторов — retry для переподключения, bufferTime/throttleTime для управления back-pressure, filter/map для маршрутизации — плюс ленивое подключение при подписке, автозакрытие при отписке и чистый teardown с takeUntilDestroyed. Также по умолчанию обрабатывается (де)сериализация JSON.

3. Как реализовать переподключение с экспоненциальным backoff?

Пропустите сокет через retry({ delay }), где функция задержки возвращает timer(Math.min(base * 2 ** attempt, cap) + jitter). Каждая неудача ждёт дольше (1с, 2с, 4с, 8с…) до предела (например, 30с), а случайный джиттер не даёт всем клиентам переподключаться одновременно. Поскольку retry переподписывается и открывает свежий сокет, нужно повторить активные подписки при переподключении — сделайте это в openObserver из запомненного Set символов.

4. Таблица на 20 тыс. инструментов подвешивает UI. Что вы сделаете?

Наслаивайте несколько техник: bufferTime(100) для батчинга обновлений ~10x/сек, throttleTime для ограничения частоты «горячего» символа, distinctUntilChanged для отбрасывания неизменившихся цен, CDK виртуальный скролл, чтобы в DOM были только видимые строки, и Web Worker для парсинга и расчёта индикаторов. Главное — подписывайтесь внутри ngZone.runOutsideAngular(), чтобы тики не запускали обнаружение изменений, переходя обратно через ngZone.run() только когда UI должен обновиться. Используйте OnPush или сигналы для финального рендера.

5. Что делает ngZone.runOutsideAngular() и когда это критично?

Он выполняет код вне зоны Angular, поэтому асинхронные коллбэки, запланированные внутри, не уведомляют Zone.js и потому не запускают обнаружение изменений. Это критично для высокочастотных источников — WebSocket-тиков, циклов requestAnimationFrame, mousemove — где запуск CD на каждое событие убил бы производительность. Вы делаете дешёвую работу вне зоны и вызываете ngZone.run() только в тот точный момент, когда UI нужно обновить. (Zoneless Angular полностью устраняет эту необходимость с помощью сигналов.)

Квиз

1. Каково числовое значение состояния готовности OPEN?
CONNECTING = 0, OPEN = 1, CLOSING = 2, CLOSED = 3. Безопасно вызывать send() можно только когда readyState равен 1 (OPEN).
2. Какие интерфейсы предоставляет subject RxJS webSocket()?
webSocket() возвращает Subject: подпишитесь для входящих сообщений, вызывайте .next() для отправки исходящих сообщений по тому же сокету.
3. Какой оператор батчит тики, чтобы UI обновлялся ~10 раз в секунду?
bufferTime(100) собирает эмиссии в течение 100мс и выдаёт их одним массивом, поэтому вы применяете ~10 батчированных обновлений в секунду вместо тысяч.
4. Зачем подписываться на WebSocket внутри ngZone.runOutsideAngular()?
Вне зоны Angular Zone.js не уведомляет Angular об асинхронном коллбэке, поэтому обнаружение изменений не запускается на каждое сообщение. Вы возвращаетесь через ngZone.run() только когда UI должен обновиться.
