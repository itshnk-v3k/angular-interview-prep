Dependency Injection (DI)Внедрение зависимостей (DI)

Angular's DI system is one of its defining features: a hierarchical tree of injectors that resolves dependencies by walking up from the requesting class. Master providedIn, provider types, InjectionToken, resolution decorators, and the modern inject() function.Система DI в Angular — одна из её определяющих особенностей: иерархическое дерево инжекторов, которое разрешает зависимости, поднимаясь вверх от запрашивающего класса. Освойте providedIn, типы провайдеров, InjectionToken, декораторы разрешения и современную функцию inject().

How DI works

Angular builds an injector tree that mirrors the component tree. When a class requests a dependency, Angular walks up this tree from the requesting injector until it finds the nearest provider for that token. The first provider found wins; if there is none, Angular throws NullInjectorError: No provider for X.

Injector — a container that knows how to create and cache dependencies for a given set of tokens. Every injector has a parent, which forms a tree.

Token — the key Angular uses to look up a provider. Usually a class type, but it can be an InjectionToken or a string.

Provider — a recipe that tells the injector how to produce a value for a token (useClass, useValue, useFactory, useExisting).

Resolution — walking up the parent injectors looking for the first matching provider. Once resolved, the instance is cached on that injector and reused by everyone below it.

Injector hierarchy (top to bottom)

1. Platform Injector — the topmost injector, created by platformBrowserDynamic()/bootstrapApplication. It holds platform-level singletons shared by all Angular apps running on the same page (for example, multiple apps in a single browser tab).

2. Root Injector — created when the application bootstraps. Everything declared as providedIn: 'root' lives here as a single instance for the whole app. The default home for application-level services.

3. Module Injector — created for each (eagerly or lazily loaded) @NgModule. Services listed in @NgModule.providers live here. Lazily loaded modules get their own child module injector.

4. Component (Element) Injector — created for each component that declares @Component.providers. It produces a new instance per component subtree, so each component (and its descendants) gets its own copy.

5. Element (directive) Injector — the most fine-grained injector, created for elements that host a @Directive with providers. Used for dependencies scoped to the directive.

// Root-level singleton: ONE PriceService for the whole app
@Injectable({ providedIn: 'root' })
export class PriceService {
  getQuote(symbol: string): Quote { /* ... */ }
}

// Component-level: a NEW instance per OrderPanel subtree
@Component({
  selector: 'app-order-panel',
  standalone: true,
  providers: [OrderDraftService],   // not shared across panels
  template: `...`,
})
export class OrderPanelComponent {
  // Angular walks up from THIS component's injector:
  // OrderDraftService found here → fresh instance;
  // PriceService not here → keeps walking up to the root injector.
  private readonly draft = inject(OrderDraftService);
  private readonly prices = inject(PriceService);
}

Mental model: resolution always goes up, never sideways or down. A sibling component cannot see a service you provided in another component — provide it higher (in the root or a shared parent) to share it.

providedIn options

The providedIn field of the @Injectable decorator declares where a service is registered, and makes it tree-shakeable: if no one injects the service, the bundler can drop it entirely.

'root' — registers in the root injector: a single singleton for the whole app, available everywhere, and tree-shakeable. The default choice for the vast majority of services.

'platform' — registers in the platform injector: one instance shared across multiple Angular apps loaded on the same page. Rare; used in micro-frontends sharing state.

'any' — creates a separate instance for each injector that requests it — in practice one per lazily loaded module plus one in the root. Use it when each lazy feature needs its own isolated copy.

MyModule — registers in a specific module's injector: the service exists only where that module is imported. It is bundled with and removed alongside the module.

// App-wide singleton (default, recommended)
@Injectable({ providedIn: 'root' })
export class PortfolioService {}

// Shared across multiple Angular apps on the same page
@Injectable({ providedIn: 'platform' })
export class SharedSessionService {}

// One instance per lazy-loaded module (isolated state)
@Injectable({ providedIn: 'any' })
export class FeatureStateService {}

// Only available where TradingModule is imported
@Injectable({ providedIn: TradingModule })
export class TradingConfigService {}
ValueInstancesTree-shakingTypical use

'platform'One per page✅Micro-frontends
'any'One per (lazy) injector✅Isolated lazy state
MyModuleOne per module import✅Module-scoped configuration

The 'any' gotcha: since each lazy module gets its own instance, it is not a singleton. To share a single source of truth across lazy features, use 'root'.

Provider types

A provider tells the injector how to build a value for a token. There are four recipes, plus the multi flag for registering many providers under a single token.

useClass

useClass — instantiates the given class (with its own dependencies injected) when the token is requested. Lets you swap one implementation for another behind the same token.

// Provide a mock feed in tests, a real feed in prod — same token
providers: [
  { provide: PriceFeedService, useClass: MockPriceFeedService },
];

useValue

useValue — supplies an exact, already-constructed value as-is (no instantiation). Ideal for configuration objects, constants, and primitives.

interface TradingConfig { maxOrderQty: number; baseCurrency: string; }

providers: [
  { provide: TRADING_CONFIG, useValue: { maxOrderQty: 1_000_000, baseCurrency: 'USD' } as TradingConfig },
];

useFactory

useFactory — calls a factory function to compute the value. The function can take its own dependencies, declared in order in the deps array.

// Build a service that depends on runtime config + another service
export function orderValidatorFactory(config: TradingConfig, prices: PriceService): OrderValidator {
  return new OrderValidator(config.maxOrderQty, prices);
}

providers: [
  {
    provide: OrderValidator,
    useFactory: orderValidatorFactory,
    deps: [TRADING_CONFIG, PriceService],   // injected in this order
  },
];

Modern alternative: inside the factory you can call inject() directly instead of listing deps, because factories run in an injection context.

useExisting

useExisting — aliases one token to another: requesting the new token returns the same instance already registered for the target token. No new object is created.

// Expose the same PriceService under a narrower read-only interface
providers: [
  PriceService,
  { provide: ReadonlyQuoteSource, useExisting: PriceService },
];
// inject(ReadonlyQuoteSource) === inject(PriceService) → same object

useExisting vs useClass: useClass creates a brand-new instance; useExisting reuses an existing one. Aliasing by mistake via useClass gives you two separate objects.

multi: true

multi: true — lets multiple providers register under one token; injecting the token returns an array of all the contributed values. The foundation of HTTP interceptors, validators, and initializers.

// Each provider adds one interceptor; inject(HTTP_INTERCEPTORS) → all of them
providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: RetryInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
];

Pitfall: mixing multi: true and non-multi providers for the same token throws an error. A token is either always multi or never multi.

ProviderProducesNew instance?Best for
useClassClass instance✅Swappable implementations
useValueLiteral value❌ (used as-is)Configuration, constants
useFactoryWhatever the factory returnsDependsDependencies computed at runtime

InjectionToken

InjectionToken — a typed, unique token used to inject non-class dependencies (configuration objects, strings, functions, primitives). Because each token is a distinct object, it avoids the key collisions inherent to plain string tokens, and it carries a generic type for type-safe injection.

// 1. Declare a typed token
export const API_URL = new InjectionToken<string>('API_URL');

// 2. Register a value for it
providers: [
  { provide: API_URL, useValue: 'https://api.trading.com' },
];

// 3a. Inject it the modern way (field initializer)
@Injectable({ providedIn: 'root' })
export class OrderApi {
  private readonly apiUrl = inject(API_URL);   // typed as string
}

// 3b. Inject it the classic way (constructor + @Inject)
@Injectable({ providedIn: 'root' })
export class OrderApiClassic {
  constructor(@Inject(API_URL) private apiUrl: string) {}
}

Token with a factory default value

An InjectionToken can carry its own tree-shakeable default value via providedIn + factory, so you don't need to register it explicitly unless you want to override it.

export interface FeatureFlags {
  darkMode: boolean;
  betaFeatures: boolean;
}

export const FEATURE_FLAGS = new InjectionToken<FeatureFlags>('FeatureFlags', {
  providedIn: 'root',
  factory: () => ({ darkMode: false, betaFeatures: false }),
});

// Works with no explicit provider — the factory supplies the default
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private readonly flags = inject(FEATURE_FLAGS); // typed FeatureFlags
}

Why not string tokens? Plain strings are global and untyped, so two libraries that both use 'config' as a token will silently conflict — whoever registers last wins. An InjectionToken is a unique object reference that cannot collide, and it carries its type.

// ❌ Fragile — collides with any other 'config' string token, no type safety
providers: [{ provide: 'config', useValue: { /* ... */ } }];

// ✅ Unique object + generic type, collision-proof
export const CONFIG = new InjectionToken<TradingConfig>('CONFIG');

DI decorators (@Self, @SkipSelf, @Optional, @Host)

These modifier decorators (and their inject() option equivalents) change how far and where Angular looks when resolving a dependency.

@Self()

@Self() — restricts resolution to the current injector only. If there is no provider there, Angular throws instead of walking up.

When to use: you need a locally provided instance and want a hard failure if it's missing.

@Component({
  selector: 'app-order-row',
  standalone: true,
  providers: [OrderDraftService],            // must be provided HERE
})
export class OrderRowComponent {
  // constructor form
  constructor(@Self() private draft: OrderDraftService) {}
  // inject() form
  // private readonly draft = inject(OrderDraftService, { self: true });
}

@SkipSelf()

@SkipSelf() — skips the current injector and starts resolution at the parent. Useful for reaching a parent's instance even when the same token is provided locally.

When to use: hierarchical services where a child must reach its parent's copy (for example, nested form groups, tree structures).

@Component({ selector: 'app-positions-node', standalone: true })
export class PositionsNodeComponent {
  // Reach the PARENT node's aggregator, not this node's own
  constructor(@SkipSelf() @Optional() private parent: PositionAggregator | null) {}
  // inject() form
  // private readonly parent = inject(PositionAggregator, { skipSelf: true, optional: true });
}

@Optional()

@Optional() — returns null instead of throwing when no provider is found anywhere up the tree.

When to use: optional collaborators, feature toggles, or tokens that may legitimately be absent.

@Injectable({ providedIn: 'root' })
export class TradingService {
  // No AnalyticsService registered? logger stays null, no crash.
  constructor(@Optional() private analytics: AnalyticsService | null) {}

  placeOrder(order: Order): void {
    this.analytics?.track('order_placed', order.id);
  }
}

@Host()

@Host() — stops walking up at the host component's injector: it looks in the current and host element injectors, but no higher. Mainly for directives that need a dependency provided by their host component.

When to use: a directive that should bind to a locally provided service of its host component (for example, a control bound to its host form).

@Directive({ selector: '[appQtyControl]', standalone: true })
export class QtyControlDirective {
  // Find the validator on the host component, not somewhere up the app tree
  constructor(@Host() private validator: OrderValidator) {}
  // inject() form
  // private readonly validator = inject(OrderValidator, { host: true });
}
DecoratorSearch scopeNot found →
@Self()Current injector onlyThrows

They combine: @SkipSelf() @Optional() means "start at the parent and give me null if no one above has it". The inject() equivalents are the options { skipSelf: true, optional: true }.

The inject() function vs constructor injection

inject() — a function (Angular 14+) that retrieves a dependency from the current injector. It works in constructors, field initializers, and factory functions, and it is especially clean for standalone components, functional guards/interceptors, and composition helper functions.

Constructor injection — the classic approach: dependencies are declared as constructor parameters. It works everywhere and is required when an injected dependency must be available to a subclass constructor via super(...).

Injection context — the limited window in which inject() is allowed to run: during construction (the constructor body or field initializers), inside provider factories, or explicitly via runInInjectionContext(). Calling inject() outside it throws NG0203.

// Modern: inject() in field initializers — no constructor boilerplate
@Injectable({ providedIn: 'root' })
export class OrderFacade {
  private readonly http = inject(HttpClient);
  private readonly prices = inject(PriceService);
  private readonly apiUrl = inject(API_URL);

  submit(order: Order) {
    return this.http.post<Order>(`${this.apiUrl}/orders`, order);
  }
}

// Classic: constructor injection — required when subclasses call super()
@Injectable()
export class BaseFeedService {
  constructor(protected http: HttpClient) {}
}
@Injectable({ providedIn: 'root' })
export class QuoteFeedService extends BaseFeedService {
  constructor(http: HttpClient) { super(http); }   // must forward via super
}

Respecting the injection context

// ✅ inject() also powers functional guards / interceptors (Angular 15+)
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);          // injection context here
  return auth.isLoggedIn();
};

// ❌ Outside an injection context → NG0203
function buildHelper() {
  const prices = inject(PriceService);       // throws if called lazily later
}

// ✅ Wrap deferred work in runInInjectionContext
@Injectable({ providedIn: 'root' })
export class LazyLoader {
  private readonly injector = inject(EnvironmentInjector);
  loadLater() {
    runInInjectionContext(this.injector, () => {
      const prices = inject(PriceService);   // now valid
    });
  }
}

Q&A — Interview Questions

1. How does Angular resolve a dependency, and what happens if a provider doesn't exist?

Angular starts at the requesting class's injector and walks up the injector tree — element/component → module → root → platform — returning the first matching provider it finds and caching the instance on that injector. If it reaches the top without a match, it throws NullInjectorError: No provider for X. Use @Optional() (or inject(X, { optional: true })) to get null instead of an error.

2. What is the difference between providedIn: 'root' and providedIn: 'any'?

'root' creates a single app-wide singleton in the root injector — everyone shares one instance. 'any' creates a separate instance for each injector that requests it, which in practice means one instance per lazily loaded module (plus one in the root). Both are tree-shakeable. Use 'root' for shared state; use 'any' when each lazy feature needs an isolated copy.

3. When would you reach for useFactory instead of useClass or useValue?

When the value must be computed at runtime and may depend on other injectables or configuration. useValue needs a value you already have; useClass simply instantiates a class. useFactory runs a function — listing its dependencies in deps (or calling inject() inside) — so you can branch on configuration, combine services, or build an object conditionally. Example: choosing a real or mock price feed based on an environment flag.

4. Why use an InjectionToken instead of a string token?

String tokens are global and untyped: two libraries that both use 'config' will collide, and the last registration silently wins. An InjectionToken is a unique object reference (collision-proof) and carries a generic type, so inject(API_URL) is typed as string, not any. Tokens can also supply a tree-shakeable default value via { providedIn: 'root', factory }.

5. What is the "injection context" and when does inject() fail?

The injection context is the window in which Angular can resolve dependencies: during a class's construction (the constructor body and field initializers), inside provider factory functions, and inside functional guards/interceptors. Calling inject() outside it — for example, in a setTimeout callback, an event handler, or a lazily invoked plain function — throws NG0203. To run inject() later, capture an EnvironmentInjector and wrap the work in runInInjectionContext().

Quiz

1. Which providedIn value gives a separate instance per lazily loaded module?
'any' creates one instance per requesting injector — effectively one per lazy module (plus the root). 'root' is a single app-wide singleton; 'platform' is shared across apps on the page; 'module' is not a valid value.
2. Which provider type aliases a token to an existing instance without creating a new one?
useExisting aliases one token to another, returning the same instance. useClass would create a separate object, breaking the "same instance" guarantee.
3. What does @SkipSelf() do during resolution?
@SkipSelf() skips the local injector and resolves the dependency starting from the parent and up. @Self() looks locally only; @Optional() returns null; @Host() stops at the host element.
4. Where can you safely call inject()?
inject() requires an injection context: constructor/field initializers, provider factories, functional guards/interceptors, or code wrapped in runInInjectionContext(). Calling it from a deferred callback throws NG0203.

Как работает DI

Angular строит дерево инжекторов, которое повторяет дерево компонентов. Когда класс запрашивает зависимость, Angular поднимается вверх по этому дереву от запрашивающего инжектора, пока не найдёт ближайший провайдер для этого токена. Побеждает первый найденный провайдер; если ни одного нет, Angular выбрасывает NullInjectorError: No provider for X.

Инжектор (Injector) — контейнер, который умеет создавать и кэшировать зависимости для заданного набора токенов. У каждого инжектора есть родитель, что образует дерево.

Токен (Token) — ключ, который Angular использует для поиска провайдера. Обычно это тип класса, но может быть InjectionToken или строкой.

Провайдер (Provider) — рецепт, который сообщает инжектору, как произвести значение для токена (useClass, useValue, useFactory, useExisting).

Разрешение (Resolution) — подъём вверх по родительским инжекторам в поисках первого подходящего провайдера. После разрешения экземпляр кэшируется на этом инжекторе и переиспользуется всеми, кто находится ниже него.

Иерархия инжекторов (сверху вниз)

1. Платформенный инжектор (Platform Injector) — самый верхний инжектор, создаваемый platformBrowserDynamic()/bootstrapApplication. Хранит синглтоны уровня платформы, общие для всех Angular-приложений, запущенных на одной странице (например, нескольких приложений в одной вкладке браузера).

2. Корневой инжектор (Root Injector) — создаётся при загрузке приложения. Всё, что объявлено как providedIn: 'root', живёт здесь единственным экземпляром на всё приложение. Дом по умолчанию для сервисов уровня приложения.

3. Инжектор модуля (Module Injector) — создаётся для каждого (загруженного сразу или лениво) @NgModule. Сервисы, перечисленные в @NgModule.providers, живут здесь. Лениво загружаемые модули получают собственный дочерний инжектор модуля.

4. Инжектор компонента (Element Injector) — создаётся для каждого компонента, объявляющего @Component.providers. Производит новый экземпляр на каждое поддерево компонента, поэтому каждый компонент (и его потомки) получает собственную копию.

5. Инжектор элемента (директивы) — самый мелкозернистый инжектор, создаваемый для элементов, на которых размещена @Directive с providers. Используется для зависимостей в области видимости директивы.

Ментальная модель: разрешение всегда идёт вверх, никогда вбок или вниз. Соседний компонент не может увидеть сервис, который вы предоставили в другом компоненте — предоставьте его выше (в корне или общем родителе), чтобы поделиться им.

Опции providedIn

Поле providedIn декоратора @Injectable объявляет, где регистрируется сервис, и делает его поддающимся tree-shaking: если сервис никто не инжектит, сборщик может полностью его удалить.

'root' — регистрирует в корневом инжекторе: единственный синглтон на всё приложение, доступный везде, и поддающийся tree-shaking. Выбор по умолчанию для подавляющего большинства сервисов.

'platform' — регистрирует в платформенном инжекторе: один экземпляр, общий для нескольких Angular-приложений, загруженных на одной странице. Редко; используется в микрофронтендах, разделяющих состояние.

'any' — создаёт отдельный экземпляр для каждого инжектора, который его запрашивает — на практике по одному на каждый лениво загружаемый модуль плюс один в корне. Используйте, когда каждой ленивой фиче нужна своя изолированная копия.

MyModule — регистрирует в инжекторе конкретного модуля: сервис существует только там, где этот модуль импортирован. Включается в бандл и удаляется вместе с модулем.

// App-wide singleton (default, recommended)
@Injectable({ providedIn: 'root' })
export class PortfolioService {}

// Shared across multiple Angular apps on the same page
@Injectable({ providedIn: 'platform' })
export class SharedSessionService {}

// One instance per lazy-loaded module (isolated state)
@Injectable({ providedIn: 'any' })
export class FeatureStateService {}

// Only available where TradingModule is imported
@Injectable({ providedIn: TradingModule })
export class TradingConfigService {}

Подвох с 'any': поскольку каждый ленивый модуль получает свой экземпляр, это не синглтон. Чтобы разделить единый источник истины между ленивыми фичами, нужен 'root'.

Типы провайдеров

Провайдер сообщает инжектору, как построить значение для токена. Есть четыре рецепта, плюс флаг multi для регистрации множества провайдеров под одним токеном.

useClass

useClass — создаёт экземпляр заданного класса (с инжектированными в него собственными зависимостями), когда запрашивается токен. Позволяет подменять одну реализацию другой за одним и тем же токеном.

// Provide a mock feed in tests, a real feed in prod — same token
providers: [
  { provide: PriceFeedService, useClass: MockPriceFeedService },
];

useValue

useValue — поставляет точное, уже сконструированное значение как есть (без создания экземпляра). Идеально для объектов конфигурации, констант и примитивов.

interface TradingConfig { maxOrderQty: number; baseCurrency: string; }

providers: [
  { provide: TRADING_CONFIG, useValue: { maxOrderQty: 1_000_000, baseCurrency: 'USD' } as TradingConfig },
];

useFactory

useFactory — вызывает фабричную функцию для вычисления значения. Функция может принимать собственные зависимости, объявленные по порядку в массиве deps.

// Build a service that depends on runtime config + another service
export function orderValidatorFactory(config: TradingConfig, prices: PriceService): OrderValidator {
  return new OrderValidator(config.maxOrderQty, prices);
}

providers: [
  {
    provide: OrderValidator,
    useFactory: orderValidatorFactory,
    deps: [TRADING_CONFIG, PriceService],   // injected in this order
  },
];

Современная альтернатива: внутри фабрики можно вызывать inject() напрямую вместо перечисления deps, потому что фабрики выполняются в контексте инъекции.

useExisting

useExisting — делает один токен псевдонимом другого: запрос нового токена возвращает тот же экземпляр, уже зарегистрированный для целевого токена. Новый объект не создаётся.

// Expose the same PriceService under a narrower read-only interface
providers: [
  PriceService,
  { provide: ReadonlyQuoteSource, useExisting: PriceService },
];
// inject(ReadonlyQuoteSource) === inject(PriceService) → same object

useExisting против useClass: useClass создаёт совершенно новый экземпляр; useExisting переиспользует существующий. Ошибочное создание псевдонима через useClass даст вам два отдельных объекта.

multi: true

multi: true — позволяет нескольким провайдерам регистрироваться под одним токеном; инъекция токена возвращает массив всех привнесённых значений. Основа HTTP-интерцепторов, валидаторов и инициализаторов.

// Each provider adds one interceptor; inject(HTTP_INTERCEPTORS) → all of them
providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: RetryInterceptor, multi: true },
  { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
];

Подводный камень: смешивание multi: true и не-multi провайдеров для одного токена приводит к ошибке. Токен либо всегда multi, либо никогда не multi.

InjectionToken

InjectionToken — типизированный уникальный токен, используемый для инъекции не-классовых зависимостей (объектов конфигурации, строк, функций, примитивов). Поскольку каждый токен — это отдельный объект, он избегает коллизий ключей, свойственных обычным строковым токенам, и несёт generic-тип для типобезопасной инъекции.

// 1. Declare a typed token
export const API_URL = new InjectionToken<string>('API_URL');

// 2. Register a value for it
providers: [
  { provide: API_URL, useValue: 'https://api.trading.com' },
];

// 3a. Inject it the modern way (field initializer)
@Injectable({ providedIn: 'root' })
export class OrderApi {
  private readonly apiUrl = inject(API_URL);   // typed as string
}

// 3b. Inject it the classic way (constructor + @Inject)
@Injectable({ providedIn: 'root' })
export class OrderApiClassic {
  constructor(@Inject(API_URL) private apiUrl: string) {}
}

Токен с фабричным значением по умолчанию

Токен InjectionToken может нести собственное tree-shakeable значение по умолчанию через providedIn + factory, поэтому вам не нужно регистрировать его явно, если только вы не хотите его переопределить.

export interface FeatureFlags {
  darkMode: boolean;
  betaFeatures: boolean;
}

export const FEATURE_FLAGS = new InjectionToken<FeatureFlags>('FeatureFlags', {
  providedIn: 'root',
  factory: () => ({ darkMode: false, betaFeatures: false }),
});

// Works with no explicit provider — the factory supplies the default
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private readonly flags = inject(FEATURE_FLAGS); // typed FeatureFlags
}

Почему не строковые токены? Обычные строки глобальны и нетипизированы, поэтому две библиотеки, которые обе используют 'config' в качестве токена, будут молча конфликтовать — побеждает тот, кто зарегистрировался последним. Токен InjectionToken — это уникальная ссылка на объект, с которой невозможно столкнуться, и она несёт свой тип.

// ❌ Fragile — collides with any other 'config' string token, no type safety
providers: [{ provide: 'config', useValue: { /* ... */ } }];

// ✅ Unique object + generic type, collision-proof
export const CONFIG = new InjectionToken<TradingConfig>('CONFIG');

Декораторы DI (@Self, @SkipSelf, @Optional, @Host)

Эти декораторы-модификаторы (и их эквиваленты в виде опций inject()) меняют, насколько далеко и где Angular ищет при разрешении зависимости.

@Self()

@Self() — ограничивает разрешение только текущим инжектором. Если там нет провайдера, Angular выбрасывает ошибку вместо подъёма вверх.

Когда использовать: вам нужен локально предоставленный экземпляр и вы хотите жёсткий сбой, если он отсутствует.

@Component({
  selector: 'app-order-row',
  standalone: true,
  providers: [OrderDraftService],            // must be provided HERE
})
export class OrderRowComponent {
  // constructor form
  constructor(@Self() private draft: OrderDraftService) {}
  // inject() form
  // private readonly draft = inject(OrderDraftService, { self: true });
}

@SkipSelf()

@SkipSelf() — пропускает текущий инжектор и начинает разрешение с родительского. Полезно, чтобы добраться до экземпляра родителя, даже когда тот же токен предоставлен локально.

Когда использовать: иерархические сервисы, где потомок должен обращаться к копии родителя (например, вложенные группы форм, древовидные структуры).

@Component({ selector: 'app-positions-node', standalone: true })
export class PositionsNodeComponent {
  // Reach the PARENT node's aggregator, not this node's own
  constructor(@SkipSelf() @Optional() private parent: PositionAggregator | null) {}
  // inject() form
  // private readonly parent = inject(PositionAggregator, { skipSelf: true, optional: true });
}

@Optional()

@Optional() — возвращает null вместо ошибки, когда провайдер не найден нигде выше по дереву.

Когда использовать: необязательные коллабораторы, переключатели фич или токены, которые могут законно отсутствовать.

@Injectable({ providedIn: 'root' })
export class TradingService {
  // No AnalyticsService registered? logger stays null, no crash.
  constructor(@Optional() private analytics: AnalyticsService | null) {}

  placeOrder(order: Order): void {
    this.analytics?.track('order_placed', order.id);
  }
}

@Host()

@Host() — останавливает подъём вверх на инжекторе хост-компонента: он смотрит в текущем и хостовом инжекторах элемента, но не выше. В основном для директив, которым нужна зависимость, предоставленная их хост-компонентом.

Когда использовать: директива, которая должна привязаться к локально предоставленному сервису своего хост-компонента (например, контрол, привязанный к своей хост-форме).

@Directive({ selector: '[appQtyControl]', standalone: true })
export class QtyControlDirective {
  // Find the validator on the host component, not somewhere up the app tree
  constructor(@Host() private validator: OrderValidator) {}
  // inject() form
  // private readonly validator = inject(OrderValidator, { host: true });
}
ДекораторОбласть поискаНе найдено →

Они комбинируются: @SkipSelf() @Optional() означает «начни с родителя и дай мне null, если ни у кого выше этого нет». Эквиваленты для inject() — это опции { skipSelf: true, optional: true }.

Функция inject() против инъекции через конструктор

inject() — функция (Angular 14+), которая извлекает зависимость из текущего инжектора. Применима в конструкторах, инициализаторах полей и фабричных функциях, и особенно чиста для standalone-компонентов, функциональных guard'ов/интерцепторов и вспомогательных функций композиции.

Инъекция через конструктор — классический подход: зависимости объявляются как параметры конструктора. Работает везде и обязательна, когда инжектируемая зависимость должна быть доступна конструктору подкласса через super(...).

Контекст инъекции — ограниченное окно, в котором inject() разрешено выполняться: во время конструирования (тело конструктора или инициализаторы полей), внутри фабрик провайдеров или явно через runInInjectionContext(). Вызов inject() вне его выбрасывает NG0203.

// Modern: inject() in field initializers — no constructor boilerplate
@Injectable({ providedIn: 'root' })
export class OrderFacade {
  private readonly http = inject(HttpClient);
  private readonly prices = inject(PriceService);
  private readonly apiUrl = inject(API_URL);

  submit(order: Order) {
    return this.http.post<Order>(`${this.apiUrl}/orders`, order);
  }
}

// Classic: constructor injection — required when subclasses call super()
@Injectable()
export class BaseFeedService {
  constructor(protected http: HttpClient) {}
}
@Injectable({ providedIn: 'root' })
export class QuoteFeedService extends BaseFeedService {
  constructor(http: HttpClient) { super(http); }   // must forward via super
}

Соблюдение контекста инъекции

// ✅ inject() also powers functional guards / interceptors (Angular 15+)
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);          // injection context here
  return auth.isLoggedIn();
};

// ❌ Outside an injection context → NG0203
function buildHelper() {
  const prices = inject(PriceService);       // throws if called lazily later
}

// ✅ Wrap deferred work in runInInjectionContext
@Injectable({ providedIn: 'root' })
export class LazyLoader {
  private readonly injector = inject(EnvironmentInjector);
  loadLater() {
    runInInjectionContext(this.injector, () => {
      const prices = inject(PriceService);   // now valid
    });
  }
}

Рекомендация: предпочитайте inject() для нового standalone-кода — он лаконичнее, работает в функциональных API и оставляет конструкторы пустыми. Сохраняйте инъекцию через конструктор там, где вы расширяете базовый класс и должны передавать зависимости через super().

Вопросы и ответы — Вопросы на собеседовании

1. Как Angular разрешает зависимость и что происходит, если провайдер не существует?

Angular начинает с инжектора запрашивающего класса и поднимается вверх по дереву инжекторов — элемент/компонент → модуль → корень → платформа — возвращая первый подходящий провайдер, который находит, и кэшируя экземпляр на этом инжекторе. Если он достигает вершины без совпадения, выбрасывается NullInjectorError: No provider for X. Используйте @Optional() (или inject(X, { optional: true })), чтобы получить null вместо ошибки.

2. В чём разница между providedIn: 'root' и providedIn: 'any'?

'root' создаёт единственный синглтон на всё приложение в корневом инжекторе — все разделяют один экземпляр. 'any' создаёт отдельный экземпляр для каждого инжектора, который его запрашивает, что на практике означает один экземпляр на каждый лениво загружаемый модуль (плюс один в корне). Оба поддаются tree-shaking. Используйте 'root' для общего состояния; используйте 'any', когда каждой ленивой фиче нужна изолированная копия.

3. Когда вы бы прибегли к useFactory вместо useClass или useValue?

Когда значение должно быть вычислено в рантайме и может зависеть от других инжектируемых сущностей или конфигурации. useValue нужно значение, которое у вас уже есть; useClass просто создаёт экземпляр класса. useFactory выполняет функцию — перечисляя её зависимости в deps (или вызывая inject() внутри) — так что вы можете ветвиться по конфигурации, объединять сервисы или строить объект условно. Пример: выбор реального или мокового источника цен на основе флага окружения.

4. Зачем использовать InjectionToken вместо строкового токена?

Строковые токены глобальны и нетипизированы: две библиотеки, обе использующие 'config', столкнутся, и последняя регистрация молча победит. InjectionToken — это уникальная ссылка на объект (защищённая от коллизий) и несёт generic-тип, поэтому inject(API_URL) типизирован как string, а не any. Токены также могут поставлять tree-shakeable значение по умолчанию через { providedIn: 'root', factory }.

5. Что такое «контекст инъекции» и когда inject() завершается с ошибкой?

Контекст инъекции — это окно, в котором Angular может разрешать зависимости: во время конструирования класса (тело конструктора и инициализаторы полей), внутри фабричных функций провайдеров и внутри функциональных guard'ов/интерцепторов. Вызов inject() вне его — например, в колбэке setTimeout, обработчике событий или лениво вызываемой обычной функции — выбрасывает NG0203. Чтобы выполнить inject() позже, захватите EnvironmentInjector и оберните работу в runInInjectionContext().

Тест

1. Какое значение providedIn даёт отдельный экземпляр на каждый лениво загружаемый модуль?
'any' создаёт один экземпляр на каждый запрашивающий инжектор — фактически один на каждый ленивый модуль (плюс корень). 'root' — это единственный синглтон на всё приложение; 'platform' разделяется между приложениями на странице; 'module' не является допустимым значением.
2. Какой тип провайдера делает токен псевдонимом существующего экземпляра, не создавая нового?
useExisting делает один токен псевдонимом другого, возвращая тот же экземпляр. useClass создал бы отдельный объект, нарушив гарантию «того же экземпляра».
3. Что делает @SkipSelf() во время разрешения?
@SkipSelf() пропускает локальный инжектор и разрешает зависимость начиная с родительского и выше. @Self() ищет только локально; @Optional() возвращает null; @Host() останавливается на хост-элементе.
4. Где можно безопасно вызывать inject()?
inject() требует контекста инъекции: конструктор/инициализаторы полей, фабрики провайдеров, функциональные guard'ы/интерцепторы или код, обёрнутый в runInInjectionContext(). Вызов из отложенного колбэка выбрасывает NG0203.
