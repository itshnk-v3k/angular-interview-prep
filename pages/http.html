HTTP и интерцепторы

Сервис HttpClient в Angular, все методы и опции запросов, а также конвейер интерцепторов, обеспечивающий авторизацию, обработку ошибок, состояние загрузки и логирование — сквозные задачи, которые каждый senior обязан уметь корректно настроить.

HTTP-методы

HTTP-глагол сообщает намерение серверу и всем кешам или прокси на пути запроса. Два свойства определяют проектные решения: идемпотентность и безопасность.

Идемпотентность — отправка одного и того же запроса несколько раз приводит к тому же состоянию сервера, что и однократная отправка. Повторённый после сбоя PUT безопасен; повторённый POST может создать дублирующиеся заказы.

Безопасность — запрос вообще не изменяет состояние сервера (только чтение). Безопасные методы по своей сути идемпотентны и свободно кешируются.

МетодОпределениеЕсть телоИдемпотентныйБезопасный
GETПолучить ресурс❌✅✅
POSTСоздать ресурс✅❌❌
PUTПолностью заменить ресурс✅✅❌
PATCHЧастично обновить ресурс✅❌❌
DELETEУдалить ресурсопционально✅❌
HEADКак GET, но без тела (только заголовки)❌✅✅
OPTIONSУзнать допустимые методы (CORS preflight)❌✅✅

Почему идемпотентность важна для повторов: HTTP-интерцептор, автоматически повторяющий неудавшиеся запросы, безопасен только для идемпотентных глаголов. Повтор POST /orders может разместить одну и ту же сделку дважды — защитите его ключом идемпотентности.

export interface Order {

HttpClient

HttpClient — внедряемый сервис Angular для выполнения HTTP-запросов. Каждый метод возвращает холодный Observable — ничего не происходит, пока вы не вызовете subscribe (или не используете пайп async / firstValueFrom). Параметр-дженерик задаёт тип распарсенного тела JSON-ответа.

Подключите его один раз через provideHttpClient() в app.config.ts (standalone) или импортируйте HttpClientModule (NgModule).

get<T>(url, options?)

get — выполняет GET-запрос и эмитит тело ответа с типом T. Безопасен и идемпотентен; используется для чтения.

getOrder(id: string): Observable<Order> {
  return this.http.get<Order>(`/api/orders/${id}`);
}

listInstruments(): Observable<Instrument[]> {
  return this.http.get<Instrument[]>('/api/instruments');
}

post<T>(url, body, options?)

post — отправляет тело для создания ресурса и эмитит созданную сущность. Не идемпотентен — каждый вызов создаёт новый заказ.

placeOrder(draft: Omit<Order, 'id' | 'status'>): Observable<Order> {
  return this.http.post<Order>('/api/orders', draft);
}

put<T>(url, body, options?)

put — заменяет ресурс целиком переданным телом. Идемпотентен — повторная отправка той же нагрузки приводит к тому же состоянию.

replaceOrder(id: string, order: Order): Observable<Order> {
  return this.http.put<Order>(`/api/orders/${id}`, order);
}

patch<T>(url, body, options?)

patch — применяет частичное обновление; вы отправляете только изменяемые поля. Идемпотентность не гарантируется.

updatePrice(id: string, price: number): Observable<Order> {
  return this.http.patch<Order>(`/api/orders/${id}`, { price });
}

delete<T>(url, options?)

delete — удаляет ресурс. Идемпотентен — удаление уже удалённого заказа оставляет то же конечное состояние.

cancelOrder(id: string): Observable<void> {
  return this.http.delete<void>(`/api/orders/${id}`);
}

HttpParams — строки запроса

HttpParams — иммутабельный построитель параметров URL-запроса. Каждый set/append возвращает новый экземпляр, поэтому нужно строить цепочку или переприсваивать. Кодирование Angular берёт на себя.

searchOrders(symbol: string, status: OrderStatus): Observable<Order[]> {
  const params = new HttpParams()
    .set('symbol', symbol)
    .set('status', status)
    .set('limit', 50);            // numbers/booleans accepted

  // GET /api/orders?symbol=EURUSD&status=FILLED&limit=50
  return this.http.get<Order[]>('/api/orders', { params });
}

Ловушка иммутабельности: params.set('a', '1') сам по себе ничего не делает — он возвращает новый HttpParams, который нужно сохранить.

HttpHeaders — заголовки запроса

HttpHeaders — иммутабельная коллекция заголовков запроса с тем же правилом «строй цепочку или переприсваивай», что и HttpParams.

submitOrder(draft: Omit<Order, 'id' | 'status'>): Observable<Order> {
  const headers = new HttpHeaders()
    .set('Content-Type', 'application/json')
    .set('X-Idempotency-Key', crypto.randomUUID());

  return this.http.post<Order>('/api/orders', draft, { headers });
}

observe: 'response' — полный ответ

observe: 'response' — эмитит весь HttpResponse<T> вместо одного только тела, давая доступ к status, headers и statusText.

getOrderWithMeta(id: string): Observable<Order | null> {
  return this.http
    .get<Order>(`/api/orders/${id}`, { observe: 'response' })
    .pipe(
      map((res: HttpResponse<Order>) => {
        console.log('status', res.status);              // 200
        console.log('etag', res.headers.get('ETag'));   // read a header
        return res.body;                                // Order | null
      }),
    );
}

observe: 'events' — прогресс и все события

observe: 'events' — эмитит каждое HttpEvent<T> по мере его возникновения: событие отправки, прогресс загрузки/скачивания и финальный ответ. В сочетании с reportProgress: true это обеспечивает работу полос прогресса загрузки.

uploadOrders(file: File): Observable<number> {
  const form = new FormData();
  form.append('file', file);

  return this.http
    .post<Order[]>('/api/orders/bulk', form, {
      observe: 'events',
      reportProgress: true,
    })
    .pipe(
      filter((e: HttpEvent<Order[]>): e is HttpProgressEvent =>
        e.type === HttpEventType.UploadProgress),
      map((e: HttpProgressEvent) =>
        e.total ? Math.round((100 * e.loaded) / e.total) : 0),
    );
}

responseType: 'blob' — скачивание файлов

responseType: 'blob' — указывает HttpClient сохранить сырой бинарный Blob вместо парсинга JSON. Используется для PDF, CSV-экспорта и изображений.

downloadStatement(id: string): Observable<Blob> {
  return this.http.get(`/api/statements/${id}`, {
    responseType: 'blob',           // no <T> — body is a Blob
  });
}

saveStatement(id: string): void {
  this.downloadStatement(id).subscribe((blob: Blob) => {
    const url = URL.createObjectURL(blob);
    const a = document.createElement('a');
    a.href = url;
    a.download = `statement-${id}.pdf`;
    a.click();
    URL.revokeObjectURL(url);
  });
}

Интерцепторы

Интерцептор — класс или функция, находящаяся в конвейере HttpClient и перехватывающая каждый исходящий HttpRequest и входящий ответ, что позволяет их инспектировать, изменять, повторять, прерывать или логировать до того, как они достигнут кода приложения. Это каноническое место для сквозных задач: токены авторизации, обработка ошибок, состояние загрузки, логирование и кеширование.

Зачем req.clone()?

Иммутабельность HttpRequest — HttpRequest иммутабелен: его свойства нельзя переприсвоить. Чтобы что-то изменить (заголовки, URL, тело, параметры), вы вызываете req.clone({ ... }), который возвращает новый запрос с применёнными переопределениями. Это гарантирует, что последующие интерцепторы и повторы всегда видят согласованный, нетронутый оригинал.

// ❌ Throws / has no effect — request is frozen
req.headers = req.headers.set('Authorization', token);

// ✅ Produce a modified copy
const authReq = req.clone({
  setHeaders: { Authorization: `Bearer ${token}` },
});

Интерцептор авторизации (на основе класса)

Прикрепляет bearer-токен к каждому исходящему запросу, клонируя его с дополнительным заголовком. Обратите внимание на дженерик <T>, пробрасываемый через intercept, чтобы тип тела сохранялся от начала и до конца.

@Injectable()
export class AuthInterceptor implements HttpInterceptor {
  constructor(private readonly auth: AuthService) {}

  intercept<T>(
    req: HttpRequest<T>,
    next: HttpHandler,
  ): Observable<HttpEvent<T>> {
    const token = this.auth.token();
    if (!token) {
      return next.handle(req);            // nothing to add
    }

    const authReq = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` },
    });
    return next.handle(authReq);          // always return the pipeline
  }
}

Интерцептор ошибок (401 → редирект)

Перехватывает ошибки из потока ответа с помощью catchError, перенаправляет на вход при 401 Unauthorized и пробрасывает ошибку дальше, чтобы обработка ошибок на уровне компонента всё равно сработала.

@Injectable()
export class ErrorInterceptor implements HttpInterceptor {
  constructor(private readonly router: Router) {}

  intercept<T>(
    req: HttpRequest<T>,
    next: HttpHandler,
  ): Observable<HttpEvent<T>> {
    return next.handle(req).pipe(
      catchError((err: HttpErrorResponse) => {
        if (err.status === 401) {
          this.router.navigate(['/login']);
        }
        return throwError(() => err);     // re-throw for callers
      }),
    );
  }
}

Интерцептор загрузки (глобальный спиннер)

Увеличивает счётчик на каждом исходящем запросе и уменьшает его в finalize (который срабатывает и при успехе, и при ошибке), управляя глобальным индикатором загрузки через сервис.

@Injectable({ providedIn: 'root' })
export class LoadingService {
  private count = 0;
  readonly loading = signal(false);

  show(): void { this.count++; this.loading.set(true); }
  hide(): void { this.count = Math.max(0, this.count - 1); this.loading.set(this.count > 0); }
}

@Injectable()
export class LoadingInterceptor implements HttpInterceptor {
  constructor(private readonly loader: LoadingService) {}

  intercept<T>(
    req: HttpRequest<T>,
    next: HttpHandler,
  ): Observable<HttpEvent<T>> {
    this.loader.show();
    return next.handle(req).pipe(
      finalize(() => this.loader.hide()), // runs on complete OR error
    );
  }
}

Интерцептор логирования (тайминг)

Записывает стартовую метку времени, затем использует tap для измерения прошедшего времени, когда приходит финальное событие ответа — лёгкий хук для наблюдаемости.

@Injectable()
export class LoggerInterceptor implements HttpInterceptor {
  intercept<T>(
    req: HttpRequest<T>,
    next: HttpHandler,
  ): Observable<HttpEvent<T>> {
    const started = Date.now();
    return next.handle(req).pipe(
      tap((event: HttpEvent<T>) => {
        if (event instanceof HttpResponse) {
          const elapsed = Date.now() - started;
          console.log(`${req.method} ${req.urlWithParams} → ${event.status} (${elapsed}ms)`);
        }
      }),
    );
  }
}

Порядок выполнения

Порядок цепочки интерцепторов — запросы проходят через интерцепторы в порядке регистрации (сверху вниз); ответы возвращаются через них в обратном порядке (снизу вверх), как слои луковицы.

// Registration order (class-based DI)
providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor,    multi: true }, // 1st
  { provide: HTTP_INTERCEPTORS, useClass: LoggerInterceptor,  multi: true }, // 2nd
  { provide: HTTP_INTERCEPTORS, useClass: ErrorInterceptor,   multi: true }, // 3rd
]

// Request : Auth  → Logger → Error → server
// Response: server → Error  → Logger → Auth

Ставьте AuthInterceptor первым, чтобы токен прикреплялся до того, как LoggerInterceptor залогирует запрос, и чтобы повторы ниже по цепочке всё ещё несли авторизацию.

Функциональные интерцепторы (Angular 15+)

HttpInterceptorFn — современный функциональный интерцептор: обычная функция вместо класса, использующая inject() для зависимостей и регистрируемая через withInterceptors([...]). Поддерживает tree-shaking и требует значительно меньше шаблонного кода.

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).token();
  return next(
    req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }),
  );
};

// Registration (standalone, app.config.ts)
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([authInterceptor, errorInterceptor]),
    ),
  ],
};

КРИТИЧЕСКИ важные правила

Никогда не вызывайте subscribe() внутри интерцептора. Используйте только операторы .pipe(). Подписка выполняет запрос лишний раз и нарушает контракт — Angular подписывается на возвращённый поток сам.

Всегда возвращайте next.handle(cloned) (или next(cloned) для функциональных интерцепторов). Забытый return ломает конвейер — запрос не уходит, а observable вызывающего кода ничего не эмитит.

req.clone() — единственный способ изменить запрос — HttpRequest иммутабелен; прямое присваивание игнорируется или выбрасывает ошибку.

Вопросы и ответы — интервью

1. Почему HttpClient возвращает холодный Observable и что это означает?

Observable холодный — сам HTTP-запрос не отправляется, пока на него кто-то не подпишется. Каждая подписка запускает отдельный запрос, поэтому двойная подписка (или забытое разделение потока) отправляет вызов дважды. Из этого следует: используйте пайп async или firstValueFrom вместо ручных двойных подписок, добавляйте shareReplay, когда несколько потребителей должны получить один результат, и помните, что Observable HTTP без подписки вообще не выполняется.

2. Почему в интерцепторе нужно вызывать req.clone() вместо мутации запроса?

HttpRequest иммутабелен — его поля доступны только для чтения и не могут быть переприсвоены. clone({ ... }) возвращает новый запрос с применёнными переопределениями (например, setHeaders, url, params). Это сохраняет исходный запрос нетронутым, чтобы последующие интерцепторы, повторы и внутренние механизмы Angular всегда видели согласованное значение, и предотвращает тонкие баги из-за разделяемого изменяемого состояния.

3. В каком порядке выполняются несколько интерцепторов для запроса и для ответа?

Запросы проходят через интерцепторы в порядке регистрации (сверху вниз в массиве providers или в массиве withInterceptors). Затем ответ возвращается через них в обратном порядке (снизу вверх). Это луковица: первый зарегистрированный интерцептор — самый внешний слой: первым касается запроса и последним касается ответа.

4. Как реализовать глобальный спиннер загрузки, корректно обрабатывающий параллельные запросы?

Используйте интерцептор, опирающийся на сервис, который хранит счётчик, а не булево значение. Увеличивайте его на каждом исходящем запросе и уменьшайте внутри finalize (который срабатывает и при успехе, и при ошибке). Спиннер виден, пока счётчик больше нуля, поэтому он остаётся до завершения последнего из нескольких параллельных запросов. Обычное булево значение было бы сброшено первым ответом, пока остальные ещё выполняются.

5. В чём разница между интерцепторами на основе классов и функциональными?

Интерцепторы на основе классов реализуют HttpInterceptor с методом intercept(req, next), используют внедрение через конструктор и регистрируются через multi-провайдер HTTP_INTERCEPTORS. Функциональные интерцепторы (Angular 15+) — это обычные функции типа HttpInterceptorFn, которые используют inject() для зависимостей, вызывают next(req) напрямую и регистрируются через withInterceptors([...]) внутри provideHttpClient. Функциональная форма поддерживает tree-shaking, требует меньше шаблонного кода и рекомендуется по умолчанию для нового кода.

Квиз

1. Какой HTTP-метод идемпотентен, но небезопасен?
PUT заменяет ресурс целиком: его повтор приводит к тому же состоянию (идемпотентность), но он изменяет сервер (небезопасен). GET и идемпотентен, и безопасен; POST и PATCH не являются ни идемпотентными, ни безопасными.
2. Как изменить исходящий запрос внутри интерцептора?
HttpRequest иммутабелен. Единственный способ его изменить — req.clone({ ... }), который возвращает новый запрос с применёнными переопределениями. Прямое присваивание игнорируется или выбрасывает ошибку.
3. Какой оператор должен управлять глобальным спиннером загрузки, чтобы он скрывался и при успехе, и при ошибке?
finalize() выполняется ровно один раз при завершении ИЛИ ошибке потока, что делает его верным местом для уменьшения счётчика загрузки. Обработчик next у tap пропускает путь ошибки, а catchError срабатывает только при ошибках.
4. Какую единственную самую важную ошибку нужно избегать внутри интерцептора?
Никогда не вызывайте subscribe() внутри интерцептора — Angular подписывается на возвращённый поток сам, поэтому повторная подписка отправляет запрос лишний раз и ломает конвейер. Используйте только операторы .pipe() и всегда возвращайте next.handle(req).
